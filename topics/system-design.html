🏗️ System Design

Scaling, load balancing, distributed systems theory, resilience patterns, and microservices.

📈

Scaling Fundamentals

  • Horizontal vs Vertical Scaling
    Vertical (scale-up): bigger machine — more CPU, RAM. Simple (no code changes), but has an upper bound, expensive at the top end, and is a single point of failure. Horizontal (scale-out): more machines behind a load balancer. Scales to near-arbitrary limits, better fault tolerance, but requires stateless services — session state must live outside the instance (Redis, DB). Good for auto-scaling with cloud providers. Database vertical scaling buys time; horizontal scaling of writes requires sharding or a distributed DB.
  • Load Balancing
    Algorithms: Round Robin (simple, ignores load), Least Connections (good for variable request duration), IP Hash / Sticky Sessions (same client → same server, needed for stateful connections), Weighted (route more to powerful instances), Random. L4 (Transport): routes based on IP/TCP — fast, no app awareness. L7 (Application): routes based on HTTP headers, URL paths, cookies — enables content-based routing, SSL termination, WebSocket upgrades. AWS: ALB (L7), NLB (L4). Health checks: passive (detect failures by observing traffic) + active (periodic HTTP probe to /health). Remove unhealthy nodes from rotation.
  • Consistent Hashing
    Problem with modulo hashing: adding/removing 1 node out of N remaps ~N-1/N of keys — catastrophic cache invalidation. Consistent hashing: nodes and keys are placed on a hash ring (0 to 2³²). A key maps to the next node clockwise. Adding/removing a node only moves O(1/N) of keys. Virtual nodes: each physical node gets K virtual nodes on the ring, improving distribution and handling heterogeneous node sizes. Used in: DynamoDB, Cassandra, Redis Cluster, Memcached client libraries (Ketama).

Distributed Systems Theory

  • CAP Theorem
    In a distributed system that may experience network partitions, you can guarantee at most two of: Consistency (every read returns the most recent write or an error), Availability (every request gets a non-error response), Partition Tolerance (system works despite network partition). Since partitions will happen, the real choice is CP vs AP. CP systems (reject requests during partition to preserve consistency): ZooKeeper, HBase, etcd. AP systems (remain available, serve possibly stale data): Cassandra, DynamoDB, CouchDB. PACELC extends CAP: even without partitions, there's a latency vs consistency tradeoff.
  • Rate Limiting Algorithms
    Token Bucket: tokens added at rate R, bucket capacity B. Each request consumes 1 token. Allows bursts up to B. Most flexible — used by AWS, Stripe. Leaky Bucket: requests fill a bucket; processed at a constant rate (like a leaky container). Smooths bursts but queues them — adds latency. Bad for real-time. Fixed Window: count requests per minute. Simple but vulnerable to boundary bursts (2× the limit across a window boundary). Sliding Window Log: record timestamp of each request; count within rolling window. Accurate but memory-intensive. Sliding Window Counter: current_window_count + (prev_window_count × % remaining in current window). Good approximation with low memory. Distributed rate limiting: store counts in Redis (atomic INCR); use Lua scripts for atomicity.
  • Circuit Breaker
    Prevents cascade failures when a downstream service is failing. Three states: Closed: normal operation, requests pass through; failure count tracked. Open: failure threshold exceeded; requests fail fast (no call to downstream). After a timeout, moves to Half-Open. Half-Open: allow a probe request through; if it succeeds, close the breaker; if it fails, reopen. Always combine with timeouts (don't wait forever for a slow dependency) and retries with exponential backoff + jitter (avoid synchronized retry storms). Libraries: Hystrix (deprecated), Resilience4j (Java), polly (.NET), opossum (Node).
  • API Gateway
    Single entry point for all client requests. Responsibilities: SSL termination, authentication (validate JWT), authorization, rate limiting, request routing (path → service), load balancing, request/response transformation, observability (logging, tracing headers), circuit breaking. Benefits: centralizes cross-cutting concerns, clients only talk to one endpoint. Risks: becomes a bottleneck / SPOF if not HA; adds latency hop; can become a God object with too much logic. Tools: Kong, AWS API Gateway, Nginx + Lua, Traefik, Envoy, APISIX.

Advanced Patterns

  • CQRS
    Command Query Responsibility Segregation: separate the write model (commands that change state) from the read model (queries that return data). Each can use the most appropriate data store — writes to a normalized relational DB, reads from a denormalized read-optimized store (Elasticsearch, Redis, materialized views). The read model is kept in sync via events or DB triggers (eventual consistency). When to use: high read/write ratio imbalance, complex domain logic, need for audit trails. Don't use it everywhere — adds significant complexity.
  • Event Sourcing
    Store the sequence of events that led to the current state, not just the current state. State is derived by replaying events. Benefits: complete audit log (compliance-friendly), time-travel debugging (replay up to any point in time), natural event-driven architecture, rebuild any read model from the event log. Challenges: event schema evolution (can't easily change past events), eventual consistency (read models lag), high disk usage for long-lived aggregates (snapshots help), mental model shift. Often paired with CQRS. Used in banking, audit systems, collaborative tools (like Google Docs).
  • Saga Pattern
    Distributed transactions without 2-Phase Commit (2PC is blocking and doesn't work across microservices). A saga is a sequence of local transactions. Each step publishes an event that triggers the next. On failure, compensating transactions undo completed steps. Choreography: services react to events autonomously — no central coordinator. Decoupled but hard to track, hard to debug, circular dependencies possible. Orchestration: a central saga orchestrator issues commands to participants and handles failures. Easier to reason about but introduces coupling to the orchestrator. Temporal.io is a popular workflow engine for orchestrated sagas.

Microservices

  • Microservices vs Monolith
    Microservices pros: independent deployment, language/tech diversity per service, fault isolation, team autonomy (each team owns their service), scales individual components. Microservices cons: distributed systems complexity (network latency, partial failure), data consistency across services, operational overhead (deploy N services, not 1), observability harder (traces span services). When to start: strong default to monolith first — simpler to build, refactor, and test. Extract services when a clear seam emerges (team ownership, independent scaling need, different SLAs). Service mesh (Istio, Linkerd): handles mTLS between services, retries, circuit breaking, tracing — via sidecar proxy, not app code.
  • Service Discovery
    How services find each other in dynamic environments where IPs change constantly. Client-side discovery: service queries a registry (Consul, Eureka) and picks an instance. Client does the load balancing. Coupling: client needs registry SDK. Server-side discovery: client calls a load balancer/router, which queries the registry. Client stays simple. Kubernetes Services work this way — stable DNS name + kube-proxy handles routing. DNS-based: my-service.namespace.svc.cluster.local in Kubernetes. Simple and universal.

Quick Review

In a system design interview, how do you estimate QPS and storage?

Quick back-of-envelope numbers to memorize:

  • 1M users × 10 requests/day ÷ 86,400 sec/day ≈ ~115 QPS
  • 1 KB average request → 115 QPS × 1 KB = ~115 KB/s throughput
  • 1M users × 10 writes/day × 1 KB = 10 GB/day → ~3.6 TB/year
  • Latency targets: p50 < 100ms, p99 < 500ms is a common starting point for APIs
  • A single-core machine handles ~10K–50K simple HTTP requests/sec; DB reads ~5K–50K QPS

Always clarify: DAU vs MAU, read:write ratio, expected data size per entity, consistency requirements.

Why is 2-Phase Commit (2PC) problematic in microservices?

2PC requires a coordinator to drive two phases: (1) prepare — all participants say "ready to commit", (2) commit — coordinator tells all to commit. Problems:

  • Blocking: if the coordinator crashes after prepare but before commit, participants are stuck holding locks indefinitely
  • Single point of failure: coordinator crash halts the whole transaction
  • Network assumption: assumes all participants are reachable — violates real-world partition scenarios
  • Latency: two full round-trips, all participants must wait for the slowest

The Saga pattern avoids these by using local transactions + compensating actions instead of distributed locking.

What's the difference between latency and throughput, and how do they interact?

Latency: time for a single request to complete (ms). Report as percentiles: p50, p95, p99, p999 — not averages (averages hide tail latency).

Throughput: requests per second the system can handle.

They interact: Little's Law: L = λW (avg queue length = arrival rate × avg wait time). At high throughput approaching capacity, queues build up and latency degrades non-linearly. Rule of thumb: keep utilization below 70–80% to avoid queue buildup. The Universal Scalability Law (USL) models how adding resources diminishes returns due to contention and coherency overhead.
