Cache strategies, Redis data structures, eviction policies, CDN, and HTTP cache headers.

🗂️

Cache Strategies

  • Cache-Aside (Lazy Loading)
    Most common pattern. App checks cache → on miss, loads from DB → writes to cache → returns data. Pros: only caches what's actually read, resilient to cache failure (fall through to DB). Cons: cold start penalty (first request always hits DB), stale data possible (DB updated but cache not invalidated), risk of cache stampede on TTL expiry. Used everywhere as a default starting point.
  • Read-Through
    App always reads from cache. On a miss, the cache itself fetches from DB and stores it (via a library or cache provider). App code doesn't know about the DB on reads. Pros: simpler app code. Cons: tighter coupling between cache and DB; same cold start issue. Cache failure means no reads unless the library falls back. Used with: Hibernate 2nd-level cache, some CDN configurations.
  • Write-Through
    Every write goes to cache and DB synchronously before the write is acknowledged. Pros: cache is always fresh, no stale reads. Cons: write latency doubles (two writes per operation), cache may hold data that's never read (write amplification), doesn't help cold start for new data. Best when reads and writes to the same data are equally frequent.
  • Write-Behind (Write-Back)
    Write to cache immediately; DB write is async (buffered, flushed in batches). Pros: extremely low write latency, DB load reduced via batching. Cons: durability risk — if cache crashes before flushing, data is lost. Requires the cache to be durable (Redis AOF/RDB persistence) and a reliable flush mechanism. Good for high-write workloads where some data loss is acceptable (counters, analytics, non-critical state).
  • Cache Invalidation
    "There are only two hard things in CS: cache invalidation and naming things." — Phil Karlton. Strategies: TTL-based (simplest, eventual consistency), Event-based (invalidate on write — harder to get right across services), Version key (embed a version in the key: user:v3:123 — bump version to invalidate all old keys at once; old entries expire via TTL), Purge API (CDN-level explicit purge on deploy or content change). Write-on-invalidation: delete the key (don't update) — avoids race conditions where a slow writer overwrites a newer value.
  • Cache Stampede (Thundering Herd)
    A popular cache key expires → thousands of requests simultaneously miss → all hit the DB at once → potential DB overload. Solutions: Mutex/Lock: first requester acquires a distributed lock, fetches and repopulates; others wait or return stale data. Probabilistic Early Expiry (PER): occasionally refresh the cache slightly before TTL expires, based on how expensive the computation is — avoids synchronized expiry. Background refresh: serve stale data while asynchronously refreshing (stale-while-revalidate).

Redis Data Structures

  • Strings
    Most versatile type — holds text, integers, binary blobs (up to 512 MB). GET / SET / DEL. SET key val EX 60 NX (set if not exists, expire in 60s — atomic distributed lock primitive). INCR / DECR / INCRBY are atomic — great for counters (rate limiters, hit counters). MGET / MSET for batch operations (fewer roundtrips). Distributed lock with Redlock: acquire on majority of N Redis nodes, release with Lua script to ensure atomicity.
  • Hashes
    Map of field → value. Ideal for representing objects without serializing the whole thing. HSET user:1 name Alice age 30 / HGET user:1 name / HMGET / HGETALL / HDEL / HKEYS. HINCRBY user:1 login_count 1 — atomic field-level increment without fetching/deserializing the whole object. Memory efficient for small hashes (Redis uses a compact ziplist encoding when fields ≤ 128 and values are small).
  • Lists
    Doubly-linked list. O(1) push/pop at both ends. O(n) access by index. RPUSH (append right) / LPUSH (prepend) / LPOP / RPOP / LRANGE key 0 -1 (all elements). Queue: RPUSH + BLPOP (blocking pop — worker waits for work, 0 timeout = wait forever). Stack: LPUSH + LPOP. Activity feed: LPUSH + LTRIM to keep last N events.
  • Sets
    Unordered collection of unique strings. O(1) add/remove/membership test. SADD / SMEMBERS / SISMEMBER / SCARD (count) / SREM. Set operations: SUNION, SINTER (intersection), SDIFF. Use cases: unique visitors per day, user tags, social graph (followers: SINTER user:1:following user:2:following = mutual follows), IP blocklist membership checks.
  • Sorted Sets (ZSets)
    Every member has a score (float). Members are unique; scores are not. ZADD leaderboard 1500 alice / ZRANGE (asc) / ZREVRANGE (desc) / ZRANK / ZSCORE / ZINCRBY. Leaderboard: scores = points, ZREVRANGE … WITHSCORES for top-N. Rate limiter (sliding window): score = timestamp, member = request_id. ZREMRANGEBYSCORE 0 (now-window) to remove old entries, ZCARD to count. Priority queue: score = priority or scheduled time.

HTTP Caching & CDN

  • Cache-Control & Conditional Requests
    Key directives: max-age=3600 (fresh for 1h), s-maxage (for shared caches/CDNs), no-cache (must revalidate before using), no-store (never store), private (browser only, not CDN), public, must-revalidate, stale-while-revalidate=60 (serve stale while refreshing in background). ETag: server sends ETag: "abc123"; client sends If-None-Match: "abc123"; server returns 304 Not Modified if unchanged (saves bandwidth, not latency). Last-Modified / If-Modified-Since: same pattern with timestamps. ETags are more precise.
  • CDN
    Geographically distributed edge servers cache content close to users. Reduces latency (physics: speed of light) and offloads origin. Origin pull (most common): edge fetches from origin on first miss, caches per Cache-Control. Origin push: you upload assets to CDN. Cache-Control headers on your origin control CDN TTLs. Use long TTLs for hashed assets (main.abc123.js), short TTLs for HTML. Purge/invalidation APIs: CloudFront Invalidation, Cloudflare Cache Purge — use on deploy. Surrogate keys / cache tags let you purge all cache entries tagged with a specific product ID. Edge computing (Cloudflare Workers, Lambda@Edge) runs code at the edge for dynamic content.

Quick Review

What are the cache eviction policies and when do you use each?

LRU (Least Recently Used): evicts the item least recently accessed. Good general-purpose choice — assumes recently accessed items will be accessed again.

LFU (Least Frequently Used): evicts the item with the lowest access count. Better for skewed access patterns (some items always popular) but more expensive to track. Redis uses LFU approximation.

TTL-based: items expire after a set time regardless of usage. Guarantees freshness. Redis: EXPIRE key 300.

Random: evicts a random key. Surprisingly effective at low overhead.

Redis maxmemory-policy options: allkeys-lru, volatile-lru (only keys with TTL), allkeys-lfu, allkeys-random, noeviction (return errors when full).

Why delete a key on invalidation rather than updating it?

Updating (SET) the cache on write can cause a race condition: if two concurrent writes race, a slower write may overwrite a newer value:

  1. Request A reads value (v1), starts computing
  2. Request B writes v2 to DB and cache
  3. Request A finishes, writes v1 to cache — now stale!

Deleting the key means the next read will fetch from DB, always getting the latest value. The race is between the delete and the next read, which is much rarer and self-correcting. This pattern is called "cache-aside with delete on write".

When would you use Redis Sorted Sets for rate limiting?

Sliding window rate limiting: store each request as a member with the timestamp as its score.

MULTI
ZREMRANGEBYSCORE key 0 (now - window_ms)
ZADD key now request_id
ZCARD key
EXPIRE key window_seconds
EXEC

If ZCARD exceeds the limit, reject the request. This gives you an exact sliding window with O(log N) operations. The downside is memory usage (storing every request); a sliding window counter approximation using two fixed-window buckets uses less memory.
