📦 Infrastructure

Containers and orchestration — Docker, Kubernetes, and the runtime infrastructure that keeps your services running at scale.

🐳

Docker & Containers

A container is a process (or group of processes) running in isolation — its own filesystem, network namespace, and process tree — but sharing the host kernel. Unlike a VM, there's no guest OS. Containers start in milliseconds, are typically a few hundred MB, and the overhead is minimal. Docker popularized the workflow of building, shipping, and running containers via a standard image format.

How Docker Works

Docker uses Linux kernel features: namespaces (isolate what the process can see — filesystem, network, PIDs) and cgroups (limit resource consumption — CPU, memory, I/O). A container is not a VM — it's an isolated process using the host kernel. On macOS and Windows, Docker Desktop runs a lightweight Linux VM; on Linux, containers run directly.

A Docker image is a read-only, layered filesystem snapshot. Each instruction in a Dockerfile creates a layer. Layers are cached and shared across images — if your base image is the same, it's not duplicated on the host. A container is a running image with a writable layer on top. That writable layer is ephemeral — it disappears when the container stops. Persistent data requires volumes.

Writing Good Dockerfiles

# Multi-stage build — keep final image small
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production  # separate layer — cache unless package.json changes
COPY . .
RUN npm run build

FROM node:20-alpine AS runtime
WORKDIR /app
# Copy only production artifacts, not source or dev tools
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
# Run as non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser
CMD ["node", "dist/server.js"]

Key practices: use multi-stage builds to keep production images small (don't ship compilers, source code, dev dependencies), order layers from least-to-most-frequently-changing (COPY package.json before COPY . . so npm install is cached unless dependencies change), run as a non-root user, use specific image tags (not node:latest — pin to a digest for reproducibility), and use .dockerignore to exclude node_modules, .git, and local build artifacts.

Volumes & Networking

Volumes persist data beyond the container lifecycle and can be shared between containers. Three types: named volumes (docker volume create dbdata, managed by Docker), bind mounts (mount a host directory into the container — useful in development, not in production), tmpfs mounts (in-memory, lost on container stop).

Docker networking: containers on the same network can communicate by container name (Docker's embedded DNS resolves them). Bridge network (default) isolates containers from the host. Host network (Linux only) gives the container direct access to the host's network interfaces — maximum performance, minimum isolation. For production, always put containers on an isolated user-defined network and expose only necessary ports to the host.

Docker Compose

Compose defines a multi-container application in a single YAML file. Essential for local development: docker compose up starts your API, database, Redis, and any other services defined, each with appropriate environment variables, volumes, and networking. Key patterns: use depends_on with condition: service_healthy (not just service_started) to wait for dependencies to be ready, use environment files (env_file: .env.local) for local secrets, and use named volumes for databases so data persists across docker compose down.

ENTRYPOINT vs CMD

Both define what runs when a container starts, but they serve different roles: ENTRYPOINT is the executable; CMD provides default arguments. When both are set, the container runs ENTRYPOINT + CMD as a single command. CMD alone is the default command, but users can override it at runtime (docker run myimage bash). ENTRYPOINT is harder to override (requires --entrypoint flag) — use it when the container has a fixed purpose.

# CMD only — easy to override, good for general-purpose images
CMD ["node", "server.js"]

# ENTRYPOINT + CMD — fixed executable, overridable args
ENTRYPOINT ["node"]
CMD ["server.js"]        # override: docker run myimage other.js

# Common pattern: wrapper script as entrypoint
ENTRYPOINT ["/docker-entrypoint.sh"]
CMD ["node", "server.js"]

Always use exec form (["executable", "arg"]), not shell form (executable arg). Shell form runs your command as a child of /bin/sh -c, which means signals like SIGTERM go to the shell, not your process. Exec form runs your process directly as PID 1 and receives signals correctly.

Signal Handling & PID 1

The process inside a container that runs as PID 1 has special responsibilities that normal init systems handle: forwarding signals to child processes and reaping zombie processes. Most application runtimes (Node, Python, Go binaries) are not designed to run as PID 1. When you run docker stop, Docker sends SIGTERM to PID 1 and waits 10 seconds before sending SIGKILL. If your app doesn't handle SIGTERM as PID 1, it gets hard-killed — no graceful shutdown, no connection draining, no in-flight request completion.

# Add tini as a lightweight init process
FROM node:20-alpine
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]

# Or use Docker's built-in init (less control, but simpler)
# docker run --init myimage

tini (and dumb-init) are minimal init systems designed for containers. They properly forward signals to your app and clean up zombie processes. Add tini to any image where graceful shutdown matters — which is essentially every production service. For Go, the runtime handles signals as PID 1 reasonably well if you set up a signal handler; for Node and Python, tini is the safest default.

BuildKit & Advanced Build Caching

BuildKit is the default build engine (enabled since Docker 23). Its killer feature for development and CI is cache mounts: directories that persist between builds but are never included in the final image. This means your package manager cache — Go's module cache, npm's node_modules cache, pip's wheel cache — survives across builds even when dependencies change.

# Go: cache module downloads and build cache separately
RUN --mount=type=cache,target=/go/pkg/mod \
    --mount=type=cache,target=/root/.cache/go-build \
    go build -o /app/server ./cmd/server

# Node: cache npm's global cache directory
RUN --mount=type=cache,target=/root/.npm \
    npm ci --prefer-offline

# Python: cache pip's wheel cache
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

For secrets at build time — API keys, private repo tokens — never put them in ENV or RUN commands. Even if you delete them in a later layer, they're still in the image history. Use --mount=type=secret instead: the secret is available during that RUN step only, never written to any layer:

# Pass a secret (never ends up in the image)
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci

# Build: docker build --secret id=npmrc,src=.npmrc .

Image Security

The most common mistake: secrets baked into image layers. Even if a later RUN rm secret-file removes the file, it still exists in the earlier layer — anyone who pulls your image can extract it with docker history or by running the intermediate layer. Use BuildKit secrets or multi-stage builds to ensure credentials never touch a committed layer.

Minimize your attack surface with smaller base images:

  • Alpine (~5 MB) — musl libc instead of glibc. Works for most apps but can cause subtle issues with CGo-linked binaries. Good default for most services.
  • Distroless (Google's gcr.io/distroless) — no shell, no package manager, no utilities. Attacker can't exec into the container even if they exploit your app. Ideal for Go and Java. Pair with a builder stage for compilation.
  • scratch — completely empty filesystem. Only viable for statically compiled Go binaries with no external dependencies. Smallest possible image.

Pin base images to a digest, not just a tag: FROM node:20-alpine@sha256:abc123.... Tags are mutable — node:20-alpine today is not the same image as next week. In CI, pull digests and pin them in your Dockerfile. Run image scanning on every build: trivy image myimage:latest or docker scout cves myimage will surface known CVEs in your base image and dependencies before they reach production.

Multi-Platform Builds

M-series Macs are arm64; most production infrastructure is linux/amd64. Images built without specifying a platform will silently produce an architecture-specific image, causing "exec format error" when deployed. Use buildx to build for multiple platforms in one command:

# Build for both architectures and push a multi-arch manifest
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag myorg/myapp:latest \
  --push .

# Or build for amd64 only when developing on Apple Silicon
docker buildx build --platform linux/amd64 -t myapp:dev --load .

Buildx uses QEMU for cross-platform emulation when the host architecture doesn't match the target. It's slower than native builds (emulated amd64 on arm64 is ~2-5× slower) but produces correct images. In CI on amd64 runners, the build is native and fast. The --push flag sends all architecture variants to the registry together as a single multi-arch manifest — users get the right variant automatically when they pull.

Kubernetes

Kubernetes (K8s) is a container orchestration platform that automates deployment, scaling, and management of containerized applications across a cluster. The core value: describe the desired state, Kubernetes continuously works to make reality match the description and recover from failures.

Core Concepts

Everything in Kubernetes is a resource with a spec (desired state) and status (actual state). The controller loop runs constantly: observe state → compare to desired → act to reconcile.

  • Pod: the smallest deployable unit — one or more containers that share a network namespace and storage. Containers in a pod communicate via localhost. Pods are ephemeral — don't manage pods directly.
  • Deployment: manages a ReplicaSet of identical pods. Specify replicas: 3 and Kubernetes keeps 3 pods running. Rolling updates by default: brings up new pods before terminating old ones. Rollback with kubectl rollout undo.
  • Service: a stable network endpoint (IP + DNS name) that load-balances across a set of pods. Pods come and go; the Service endpoint stays constant. Types: ClusterIP (internal only), NodePort (exposes on each node's IP), LoadBalancer (provisions a cloud LB), ExternalName (DNS alias).
  • Ingress: Layer 7 (HTTP) routing rules. Route /api/* to the API service, / to the frontend service, by hostname to different services. Requires an Ingress Controller (nginx-ingress, Traefik, AWS ALB Ingress Controller).
  • ConfigMap: non-sensitive configuration as key-value pairs, mounted as env vars or files. Secret: same but base64-encoded and with stricter RBAC. Neither actually encrypts at rest by default — use Sealed Secrets or an external secrets manager (AWS Secrets Manager via External Secrets Operator).
  • PersistentVolumeClaim (PVC): a request for persistent storage. The cluster provisions a PersistentVolume (actual disk) to satisfy the claim. Abstracts the underlying storage type (EBS, GCE PD, NFS, local SSD).

Resource Requests & Limits

Requests: the amount of CPU/memory the scheduler guarantees the pod. The scheduler only places the pod on a node with enough unallocated resources. Limits: the maximum the pod can use. Exceed the CPU limit → throttled. Exceed the memory limit → OOMKilled. requests should reflect steady-state usage; limits should be 1.5–2× requests to handle spikes without OOM. Setting requests = limits is predictable but wastes resources. Never leave limits unset (noisy neighbor problem).

resources:
  requests:
    cpu: "100m"     # 0.1 CPU cores
    memory: "128Mi"
  limits:
    cpu: "500m"     # 0.5 CPU cores
    memory: "512Mi"

Health Checks

Liveness probe: is the container still alive? If it fails, Kubernetes restarts the container. Use for: detecting deadlocks, stuck processes. Don't fail the liveness probe for upstream dependency failures — Kubernetes will restart your pod in a crash loop but the dependency is still down. Readiness probe: is the container ready to receive traffic? If it fails, the pod is removed from Service endpoints. Use for: warmup time, temporary unavailability. Fail readiness when your pod can't handle requests. Startup probe: for slow-starting containers, give them time before the liveness probe takes over.

Horizontal Pod Autoscaler (HPA)

HPA scales the number of pod replicas based on metrics. CPU utilization is the built-in metric: if average CPU across all pods exceeds 70%, add more replicas. Custom metrics (request rate, queue depth, latency) require Prometheus + KEDA or a custom metrics adapter. HPA works within bounds you set: minReplicas: 2, maxReplicas: 20. The Cluster Autoscaler adds/removes nodes when pods can't be scheduled.

Namespace & RBAC

Namespaces provide soft isolation within a cluster — logical grouping, resource quotas, and separate RBAC policies. Common pattern: production, staging, development namespaces in the same cluster. Kubernetes RBAC controls who can do what to which resources: Role or ClusterRole defines permissions; RoleBinding grants them to a ServiceAccount, user, or group. Follow least privilege: give each service account only the API verbs it actually needs.
