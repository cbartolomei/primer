Containers

  • Docker Fundamentals
    Image: read-only, layered filesystem (Union FS). Each RUN/COPY instruction adds a layer. Container: running instance of an image + a writable container layer (ephemeral by default). Key Dockerfile instructions: FROM, RUN (shell command, creates a layer), COPY (files into image), EXPOSE (documentation only), ENV, ARG, CMD (default command, overridable), ENTRYPOINT (fixed command, CMD becomes args). Multi-stage builds: FROM node:20 AS builder → build → FROM node:20-alpine → COPY --from=builder /app/dist .. Final image only has runtime artifacts, not build tools.
  • Docker Compose
    Defines and runs multi-container applications. Core concepts: services, networks, volumes. depends_on controls start order but not readiness — use health checks with condition: service_healthy or a wait script for true readiness. Shared volumes for local dev (mount source code into container for hot reload). Named volumes for persistent data (DB files). Good for: local development, integration test environments. Not recommended for production orchestration (use Kubernetes).
  • Image Best Practices
    Use specific tags in production (node:20.11.1-alpine, not node:latest). Minimize image size: use alpine/distroless base images. Remove build tools in multi-stage builds. Clean up package caches in the same RUN layer that installed them. Layer caching: order Dockerfile instructions from least to most frequently changing. COPY package.json . → RUN npm install → COPY . . — dependency install layer cached unless package.json changes. Non-root user: USER node — don't run containers as root. Vulnerability scanning: Trivy, Snyk, Docker Scout — integrate into CI. .dockerignore: exclude node_modules, .git, secrets from build context.

Kubernetes

  • Core Resources
    Pod: smallest deployable unit. 1+ containers sharing network namespace (same IP, can talk on localhost) and volumes. Ephemeral — don't run pods directly. Deployment: manages a ReplicaSet for desired count + rolling updates. Use this for stateless apps. Service: stable DNS name + virtual IP routing to pods (via label selector). Types: ClusterIP (internal), NodePort (external via node IP), LoadBalancer (cloud LB). ConfigMap: non-secret config (env vars, config files). Secret: base64-encoded (not encrypted at rest by default — enable etcd encryption or use external secrets operator). Use sealed-secrets or external-secrets for GitOps. Namespace: virtual cluster for resource isolation and quotas.
  • Workload Types & Autoscaling
    Deployment: stateless apps, rolling updates. StatefulSet: stable pod identity (pod-0, pod-1), stable storage. For databases, Kafka, Zookeeper. DaemonSet: one pod per node — for log collectors (Fluentd), node monitoring (node-exporter), network plugins. Job: runs to completion (batch processing, DB migrations). CronJob: scheduled jobs. HPA (Horizontal Pod Autoscaler): scales Deployment replicas based on CPU/memory or custom metrics (Prometheus adapter). VPA (Vertical): adjusts resource requests/limits. KEDA: event-driven autoscaling (scale-to-zero based on queue depth, etc.). Always set resource requests (scheduling) and limits (enforcement). Requests < Limits = Burstable QoS.
  • Networking & Ingress
    Ingress: L7 HTTP(S) routing — route by hostname (api.example.com) or path (/api/) to Services. Requires an Ingress Controller (nginx-ingress, Traefik, AWS ALB Ingress Controller). kube-dns / CoreDNS: service discovery via DNS. my-service.my-namespace.svc.cluster.local resolves to the Service ClusterIP. Network Policies: firewall rules for pods (default: all pods can talk to all pods). Requires a CNI plugin that supports it (Calico, Cilium). Pod-to-pod communication: all pods can reach all other pods by IP across nodes (flat network). Container-to-container in same pod: localhost.

CI/CD & Observability

  • CI/CD & Deployment Strategies
    Typical pipeline: Code push → Lint/Test → Build image → Security scan → Push to registry → Deploy to staging → Smoke tests → Deploy to prod. Blue/Green: two identical environments. Switch traffic from Blue (current) to Green (new) atomically. Instant rollback by switching back. Requires double infrastructure. Canary: route X% of traffic to new version. Gradually increase percentage. Catch bugs with limited blast radius. Requires traffic splitting at LB or service mesh layer. Feature flags (LaunchDarkly, Unleash): decouple deployment from feature release. Deploy code with flags off; enable for % of users. Rollback = turn flag off. GitOps (Argo CD, Flux): Git is the single source of truth for infrastructure state. A controller continuously reconciles cluster state with the Git repo.
  • Observability
    Three pillars: Logs (what happened), Metrics (how much/fast — quantitative), Traces (request journey across services). Structured logging: JSON logs with consistent fields (level, timestamp, trace_id, service). Enables filtering and querying in Datadog/Splunk/CloudWatch. Correlation IDs: generate a X-Request-ID at the edge, propagate through all service calls — link logs across services for a single user request. Metrics: Prometheus (pull-based, time-series) + Grafana (dashboards). The four golden signals: Latency, Traffic, Errors, Saturation. Distributed tracing: Jaeger, Zipkin, Datadog APM. OpenTelemetry: vendor-neutral instrumentation standard for logs, metrics, and traces.

Quick Review

Why is a Kubernetes Secret not actually secret?

Kubernetes Secrets are base64-encoded, which is encoding, not encryption. Anyone with access to the etcd cluster (or the right RBAC permissions) can decode them trivially: echo "dXNlcjpwYXNz" | base64 -d.

To actually protect secrets: (1) Enable etcd encryption at rest — Kubernetes supports this natively with an encryption provider config. (2) Use external secrets — store secrets in AWS Secrets Manager/Vault and sync them into the cluster with the External Secrets Operator or sealed-secrets. This way the source of truth is the external store (with proper ACLs), not plaintext in your Git repo or etcd.

What's the difference between resource requests and limits in Kubernetes?

Requests: the amount of resources the scheduler guarantees the container will have. Used for scheduling decisions — a pod is placed on a node with enough unreserved capacity.

Limits: the maximum a container can use. CPU: container is throttled if it exceeds the limit (it doesn't get killed). Memory: container is OOM-killed if it exceeds the limit.

If requests = limits → Guaranteed QoS (highest priority, never evicted under memory pressure unless it exceeds limit). If requests < limits → Burstable QoS. No requests/limits → BestEffort (evicted first).

What are the four golden signals and what do they measure?

From the Google SRE Book:

  • Latency: time to serve a request. Track both successful and failed request latency. Report as percentiles (p50, p95, p99), never just averages.
  • Traffic: demand on the system — requests/sec, queries/sec, messages/sec. Baseline for capacity planning.
  • Errors: rate of requests that fail. Explicit (5xx) + implicit (200 with wrong content). Track error budget against SLO.
  • Saturation: how "full" a service is — CPU %, memory %, queue depth, thread pool utilization. Most services degrade before reaching 100% saturation.
