🌐 HTTP & APIs

How the web actually works — HTTP evolution, the tradeoffs between API paradigms, and the network fundamentals that affect every service you build.

📡

HTTP Protocol

HTTP is the application-layer protocol for the web. Understanding its evolution helps you reason about what performance problems each version solves — and what its remaining constraints are.

HTTP/1.1

The protocol that made the modern web. Persistent connections (keep-alive) eliminated the TCP handshake overhead of HTTP/1.0's one-connection-per-request model. Pipelining (sending multiple requests before receiving responses) was added but never reliably implemented — buggy proxies broke it, so browsers kept it off by default.

The main problem: head-of-line blocking. Requests on a connection are processed in order. If the browser has 30 resources to fetch and the first response is slow, all 30 wait. Browsers worked around this by opening 6 parallel connections per domain — but that means 6 TCP handshakes, 6 TLS handshakes, and 6 flow-control states. It's a hack that worked, not a solution.

HTTP/2

Designed to fix HTTP/1.1's performance problems without changing semantics. Key features:

  • Binary framing: replaces plaintext with a binary frame layer. Efficient to parse, no ambiguity.
  • Multiplexing: multiple streams on a single TCP connection. Each stream is an independent request/response. No head-of-line blocking at the HTTP layer — a slow stream doesn't block others. Eliminates the need for multiple connections.
  • Header compression (HPACK): HTTP headers repeat heavily across requests (User-Agent, Accept, Authorization). HPACK compresses them using a static table of common headers and a dynamic table of recently-seen headers. Reduces header overhead by 80–90%.
  • Server push: server can push resources the client will need before it asks. In theory, push CSS before the browser parses HTML and requests it. In practice, widely misused (pushing already-cached resources), and removed from Chrome's implementation in 2022. Largely a failed experiment.

HTTP/2 still has head-of-line blocking at the TCP layer. One packet loss stalls all streams on the connection — because TCP guarantees ordered delivery and waits for the retransmit before delivering subsequent segments.

HTTP/3 & QUIC

HTTP/3 moves from TCP to QUIC — a new transport protocol built on UDP, developed by Google and standardized by IETF. QUIC solves TCP's head-of-line blocking: each stream has independent flow control, so a lost packet only stalls its own stream, not all streams.

QUIC also integrates TLS 1.3 into the handshake. A new connection to a known server takes 1 RTT (vs TCP+TLS's 2-3 RTTs). A resumed connection takes 0 RTT (the client can send data in the first packet, using a session ticket from the previous connection). QUIC also handles connection migration — when your phone switches from WiFi to cellular, the connection ID changes but the QUIC connection survives without re-handshaking. The connection is identified by a connection ID, not a 4-tuple (src IP, src port, dst IP, dst port).

TLS

TLS (Transport Layer Security) is the cryptographic layer beneath HTTPS. The handshake establishes: which cipher suite to use, exchanges certificates for authentication, and derives a shared session key for symmetric encryption. TLS 1.3 (current) simplified to 2 supported key exchange mechanisms (ECDHE only), removed weak cipher suites, and made the handshake 1 RTT (down from 2 in TLS 1.2).

Certificate chain: your server cert is signed by an intermediate CA, which is signed by a root CA trusted by the OS/browser. The browser validates the chain up to a known root. If any cert in the chain is expired or revoked (OCSP), the handshake fails. Misconfigurations — like not serving the intermediate cert — cause validation failures on some clients even if your cert is valid.

HSTS (HTTP Strict Transport Security): response header that tells the browser to only connect over HTTPS for a domain for N seconds. Prevents SSL-stripping attacks. Strict-Transport-Security: max-age=31536000; includeSubDomains; preload. The preload directive adds you to the browser's hardcoded list — no HTTP connection is ever attempted, even on first visit.

Status Codes That Matter

Beyond the basics: 200 OK, 201 Created (new resource, return Location header), 204 No Content (success, no body — for DELETE or updates with no return value), 301 Moved Permanently (change bookmark, browsers cache indefinitely), 302 Found / 307 Temporary Redirect (don't cache), 304 Not Modified (conditional GET, use cached version), 400 Bad Request (client error, don't retry), 401 Unauthorized (authentication required), 403 Forbidden (authenticated but not authorized), 404 Not Found, 409 Conflict (optimistic lock conflict), 429 Too Many Requests (rate limited — include Retry-After header), 500 Internal Server Error (retry is safe if idempotent), 502 Bad Gateway (upstream unreachable), 503 Service Unavailable (shed load, include Retry-After), 504 Gateway Timeout.

API Paradigms

REST, GraphQL, and gRPC are the three dominant API paradigms. Each reflects different tradeoffs around performance, type safety, flexibility, and operational complexity. Knowing when to use which is a real engineering judgment.

REST

REST (Representational State Transfer) is an architectural style, not a protocol. RESTful APIs use HTTP methods (GET, POST, PUT, PATCH, DELETE) to operate on resources identified by URLs. The constraints that make an API "RESTful": stateless (each request is self-contained, no server-side session), uniform interface (resources, standard methods, hypermedia as the engine of application state — HATEOAS, though almost no one implements HATEOAS), cacheable, layered system.

REST works well because: HTTP infrastructure (caches, proxies, CDNs, load balancers) understands it, it's human-readable, and it's universally supported. Its problems: over-fetching (endpoint returns 50 fields, client needs 3), under-fetching (must make multiple requests to compose a view), no strong schema enforcement without OpenAPI, and versioning is painful (v1/, v2/, or header-based versioning both have costs).

Good REST API design: nouns, not verbs in URLs (/users/123/orders, not /getUserOrders). Use HTTP methods for semantics. Return consistent error shapes. Use pagination (cursor-based, not offset). Version from the start (/v1/). Always use HTTPS. Document with OpenAPI/Swagger — it generates client SDKs and provides a contract.

GraphQL

GraphQL is a query language and runtime for APIs. The client specifies exactly what data it needs; the server returns exactly that. One endpoint, flexible queries. Solves REST's over/under-fetching problems.

query {
  user(id: "123") {
    name
    email
    orders(last: 5) {
      id
      total
      items { productName quantity }
    }
  }
}

Benefits: no over-fetching, no under-fetching, strong schema (the schema is a contract between client and server, introspectable), great for clients with variable data needs (mobile vs web), and easy to add new fields without versioning. Tradeoffs: N+1 query problem is severe (each orders field resolver runs a query — 100 users = 100 queries, unless you use DataLoader for batching), complex queries can be expensive (no easy per-endpoint rate limiting — must use query complexity analysis), HTTP caching is harder (all queries POST to one endpoint), and the learning curve is real. GraphQL is a great fit for: BFF layer aggregating multiple services, client-driven applications with diverse data needs. It's overkill for: simple CRUD APIs or internal service-to-service calls.

gRPC

gRPC (Google Remote Procedure Call) uses Protocol Buffers (protobuf) for serialization and HTTP/2 for transport. You define services and message types in a .proto file; the gRPC toolchain generates client and server stubs in your language.

// user.proto
service UserService {
  rpc GetUser(GetUserRequest) returns (User);
  rpc ListUsers(ListUsersRequest) returns (stream User);
}
message User {
  string id = 1;
  string name = 2;
  string email = 3;
}

Advantages: protobuf is 3-10× smaller than JSON and faster to serialize/deserialize, HTTP/2 multiplexing enables streaming (server streaming, client streaming, bidirectional streaming), the generated stubs are strongly typed in every language, and performance is excellent. Tradeoffs: protobuf is not human-readable (debugging requires tooling), browser support requires a grpc-web proxy (gRPC uses HTTP/2 trailers which browsers can't access directly), and schema changes require regenerating code. gRPC is the default choice for: internal service-to-service communication, high-throughput microservices, mobile backends where bandwidth matters.

WebSockets

WebSockets provide full-duplex, persistent connections between client and server. After an HTTP upgrade handshake, the connection becomes a TCP stream — either side can send messages at any time. Use for: real-time features (chat, live dashboards, collaborative editing, live game state). Not appropriate for: simple request/response patterns (the overhead of a persistent connection isn't worth it) or fire-and-forget notifications (use Server-Sent Events or push notifications instead). WebSocket connections are stateful — horizontal scaling requires sticky sessions or a pub/sub layer (Redis) to broadcast messages across server instances.

Network Fundamentals

You don't need to implement TCP, but you need to understand its behavior to reason about latency, connection costs, and why certain performance patterns work.

TCP

TCP is a reliable, ordered, connection-oriented transport protocol. Before sending data, it establishes a connection with a 3-way handshake: SYN → SYN-ACK → ACK. This costs 1 RTT. For HTTPS: add 1-2 RTTs for TLS. So before your first byte of actual data, a new connection to a server incurs 2–3 RTTs — up to 300ms from across continents.

TCP slow start: a new connection doesn't immediately use all available bandwidth. TCP starts with a small congestion window and doubles it each RTT until packet loss occurs, then backs off. For small responses (HTML documents, API responses), the connection may never leave slow start — the transfer completes before the window grows large enough to matter. This is why HTTP/2 multiplexing over a single warmed-up connection is faster than multiple HTTP/1.1 connections: the existing connection has a large window.

TIME_WAIT: after a TCP connection closes, the port enters TIME_WAIT for 2 MSL (~60s) to handle late-arriving packets. Under high connection churn, servers can exhaust ephemeral ports. Solution: keep-alive connections (reuse existing), or increase the ephemeral port range.

DNS

DNS resolves hostnames to IP addresses. The lookup chain: local cache → OS cache → recursive resolver (your ISP or 8.8.8.8) → root nameservers → TLD nameservers → authoritative nameservers. Each step adds latency. TTL controls how long results are cached — short TTLs mean more DNS lookups but faster failover during incidents. Long TTLs mean DNS changes take longer to propagate.

Practical implications: when you change a DNS record, old records are cached for up to the TTL duration — lower TTL before a planned migration, raise it after. DNS lookups add latency to first requests (before the result is cached). Browsers parallelize DNS lookups for resources on different domains. HTTP/2 and connection reuse reduce the cost of DNS — if everything is on one domain, there's only one lookup. CDNs use DNS as a traffic routing mechanism — returning different IPs based on the resolver's location (latency-based routing).

Connection Pooling

Creating a TCP connection (and TLS handshake) is expensive — 2-3 RTTs. Every database driver, HTTP client, and gRPC client should use connection pooling. A pool maintains a set of open connections and reuses them across requests. For databases: pool size should be roughly the number of CPU cores on the DB server — larger pools don't help (DB cores are the bottleneck) and increase memory overhead. PgBouncer pools connections to Postgres at the connection level, allowing many app connections to multiplex over fewer DB connections (Postgres has a per-connection overhead of ~5MB).
