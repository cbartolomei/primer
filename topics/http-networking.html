🌐 HTTP & APIs

HTTP protocol versions, API paradigms (REST, GraphQL, gRPC), WebSockets, and network fundamentals.

📡

HTTP Protocol

  • HTTP Methods
    GET — safe, idempotent, cacheable. POST — not idempotent, not safe, creates resources. PUT — full replace, idempotent. PATCH — partial update, not necessarily idempotent. DELETE — idempotent. HEAD — like GET but no body (check headers/existence). OPTIONS — returns allowed methods; used in CORS preflight. Safe = no server side-effects. Idempotent = same result for N identical calls.
  • Status Codes
    1xx informational (101 Switching Protocols for WebSocket upgrade). 2xx success: 200 OK, 201 Created, 202 Accepted (async), 204 No Content. 3xx redirect: 301 Permanent, 302 Temporary, 304 Not Modified (conditional GET cache hit). 4xx client error: 400 Bad Request, 401 Unauthenticated, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable, 429 Too Many Requests. 5xx server error: 500 Internal, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout.
  • Common Headers
    Request: Authorization (Bearer token / Basic), Content-Type (body format), Accept (expected response format), Accept-Encoding (gzip, br), Cookie, X-Request-ID (tracing). Response: Content-Type, Cache-Control, ETag, Location (redirects/created resource), Set-Cookie, Retry-After (with 429/503), X-RateLimit-*.
  • HTTP/1.1 vs HTTP/2 vs HTTP/3
    HTTP/1.1: persistent connections (keep-alive), pipelining (rarely used due to HOL blocking — a slow response blocks subsequent ones on the same connection), text-based. HTTP/2: binary framing, multiplexing (many requests on one TCP connection, no HOL at app layer), HPACK header compression, server push (deprecated in Chrome). HOL still exists at TCP layer. HTTP/3: QUIC over UDP, eliminates TCP-layer HOL blocking, built-in TLS 1.3, connection migration (change IP without reconnecting — great for mobile). Each stream is independent.

API Paradigms

  • REST
    An architectural style, not a protocol. Fielding's 6 constraints: Stateless (server holds no session state — each request is self-contained), Client-Server, Cacheable, Uniform Interface (resource-based URLs, standard methods, HATEOAS), Layered System, Code on Demand (optional). Resource URLs are nouns (/users/42/orders), not verbs (/getOrders). Versioning strategies: URL path (/v1/), header (Accept-Version), query param.
  • GraphQL
    Schema-first (SDL). Query (read), Mutation (write), Subscription (real-time over WebSocket). Solves REST's over-fetching (too many fields) and under-fetching (multiple roundtrips). N+1 problem: fetching a list then querying each item separately — fix with DataLoader (batches per request tick). Trade-offs: harder to cache (all POSTs to one endpoint), introspection exposure, complexity. Persisted queries help caching. Use for client-driven data needs; overkill for simple APIs.
  • gRPC
    Uses Protocol Buffers (strongly-typed IDL + binary serialization — much smaller than JSON). Runs over HTTP/2. Code generated from .proto files in any supported language. 4 streaming modes: Unary (1 req → 1 resp), Server Streaming, Client Streaming, Bidirectional Streaming. ~5–7× smaller payload and faster than REST+JSON. Ideal for internal microservice communication. Harder to debug (binary), requires HTTP/2, limited browser support (use gRPC-Web).
  • WebSockets
    HTTP upgrade handshake (Upgrade: websocket, 101 response) establishes a persistent full-duplex TCP connection. Low latency, bidirectional. Use for: live chat, collaborative editing, live dashboards, multiplayer games. Scaling challenge: connections are stateful, so horizontal scaling needs sticky sessions or a pub/sub broker (Redis Pub/Sub, Kafka) so any server can fan out to all clients. ws:// (plain) / wss:// (TLS). Consider SSE (Server-Sent Events) for server→client-only streams — simpler and HTTP/2-native.

Network Fundamentals

  • TLS Handshake
    TLS 1.3 (1-RTT): ClientHello (supported ciphers, key share) → ServerHello (chosen cipher, key share, certificate) → client verifies cert → Finished (both sides derive session keys). Perfect Forward Secrecy (PFS): ephemeral keys per session — compromising the server's private key later can't decrypt past sessions. Certificate chain: leaf cert → intermediate CA → root CA (trusted by OS/browser). HSTS (Strict-Transport-Security): browser always uses HTTPS for the domain. Let's Encrypt: free, automated DV certs via ACME protocol.
  • DNS
    Resolution chain: OS cache → stub resolver → recursive resolver (e.g. 8.8.8.8) → root nameserver → TLD nameserver (.com) → authoritative nameserver → answer. Record types: A (IPv4), AAAA (IPv6), CNAME (alias, can't be at root/apex), MX (mail), TXT (SPF/DKIM/verification), NS (nameservers), SOA (zone metadata). TTL controls caching — lower TTL = faster propagation but more DNS load. Negative TTL caches NXDOMAIN.
  • TCP vs UDP & OSI Model
    TCP: connection-oriented (3-way handshake: SYN → SYN-ACK → ACK), reliable delivery (retransmits), ordered, flow control (sliding window), congestion control (slow start, AIMD). UDP: connectionless, no guarantee of delivery/order, low overhead — good for DNS, video streaming, games, QUIC. OSI 7 layers (mnemonic: Please Do Not Throw Sausage Pizza Away): Physical, Data Link, Network, Transport, Session, Presentation, Application. TCP/IP 4 layers: Network Access, Internet (IP), Transport (TCP/UDP), Application (HTTP).

Quick Review

Why can't you cache POST requests but you can cache GET requests?

GET is defined as safe (no side effects) and idempotent, so caches (browser, CDN, proxy) can safely store and reuse the response. POST is neither safe nor idempotent — the same request sent twice could create two resources, so intermediate caches must not store POST responses by default (though the response can include Cache-Control to allow it explicitly).

What exactly does HTTP/2 multiplexing solve?

HTTP/1.1 suffers from head-of-line (HOL) blocking at the application layer: if a large resource is in flight on a connection, subsequent requests on that same connection have to wait. Browsers work around this by opening 6–8 parallel connections per origin — wasteful.

HTTP/2 multiplexing lets many independent streams share a single TCP connection. A slow stream doesn't block others. This eliminates the need for domain sharding and request bundling hacks. However, HOL blocking still exists at the TCP layer — a lost packet stalls all streams, which is why HTTP/3 (QUIC/UDP) was needed.

What is the CORS preflight and when does it happen?

The browser sends an OPTIONS preflight request before the actual cross-origin request when any of these are true:

  • Method is not GET, POST, or HEAD
  • Content-Type is not application/x-www-form-urlencoded, multipart/form-data, or text/plain
  • Custom headers are included (e.g. Authorization)

The preflight checks whether the server allows the actual request. The server responds with Access-Control-Allow-* headers. Access-Control-Max-Age caches the preflight result to avoid sending it every time.

When would you choose gRPC over REST?

gRPC is a strong choice for internal service-to-service communication where you control both ends: lower latency (binary protobuf vs JSON), strict contracts (proto schema = compile-time type safety), streaming support, and generated client code. The 5–7× smaller payload matters at scale.

Stick with REST for public APIs: easier discoverability, universal tooling, human-readable, no special library needed. REST also wins when browser clients are the primary consumer (gRPC requires gRPC-Web + a proxy layer).

What's the difference between 401 and 403?

401 Unauthorized means unauthenticated — the server doesn't know who you are. The response must include a WWW-Authenticate header indicating how to authenticate. Despite the name, it's about identity, not permission.

403 Forbidden means unauthorized — the server knows who you are but you don't have permission to access the resource. No authentication challenge is sent.
