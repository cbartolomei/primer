🔍 Search

Full-text search, relevance ranking, and the engine internals that make Elasticsearch fast. When to use search, how to design indexes, and the patterns that power autocomplete, faceted filtering, and geo queries.

📑

How Search Works: Inverted Indexes

Why LIKE is slow: SELECT * FROM posts WHERE body LIKE '%javascript%' performs a full table scan — every row must be read, every character checked. No index can help a leading-wildcard LIKE. At 10M rows it's seconds; at 100M rows it's minutes. Full-text search exists to solve this.

The inverted index: the core data structure behind every search engine. Instead of storing documents and scanning them, build a mapping from terms to the list of documents containing them.

Document 1: "Kafka processes events in real time"
Document 2: "Real-time event processing with Flink"
Document 3: "Kafka streams and Flink both process events"

Inverted index:
kafka     → [doc1, doc3]
processes → [doc1]
events    → [doc1, doc2, doc3]
real      → [doc1, doc2]
time      → [doc1, doc2]
event     → [doc2, doc3]
flink     → [doc2, doc3]
streams   → [doc3]
both      → [doc3]
process   → [doc3]

A query for "kafka events" intersects the posting lists for kafka ([doc1, doc3]) and events ([doc1, doc2, doc3]) → [doc1, doc3]. Both posting lists are sorted by doc ID — intersection is O(N) via merge.

Posting lists: each entry in the inverted index maps a term to a posting list — an ordered list of document IDs that contain that term. Each entry can also store term frequency (how many times the term appears in that doc), field position (for phrase queries — "exact phrase" matching requires positions), and other metadata.

Segments: Lucene (the library underlying Elasticsearch) writes index data in immutable segments. A segment is a complete mini-index. New documents go into an in-memory buffer, then flushed as a new segment. Segments are merged periodically (fewer segments = faster queries). Deleted documents are marked with a tombstone in the segment's delete bitset — they're excluded from results until the segment is merged away. This is why deletes don't immediately free space.

Near-real-time search: Lucene's refresh operation flushes the in-memory buffer to a new (unsearched) segment and makes it visible. Default refresh interval in Elasticsearch: 1 second. This means a document is searchable ~1s after indexing. Not instant, but close. Refresh is cheap (no fsync); flush (durability) is expensive (fsync to disk).

Why search scales differently than databases: a database update is in-place — find the row, change bytes. A search index update means: delete old document (tombstone), index new document (new segment). High update rates produce many small segments and many tombstones. Elasticsearch merge policies handle this, but write-heavy workloads require tuning (larger index.refresh_interval, bulk indexing, index.merge.policy).

Relevance Scoring: TF-IDF & BM25

The problem: you have 10,000 documents containing the word "java". Which should rank highest for the query "java concurrency"?

TF-IDF (Term Frequency–Inverse Document Frequency): the classic scoring formula, used by Lucene for decades.

  • TF (term frequency): how often the term appears in this document. More occurrences → higher score. A document mentioning "concurrency" 10 times is more about concurrency than one mentioning it once.
  • IDF (inverse document frequency): how rare the term is across all documents. A term appearing in 90% of documents (like "the") is useless for discrimination — low IDF. A term appearing in 0.1% of documents is highly discriminating — high IDF.
  • Score = TF × IDF (roughly). In practice, TF is sqrt-dampened and IDF is log-scaled.

BM25 (Best Match 25): the current default in Elasticsearch (since version 5). Improves TF-IDF in two ways:

  • TF saturation: TF-IDF reward grows unboundedly with more occurrences. BM25 applies saturation — a document mentioning a term 100× is not 10× better than one mentioning it 10×. Controlled by parameter k1 (default 1.2).
  • Field length normalization: a term in a short title is more significant than the same term buried in a 10,000-word document. BM25 normalizes by field length. Controlled by parameter b (default 0.75, 0=no normalization, 1=full normalization).
BM25 score = IDF × (TF × (k1 + 1)) / (TF + k1 × (1 - b + b × fieldLen/avgFieldLen))

Field boosting: different fields should contribute differently to relevance. A match in the title is more important than a match in the body. Boost title matches 3×:

{
  "query": {
    "multi_match": {
      "query": "java concurrency",
      "fields": ["title^3", "body", "tags^2"]
    }
  }
}

Function score queries: combine text relevance with business signals. Boost newer documents, boost documents with more clicks, decay score by distance from user location. Example: news search — a highly relevant but week-old article should score lower than a slightly less relevant but fresh article.

{
  "function_score": {
    "query": { "match": { "body": "kafka" } },
    "functions": [
      { "gauss": { "published_at": { "origin": "now", "scale": "7d", "decay": 0.5 } } }
    ]
  }
}

When scoring matters less than you think: for many use cases, filtering is more important than ranking. Searching a product catalog for "blue 512GB SSD under $200" — the filter (category, color, storage, price) narrows results to 5 products, and ranking within 5 is barely noticed. BM25 tuning matters most when result counts are high and users care about rank.

Analyzers & Text Processing

What an analyzer does: transforms raw text into terms (tokens) for the inverted index. Applied at index time AND at query time (using the same analyzer so they match). Three stages: character filters → tokenizer → token filters.

Character filters: preprocessing on raw text. Strip HTML tags (html_strip), replace characters (convert & to and), apply regex substitutions. Run before tokenization.

Tokenizers: split text into tokens. standard tokenizer: splits on whitespace and punctuation, lowercases. whitespace: splits only on whitespace (preserves punctuation). keyword: no splitting — the entire field value is one token. ngram / edge_ngram: generate character substrings (used for prefix/autocomplete).

Token filters: transform individual tokens. lowercase (essential — "Java" and "java" should match). stop (remove stopwords — "the", "a", "is"). stemmer (reduce to root form — "running" → "run", "bikes" → "bike"). synonym (expand — "auto" → ["auto", "car", "automobile"]). ascii_folding (convert "café" → "cafe").

The standard analyzer (Elasticsearch default): standard tokenizer + lowercase + stop words. Good for English prose, bad for code, URLs, emails, product SKUs.

Custom analyzer example — for a code documentation search:

{
  "settings": {
    "analysis": {
      "analyzer": {
        "code_analyzer": {
          "tokenizer": "standard",
          "filter": ["lowercase", "word_delimiter_graph"]
        },
        "autocomplete_analyzer": {
          "tokenizer": "standard",
          "filter": ["lowercase", "autocomplete_filter"]
        },
        "autocomplete_search_analyzer": {
          "tokenizer": "standard",
          "filter": ["lowercase"]
        }
      },
      "filter": {
        "autocomplete_filter": {
          "type": "edge_ngram",
          "min_gram": 2,
          "max_gram": 20
        }
      }
    }
  }
}

Language-specific analyzers: Elasticsearch ships with language analyzers for 30+ languages. english analyzer applies Porter stemmer + English stopwords. german handles compound words (German: Kraftfahrzeugbrief → kraft, fahr, zeug, brief). Always use language-specific analyzers for user-facing text in that language.

Testing analyzers: always test before indexing data:

GET /my-index/_analyze
{
  "analyzer": "english",
  "text": "The running foxes are quickly jumping"
}
# Returns: [run, fox, quick, jump]

The search-as-you-type field type: Elasticsearch's built-in solution for prefix matching. Creates sub-fields with edge_ngram indexing automatically. Simpler than a custom analyzer for basic autocomplete.

Elasticsearch in Practice

Index design: an index is Elasticsearch's equivalent of a database table. Key decisions:

  • Shards: an index is split into N primary shards (default 1 since ES 7). Each shard is a Lucene index. More shards = more parallelism for queries and indexing, but more overhead. Rule of thumb: shards of 10–50GB. Over-sharding is a common mistake.
  • Replicas: copies of each shard for fault tolerance and read throughput. number_of_replicas: 1 = 2 copies total. All shards (primary + replica) are involved in search queries.
  • Mappings: define field types explicitly. Dynamic mapping (auto-detection) is dangerous in production — Elasticsearch might infer a field as long when you need keyword, or index a JSON blob and create hundreds of unintended fields ("mapping explosion"). Always define explicit mappings.

Field types:

  • text: analyzed, full-text searchable, not usable for exact match, sort, or aggregations
  • keyword: not analyzed, exact match and aggregations. Use for: IDs, status fields, tags, email addresses, URLs.
  • integer/long/float/double: numeric
  • date: ISO 8601 or custom format, supports range queries
  • boolean
  • nested: array of objects where each object should be independently searchable (avoids the "object flattening" problem)
  • dense_vector: for vector/semantic search (kNN)

A realistic mapping:

{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "english",
        "fields": {
          "keyword": { "type": "keyword" }
        }
      },
      "body": { "type": "text", "analyzer": "english" },
      "author_id": { "type": "keyword" },
      "tags": { "type": "keyword" },
      "published_at": { "type": "date" },
      "view_count": { "type": "integer" }
    }
  }
}

Query DSL essentials:

// Full-text search across fields
{ "multi_match": { "query": "kafka streams", "fields": ["title^2", "body"] } }

// Exact match (keyword field)
{ "term": { "status": "published" } }

// Multiple exact matches
{ "terms": { "tags": ["kafka", "streaming"] } }

// Range
{ "range": { "published_at": { "gte": "2024-01-01", "lt": "2025-01-01" } } }

// Boolean combination
{
  "bool": {
    "must": [{ "match": { "body": "kafka" } }],
    "filter": [{ "term": { "status": "published" } }],
    "should": [{ "term": { "tags": "featured" } }],
    "must_not": [{ "term": { "is_deleted": true } }]
  }
}

filter clauses don't affect scoring and are cached — always use filter for non-relevance conditions (status, date range, user_id). must clauses affect scoring — use for text search. This is the most important query optimization pattern.

Aggregations: Elasticsearch's analytics capability. Run alongside search queries.

{
  "query": { "match": { "body": "kubernetes" } },
  "aggs": {
    "by_tag": { "terms": { "field": "tags", "size": 10 } },
    "over_time": {
      "date_histogram": {
        "field": "published_at",
        "calendar_interval": "month"
      }
    }
  }
}

Indexing performance:

  • Bulk API: never index one document at a time. Use _bulk API — batch 1000–5000 documents per request.
  • Disable refresh during bulk load: PUT /index/_settings {"index.refresh_interval": "-1"}. Re-enable after.
  • Increase index.number_of_replicas to 0 during initial load, restore after.
  • _source storage: Elasticsearch stores the original JSON (_source) for every document. Required for returning results, but doubles storage. Can be disabled if you only need to retrieve from another store.

Search Patterns

Autocomplete / prefix search: show suggestions as the user types. Two approaches:

  1. Edge n-gram: index "kafka" as ["ka", "kaf", "kafk", "kafka"] at index time. At query time, search for the exact typed prefix. Fast at query time, large index size. Best for: predictable prefixes, single-field autocomplete.
  2. Completion suggester: Elasticsearch's built-in autocomplete field type. Stores an FST (finite state transducer) in memory for very fast prefix lookups. Returns documents sorted by weight. Doesn't support BM25 scoring or filters well. Best for: simple autocomplete with explicit weights (e.g., search by popularity).
// Completion suggester query
{
  "suggest": {
    "title-suggest": {
      "prefix": "kaf",
      "completion": {
        "field": "title.suggest",
        "size": 5,
        "contexts": {
          "status": ["published"]
        }
      }
    }
  }
}

Faceted search / aggregations: the filtering sidebar on e-commerce sites ("Brand: Nike (142), Adidas (89)"). The count next to each facet is an aggregation on the filtered result set. Key pattern: use post_filter to filter search results WITHOUT affecting aggregations (so facet counts show what's available if you change the filter, not just what matches the current filter).

{
  "query": { "match_all": {} },
  "post_filter": { "term": { "brand": "Nike" } },
  "aggs": {
    "all_brands": { "terms": { "field": "brand", "size": 20 } },
    "price_ranges": {
      "range": {
        "field": "price",
        "ranges": [{"to": 50}, {"from": 50, "to": 100}, {"from": 100}]
      }
    }
  }
}

Geo search: search within a radius of a point, sort by distance, filter by bounding box.

// All restaurants within 5km of a point
{
  "query": {
    "bool": {
      "filter": {
        "geo_distance": {
          "distance": "5km",
          "location": { "lat": 40.7128, "lon": -74.0060 }
        }
      }
    }
  },
  "sort": [
    { "_geo_distance": { "location": { "lat": 40.7128, "lon": -74.0060 }, "order": "asc" } }
  ]
}

Field type must be geo_point. Elasticsearch uses BKD trees (space-filling curves) for efficient geo queries.

Fuzzy search / typo tolerance: match "elasticserach" when the user meant "elasticsearch". Elasticsearch supports fuzziness: "AUTO" which allows 1 edit (insert/delete/substitute/transpose) for strings 3–5 chars, 2 edits for 6+ chars.

{ "match": { "title": { "query": "elasticserach", "fuzziness": "AUTO" } } }

Fuzzy matching is expensive — it generates many candidate terms. Use on specific fields (title, name), not large text bodies.

Highlighting: return snippets showing where the query terms matched, with terms highlighted. Used to show "why" a result matched.

{
  "query": { "match": { "body": "kafka streams" } },
  "highlight": {
    "fields": {
      "body": { "fragment_size": 150, "number_of_fragments": 3 }
    }
  }
}

When to Use Search

Database LIKE vs full-text search:

ScenarioUse
WHERE name = 'Alice'Database (exact, indexed)
WHERE status IN ('active', 'pending')Database (exact, indexed)
WHERE title LIKE 'kafka%' (prefix only)Database (B-tree covers this)
WHERE body LIKE '%kafka%'Full-text search
"Find articles about Kafka and streaming"Full-text search
Ranked results by relevanceFull-text search
Autocomplete from millions of valuesFull-text search or Redis sorted sets
Faceted filters with countsFull-text search (aggregations)

PostgreSQL full-text search: built-in, no extra infrastructure. tsvector and tsquery types, GIN index on the vector. Good for: moderate data (< 10M rows), basic relevance, single-language, tight PostgreSQL integration. Limitations: no distributed scale, limited relevance tuning, no geo search, no aggregations. Start here if your search needs are simple.

-- PostgreSQL full-text search
ALTER TABLE articles ADD COLUMN search_vector tsvector;
UPDATE articles SET search_vector = to_tsvector('english', title || ' ' || body);
CREATE INDEX ON articles USING GIN(search_vector);

SELECT title, ts_rank(search_vector, query) AS rank
FROM articles, to_tsquery('english', 'kafka & streams') query
WHERE search_vector @@ query
ORDER BY rank DESC;

Elasticsearch operational considerations:

  • Separate cluster from your operational database — Elasticsearch heap sizes, GC pauses, and disk I/O patterns are different from OLTP workloads.
  • Heap: JVM heap at 50% of RAM, max 32GB (beyond 32GB, JVM loses compressed oops and memory efficiency degrades sharply).
  • Cluster sizing: 3 master-eligible nodes minimum for quorum. Data nodes sized by data volume and query throughput.
  • OpenSearch: AWS's open-source fork of Elasticsearch. API-compatible, often the choice when running on AWS.
  • Managed services: Elastic Cloud, OpenSearch Service (AWS), Bonsai — eliminate operational burden at a cost premium.

Vector search / semantic search: traditional keyword search fails for semantic queries ("articles about distributed systems fault tolerance" won't match documents that use different vocabulary). Vector search: embed documents and queries into high-dimensional vectors (using a model like OpenAI's embeddings or sentence-transformers), then find the nearest neighbors. Elasticsearch supports dense_vector + kNN search. Combine with keyword search for hybrid: keyword for exact terms, vector for semantic similarity. This is the architecture behind most modern AI-powered search features.

The synchronization problem: your Elasticsearch index is a derived view of data that lives in your database. When data changes, you must update the index. Three patterns:

  1. Synchronous dual-write: write to DB + index in same request. Simple, but Elasticsearch failure fails the whole operation, and no transaction spanning both.
  2. Async via message queue: write to DB, publish change event, consumer updates index. Eventual consistency (~seconds), more resilient.
  3. CDC (Change Data Capture): read DB replication log, apply changes to index. Most robust, no application coupling. Use Debezium → Kafka → Elasticsearch sink connector.
