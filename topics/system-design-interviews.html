🎯 System Design Interviews

Worked examples of the most common system design questions — URL shorteners, rate limiters, notification systems, and social feeds. The framework and the tradeoffs interviewers are actually looking for.

🗺️

The Framework

What interviewers are actually evaluating: not whether you get "the right answer" (there isn't one), but whether you can: scope a problem, identify the right tradeoffs for the given scale, make defensible decisions, and communicate clearly while thinking. A junior engineer produces a solution; a senior engineer produces a solution with explicit reasoning about what was sacrificed.

The 4-Step Framework

Step 1 — Clarify requirements (5 min). Never start designing immediately. Ask until you know:

  • Scale: DAU, QPS (read vs write separately), data size, growth rate
  • Latency requirements: p99 < 100ms? Or is eventual okay?
  • Availability: 99.9% (8.7h downtime/year) vs 99.99% (52 min/year)?
  • Consistency: must reads always reflect latest write? Or eventual?
  • Existing constraints: must use AWS? Specific languages? Existing systems to integrate?
  • What's in scope: "Design Twitter" is infinite — pin down which features

Step 2 — Capacity estimation (3-5 min). Back-of-envelope math. This drives your architecture choices.

  • QPS: DAU × actions_per_day / 86400. 100M DAU × 10 actions = ~12K QPS average, ~50K peak.
  • Storage: object_size × objects_per_day × retention_days
  • Bandwidth: QPS × response_size
  • Don't be precise — orders of magnitude matter. 10K QPS means distributed; 100 QPS means a single server.

Step 3 — High-level design (10 min). Start with a simple design that works, then scale it.

  • Start with: clients → load balancer → application servers → database
  • Identify the bottlenecks: reads? writes? storage? compute?
  • Add components to address each bottleneck: cache for reads, queue for write smoothing, CDN for static assets, read replicas for read scale, sharding for write scale

Step 4 — Deep dive (15 min). Pick 2-3 of the most interesting/risky components and go deep:

  • Data model: exact schema, how queries are served
  • Algorithm: how does the core logic work
  • Failure scenarios: what happens if component X dies?
  • Tradeoffs: why this choice over the alternatives?

What to Say Out Loud

Narrate your thinking. "I'm starting simple and will add complexity as needed." "This choice trades consistency for availability, which is acceptable here because..." "I'm going to flag this as a potential bottleneck and come back to it." Silence for 2 minutes while drawing is bad; silence with narration ("let me think through the write path") is fine.

Design a URL Shortener

This is the "hello world" of system design. Use it to practice the framework end-to-end.

Clarify

  • 100M new URLs/day (1150 writes/sec)
  • 10:1 read:write ratio → 11,500 reads/sec
  • Links live for 10 years
  • Short code: 7 characters (base62: a-z, A-Z, 0-9 → 62^7 = 3.5 trillion unique codes)
  • No custom aliases for now

Capacity

  • Storage: 100M URLs/day × 365 × 10 = 365B URLs × 500 bytes = ~180TB over 10 years
  • Read QPS: ~11,500/s → single cache-backed service handles this

Core Design

User → Load Balancer → App Servers → PostgreSQL (source of truth)
                              ↓
                          Redis cache (short_code → long_url)

Generating Short Codes

Three approaches:

  1. Hash-based: MD5/SHA1 the long URL, take first 7 characters. Problem: collisions. Two URLs could hash to the same 7-char prefix. Need collision detection and retry. Also: same URL twice produces same short code (good or bad depending on requirements).
  2. Random ID: generate a random 7-char base62 string, check uniqueness in DB. Problem: at scale, collision probability grows (birthday problem). At 3.5T space with 365B URLs: ~10% collision rate by year 10. Still manageable with retry.
  3. Counter-based (recommended): maintain a global counter (or per-service counter ranges). Encode the counter as base62. 0→"a", 1→"b", 62→"ba". Guarantees uniqueness, no collision checks. Problem: predictable (users can enumerate URLs) and requires coordination (distributed counter needs a service like Redis INCR or a ticket server like Flickr's approach).

Data Model

CREATE TABLE urls (
  id BIGINT PRIMARY KEY,           -- the counter value
  short_code VARCHAR(8) NOT NULL,
  long_url TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ,
  user_id BIGINT                   -- nullable, for analytics
);
CREATE UNIQUE INDEX ON urls(short_code);
CREATE INDEX ON urls(long_url);    -- for deduplication if needed

The Redirect Flow

  1. Request: GET /abc1234
  2. Check Redis cache: GET url:abc1234 → cache hit → return 301/302 redirect
  3. Cache miss → query PostgreSQL → cache result with TTL → redirect
  4. 301 (permanent): browser caches, reduces future load, can't track clicks. 302 (temporary): browser doesn't cache, allows click tracking, slightly higher server load. Most URL shorteners use 302 for analytics.

Scaling to 10× Load (100K reads/sec)

  • Redis cluster (sharded by short_code) — cache hit rate ~99% at 10 years, so DB load stays low
  • CDN: put the redirect service behind CloudFront. Most redirects happen at the edge with zero origin hit.
  • Read replicas for DB: the 1% cache miss goes to a read replica
  • Writes stay on primary — 1150 writes/sec is fine for PostgreSQL

Custom domains: go.example.com/abc1234 — store the domain with the URL record, route by domain + code.

Design a Rate Limiter

Clarify

  • Limit API requests per user: 1000 req/10 min
  • Applied at API gateway level (before business logic)
  • Distributed: multiple gateway nodes must share state
  • Return 429 with Retry-After header when exceeded

Algorithms

Covered in-depth in the API Design topic — summary here for interview context:

  • Token bucket: N tokens, refill at R/sec. Each request consumes a token. Allows bursts up to N. Implementation: store {tokens, last_refill} in Redis, update on each request.
  • Sliding window log: store timestamps of all requests in last window. Count timestamps. Exact but memory-intensive (store every request timestamp).
  • Sliding window counter (recommended): divide into fixed windows, interpolate between current and previous window count. count = prev_count × (1 - elapsed/window) + curr_count. Memory efficient, approximately accurate.

Storage

Redis is the standard choice. Single INCR operation is atomic. No race conditions. Example using Redis for fixed window:

Key: rate_limit:{user_id}:{window_start_unix}
Value: INCR → atomic increment, EXPIRE → auto-cleanup

On each request:
  key = "rl:user123:$(floor(now/600))"  # 10-minute window
  count = INCR key
  if count == 1: EXPIRE key 600
  if count > 1000: return 429

Sliding Window with Sorted Set (More Accurate)

Key: rate_limit:{user_id}
Operation:
  now = current timestamp (ms)
  window_start = now - 600000ms
  MULTI/EXEC:
    ZREMRANGEBYSCORE key 0 window_start   # remove old entries
    ZADD key now now                       # add current request
    ZCARD key                              # count in window
    EXPIRE key 600
  if count > 1000: return 429

Atomic via Lua script or Redis transaction.

Distributed Rate Limiter Design

Request → API Gateway (any node) → Redis Cluster → Allow/Deny
                                 (rate limit state)

All gateway nodes share the same Redis cluster. Consistent rate limits regardless of which node handles the request. Redis Cluster shards by user_id key — all requests for a user go to the same shard.

Headers to Return

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 1712345678    # Unix timestamp when window resets
Retry-After: 42                  # seconds until retry (on 429)

Edge Cases to Discuss

  • What if Redis is down? Fail open (allow requests) or fail closed (deny all). Usually fail open — availability > strict limiting for most APIs.
  • Distributed clock skew? Small skew (ms) is acceptable for rate limiting. Use Redis time (TIME command) as the authoritative clock.
  • Per-endpoint limits? Store rl:{user_id}:{endpoint} or rl:{user_id} for global. Separate keys.
  • IP-based limiting for unauthenticated routes? Same pattern, key by IP.

Design a Notification Service

Clarify

  • Channels: push (iOS, Android), email, SMS
  • Volume: 10M notifications/day (~115/sec average, 1000/sec peak)
  • Delivery: best-effort (not transactional — missing a "like" notification is okay)
  • Priority: some notifications are high priority (security alerts), most are normal
  • Personalization: user preferences (opt-out by channel/type)

High-Level Flow

Triggering service → Notification Service API → Queue → Workers → 3rd party providers
                                                         ├── iOS: APNs (Apple)
                                                         ├── Android: FCM (Google)
                                                         ├── Email: SendGrid / SES
                                                         └── SMS: Twilio

Why a queue between API and workers: the triggering service (e.g., "user got a like") just needs to enqueue a notification request and return. Workers pick up jobs and call the slow 3rd-party APIs (APNs, FCM, etc.). Decouples throughput from 3rd-party latency. Allows retry on failure. Separate queues per channel (email queue, push queue) — an APNs outage doesn't block email sending.

Data Model

-- User notification preferences
CREATE TABLE user_preferences (
  user_id BIGINT,
  channel VARCHAR(10),       -- 'push', 'email', 'sms'
  notif_type VARCHAR(50),    -- 'like', 'comment', 'security'
  enabled BOOLEAN DEFAULT true,
  PRIMARY KEY (user_id, channel, notif_type)
);

-- Device tokens for push
CREATE TABLE device_tokens (
  user_id BIGINT,
  token TEXT,
  platform VARCHAR(10),     -- 'ios', 'android'
  created_at TIMESTAMPTZ
);

-- Notification log (for deduplication and analytics)
CREATE TABLE notifications (
  id UUID PRIMARY KEY,
  user_id BIGINT,
  type VARCHAR(50),
  channel VARCHAR(10),
  status VARCHAR(20),        -- 'pending', 'sent', 'failed', 'unsubscribed'
  sent_at TIMESTAMPTZ,
  payload JSONB
);

Worker Logic

  1. Pull from queue
  2. Load user preferences — skip if user opted out of this channel/type
  3. Load device tokens (for push) or contact info (email/SMS)
  4. Call 3rd-party provider (APNs, FCM, SendGrid)
  5. Handle response: mark sent, or retry with backoff on failure
  6. Handle unsubscribe/invalid token responses from providers (FCM returns NotRegistered for uninstalled apps → delete that token)

Retry Strategy

Exponential backoff with jitter. 3rd-party providers rate limit and have transient failures. Dead-letter queue after N retries — alert the ops team, don't retry indefinitely.

Deduplication

If the triggering service sends the same notification twice (retry on its end), you must not send it twice. Deduplicate on (user_id, notif_type, reference_id) — e.g., (user_123, 'like', post_456). Check against the notification log before enqueueing.

Fan-Out for Broadcast Notifications

Sending to 100M users → can't do this in a single job. Partition users (by ID range), spawn parallel workers. Pre-compute batches during off-peak hours. Track progress so a crash can resume.

Priority Queues

Separate high-priority queue (security alerts: password change, login from new device) processed by dedicated workers with no throttling. Normal queue can shed load during peaks.

Design a Social Feed

The news feed problem (Twitter/Instagram home timeline) is a canonical example of read vs write amplification tradeoffs.

Clarify

  • 100M DAU, 50M posts/day (580 writes/sec)
  • Each user follows ~200 others on average
  • Feed shows most recent 1000 posts from followed users
  • Read:write ratio = 100:1 (users scroll far more than they post)
  • Latency: feed must load in < 200ms

The Naive Approach — Pull Model (Fan-Out on Read)

On each feed request: get all followees → query their posts → merge and sort.

Problem: following 200 users = 200 DB queries per feed load. At 100M DAU and 100 feed loads/day = 10B queries/day = 115K QPS just for feed reads. Even with caching, this doesn't work at scale.

Push Model (Fan-Out on Write)

When a user posts, push the post to every follower's feed cache immediately.

User posts → write to DB → publish event → fan-out worker →
  for each follower: prepend post to follower's feed cache (Redis list)

Feed read = single Redis list read. O(1), sub-millisecond.

Problem: celebrity users (10M followers). A tweet from @elonmusk triggers 10M fan-out writes. At 580 celebrity tweets/day, that's 5.8B writes/day from fan-out alone.

Hybrid Approach (Twitter's Actual Solution)

  • Normal users (< N followers, e.g., 10K): fan-out on write. Their posts are pushed to all followers' feed caches immediately.
  • Celebrity users (> N followers): fan-out on read for their posts. Their posts are NOT pushed. When you load your feed, the service fetches recent posts from celebrity accounts you follow and merges them with your pre-computed feed.
Feed load:
  1. Fetch your pre-computed feed from Redis (fan-out-on-write posts)
  2. Fetch recent posts from your followed celebrities (fan-out-on-read)
  3. Merge and sort by timestamp
  4. Return top 20

The merge is cheap because celebrity count is small (you follow few celebrities) and their posts are cached globally (not per-follower).

Data Model

-- Posts table
CREATE TABLE posts (
  post_id BIGINT PRIMARY KEY,
  user_id BIGINT,
  content TEXT,
  created_at TIMESTAMPTZ,
  media_url TEXT
);
CREATE INDEX ON posts(user_id, created_at DESC);

-- Social graph (who follows whom)
CREATE TABLE follows (
  follower_id BIGINT,
  followee_id BIGINT,
  created_at TIMESTAMPTZ,
  PRIMARY KEY (follower_id, followee_id)
);
CREATE INDEX ON follows(followee_id);  -- "give me all followers of X"

Feed Cache (Redis)

  • Key: feed:{user_id}
  • Value: Redis sorted set, scored by timestamp, members are post IDs
  • Size limit: keep last 1000 post IDs. Older posts fetched from DB.
  • TTL: evict if user hasn't logged in for 30 days (don't fan-out to inactive users)

Ranking

Simple reverse chronological is easy. Algorithmic ranking (like Instagram/TikTok) scores posts by engagement signals (likes, comments, shares, watch time) weighted by recency and relationship strength. This requires a scoring service that runs models. Out of scope for basic interview, worth mentioning as a follow-up.

Deep-Dive Questions the Interviewer Might Ask

  • "What if the DB with posts goes down?" — read from cache, degrade gracefully for new posts
  • "How do you handle a new user with no follows?" — onboarding recommendations, trending posts
  • "User unfollows someone — remove their posts from the feed?" — lazy deletion (posts stay until feed is refreshed), or active fan-out of delete events
  • "How do you serve the feed globally?" — regional Redis clusters, geo-routing users to nearest region, cross-region replication of write events
