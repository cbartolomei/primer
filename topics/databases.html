Transaction Fundamentals

  • ACID Properties
    Atomicity: all operations in a transaction succeed or all are rolled back — no partial writes. Consistency: a transaction moves the database from one valid state to another, respecting all constraints. Isolation: concurrent transactions execute as if they were serial — one doesn't see the intermediate state of another. Durability: once committed, data survives crashes. Enabled by the Write-Ahead Log (WAL) — changes are logged before being applied to data pages.
  • Isolation Levels & Read Phenomena
    Read Uncommitted: can read dirty (uncommitted) data. Practically unused. Read Committed (PostgreSQL default): only reads committed data, but non-repeatable reads possible (re-reading a row may give different values within a transaction). Repeatable Read: same row always returns same value within a transaction; phantom reads (new rows matching a WHERE) possible. Serializable: full isolation, slowest. PostgreSQL uses SSI (Serializable Snapshot Isolation) — avoids locking by detecting conflicts.
  • Transactions & Locking
    BEGIN / COMMIT / ROLLBACK. SAVEPOINT name for partial rollbacks. Optimistic locking: read a version column, check it hasn't changed on update (WHERE version = $old). Good when conflicts are rare — no blocking. Pessimistic locking: SELECT ... FOR UPDATE acquires a row-level lock. Prevents concurrent modification but can cause deadlocks. Deadlock: T1 holds lock A, wants B; T2 holds B, wants A. PostgreSQL detects and aborts one.

Indexing & Query Optimization

  • Index Types
    B-tree (default): balanced tree, great for equality and range queries (=, <, BETWEEN, LIKE 'foo%'). Hash: O(1) equality lookups only, no range support. GIN (Generalized Inverted Index): for full-text search, JSONB containment, array operators. GiST: geometric types, full-text. BRIN: Block Range — tiny, great for naturally ordered columns (timestamps, auto-increment IDs). Covering index: CREATE INDEX ON orders(user_id) INCLUDE (status, created_at) — query satisfied entirely from index (no heap fetch). Massive speedup for frequently-run queries. Trade-off: every index slows writes (INSERT/UPDATE/DELETE must update all indexes) and uses disk.
  • EXPLAIN ANALYZE
    EXPLAIN ANALYZE SELECT ... actually executes the query and shows actual vs estimated rows, loop counts, and timings. Key nodes: Seq Scan (reads whole table — often bad on large tables), Index Scan (uses index then fetches heap), Index Only Scan (all data in index, best), Bitmap Index Scan (OR-ing multiple indexes). Join strategies: Nested Loop (small tables), Hash Join (unsorted, large sets), Merge Join (pre-sorted, good for ranges). Look for: high actual vs estimated row discrepancy (stale stats → run ANALYZE), unexpected Seq Scans, high loop counts.
  • JOIN Types
    INNER JOIN: only matching rows from both sides. LEFT OUTER JOIN: all rows from left + matching right (nulls where no match). RIGHT OUTER JOIN: inverse of LEFT. FULL OUTER JOIN: all rows from both, nulls where no match. CROSS JOIN: Cartesian product (every row × every row) — rarely intentional. Self-join: table joined to itself (e.g. employee → manager). Anti-join pattern: LEFT JOIN … WHERE right.id IS NULL — find rows with no match (cheaper than NOT IN / NOT EXISTS in some cases).

Design & Scaling

  • Normalization & Denormalization
    1NF: atomic values, no repeating groups. 2NF: 1NF + no partial dependency on composite primary key. 3NF: 2NF + no transitive dependency (non-key column depends only on the PK, not other non-key columns). BCNF: every determinant is a candidate key. Denormalization: intentional redundancy for read performance — store a computed count column, duplicate a name field to avoid a join. Trade-off: write complexity and inconsistency risk. OLAP schemas (star/snowflake) are intentionally denormalized for query speed.
  • Replication
    Primary-replica: all writes go to primary; replicas catch up via WAL streaming. Synchronous replication: write is only committed once at least one replica has confirmed — zero data loss, higher write latency. Asynchronous replication (default): lower latency, but a failover may lose a few transactions. Read replicas: scale read traffic, but reads may be slightly stale. Route reads that can tolerate stale data to replicas. Logical replication: replicate specific tables or filtered rows. Physical/streaming: byte-for-byte copy of WAL — can't replicate to different Postgres major version.
  • Sharding
    Horizontally partitioning data across multiple databases. Each shard holds a subset of rows. Shard key selection is critical: bad key → hot spots (one shard gets all traffic). Good keys have high cardinality and distribute writes evenly (e.g. user_id, not country_code). Strategies: Range-based (easy range queries, prone to hot spots), Hash-based (even distribution, hard to range query), Directory-based (lookup table, flexible but extra hop). Pitfalls: cross-shard JOINs are expensive/impossible, cross-shard transactions are hard, rebalancing when adding shards requires data migration.

NoSQL & Advanced PostgreSQL

  • NoSQL Types
    Document (MongoDB, Firestore): JSON/BSON documents, flexible schema, rich queries. Good for content, catalogs, user profiles. Key-Value (Redis, DynamoDB): simplest model, ultra-fast, limited queries. Good for sessions, caching, leaderboards. Wide-Column (Cassandra, Bigtable): rows have dynamic columns grouped into column families. Write-optimized (LSM tree), excellent for time-series, IoT. Queries must be designed around access patterns. Graph (Neo4j, Amazon Neptune): nodes + edges with properties. Best when relationships are first-class (social graphs, fraud detection, recommendation engines). Time-series (InfluxDB, TimescaleDB): optimized for append-only timestamped data with fast range + aggregation queries.
  • Window Functions
    Apply a function over a set of rows related to the current row without collapsing rows (unlike GROUP BY). ROW_NUMBER() — unique sequential number per partition. RANK() — gaps for ties; DENSE_RANK() — no gaps. LAG(col, n) / LEAD(col, n) — access previous/next row value. SUM() OVER (PARTITION BY ... ORDER BY ...) — running total. FIRST_VALUE() / LAST_VALUE() / NTH_VALUE(). Used for rankings, moving averages, running totals, detecting gaps in sequences.
  • PostgreSQL JSONB
    JSONB stores parsed binary (fast querying, deduplicates keys, loses key order/whitespace). JSON stores as-is text (fast insert, slow query — must parse every time). Key operators: -> (extract JSON), ->> (extract as text), #> (path), @> (containment), ? (key exists). Index JSONB with GIN for fast containment (@>) and key existence (?) queries. For frequently-queried fields, a generated column + regular B-tree index is often faster than GIN.
  • Connection Pooling
    PostgreSQL spawns a process per connection — opening connections is expensive. With 1000 app instances each holding 10 connections, you need 10,000 DB processes. PgBouncer modes: Session (1:1, client gets connection for session lifetime), Transaction (connection returned to pool after each transaction — recommended), Statement (after each statement, breaks multi-statement transactions). Sizing rule of thumb: connections = (2 × num_cpu_cores) + num_disk_spindles. Default max_connections in PostgreSQL is 100. In cloud: AWS RDS Proxy, PlanetScale's proxy, Supabase's PgBouncer are managed options.

Why is selecting a good shard key so important?

A bad shard key creates hot spots — one shard receives a disproportionate share of writes. Example: sharding by created_at day means all writes today go to one shard. Example: sharding by country means the US shard may get 40% of traffic.

A good key: high cardinality (many possible values), even write distribution, ideally part of common query predicates (so queries can be routed to one shard). user_id or a UUID is often ideal. Consistent hashing makes future rebalancing less painful than modulo-based assignment.

When should you choose NoSQL over PostgreSQL?

PostgreSQL is often the right default — it handles flexible schema (JSONB), full-text search, JSON APIs, time-series (with TimescaleDB), and graph-adjacent queries well.

Reach for NoSQL when: (1) write throughput exceeds what a single primary can handle and you need native horizontal write scaling (Cassandra, DynamoDB); (2) your data is inherently a document with no relational structure and schema flexibility is paramount; (3) you need sub-millisecond reads at extreme scale (Redis/DynamoDB for caching and hot paths); (4) your access patterns are fixed and known upfront (wide-column databases are designed around queries, not flexibility).

What's the difference between a clustered and non-clustered index?

A clustered index determines the physical order of rows on disk — there can only be one. In MySQL/InnoDB, the primary key is the clustered index; the table is the index. In PostgreSQL, there's no clustered index concept by default (heap is unordered), but you can run CLUSTER table USING index to physically reorder a table.

A non-clustered index (all indexes in PostgreSQL) is a separate structure that stores index key + a pointer (heap tuple ID) to the actual row. Fetching the row requires an extra lookup to the heap ("heap fetch"). A covering index avoids this by including all needed columns in the index itself.
