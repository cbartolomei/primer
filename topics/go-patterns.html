The conventions and patterns that distinguish Go written by an expert from Go written by someone who just learned the syntax. Learn to think in Go.

Interfaces in Practice

Interfaces are Go's most powerful tool for writing flexible, testable code. The key insight: define interfaces where you use them, not where you implement them. A caller defines the interface that describes exactly the behavior it needs — and any concrete type that has those methods satisfies it automatically.

Small Interfaces

The most influential interfaces in the standard library are tiny — often just one method. This is deliberate: small interfaces are easy to implement, easy to mock, and compose naturally:

// From the standard library — one method each
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Stringer interface {
    String() string
}

type error interface {
    Error() string
}

Anything that can be read satisfies io.Reader: a file, a network connection, a byte buffer, a gzip stream, a test string. Functions that accept io.Reader work with all of them — no changes needed when you swap a file for a network socket. This is the power of Go interfaces in practice.

// Works with any Reader — file, HTTP body, string, stdin, gzip...
func countLines(r io.Reader) (int, error) {
    scanner := bufio.NewScanner(r)
    count := 0
    for scanner.Scan() {
        count++
    }
    return count, scanner.Err()
}

// Call with a file
f, _ := os.Open("data.txt")
defer f.Close()
n, _ := countLines(f)

// Call with an HTTP response body
resp, _ := http.Get(url)
defer resp.Body.Close()
n, _ = countLines(resp.Body)

// Call in a test with a string
n, _ = countLines(strings.NewReader("line1\nline2\nline3"))

Accept Interfaces, Return Structs

The principle: function parameters should be interfaces (broad, flexible), function return values should be concrete types (specific, useful). Returning an interface forces callers to type-assert to access concrete methods — returning a struct lets them use everything directly:

// GOOD — accepts an interface, returns a concrete type
func NewCache(store Storage) *Cache {
    return &Cache{store: store}
}

// AVOID — returning an interface hides what the caller gets
func NewCache(store Storage) CacheInterface {
    return &Cache{store: store}
}

The exception: error. Always return the error interface, not a concrete error type — callers should use errors.Is/errors.As for inspection.

The Empty Interface

interface{} (or any since Go 1.18) has no methods — every type satisfies it. Before generics, it was the only way to write functions that work with any type. You lose type safety and pay for type assertions at runtime:

// Before generics — any loses type information
func Map(s []interface{}, fn func(interface{}) interface{}) []interface{} { ... }

// With generics (Go 1.18+) — type-safe, zero overhead
func Map[T, U any](s []T, fn func(T) U) []U {
    result := make([]U, len(s))
    for i, v := range s {
        result[i] = fn(v)
    }
    return result
}

doubled := Map([]int{1, 2, 3}, func(n int) int { return n * 2 })

In new code, prefer generics over any for containers and utilities. Keep any for cases where the type truly varies at runtime (JSON unmarshaling into unknown structure, plugin systems).

Interfaces for Testing

The most practical use of interfaces: replace real dependencies in tests with simple fakes:

// Define what the service needs — not the full DB client, just what's used
type UserRepository interface {
    GetByID(ctx context.Context, id int) (*User, error)
    Save(ctx context.Context, u *User) error
}

type UserService struct {
    repo UserRepository
}

// In tests — implement UserRepository with a fake
type fakeRepo struct {
    users map[int]*User
}

func (f *fakeRepo) GetByID(_ context.Context, id int) (*User, error) {
    u, ok := f.users[id]
    if !ok {
        return nil, ErrNotFound
    }
    return u, nil
}

func (f *fakeRepo) Save(_ context.Context, u *User) error {
    f.users[u.ID] = u
    return nil
}

func TestPromoteUser(t *testing.T) {
    repo := &fakeRepo{users: map[int]*User{1: {ID: 1, Name: "Alice"}}}
    svc := UserService{repo: repo}
    // test against the fake
}

Notice: the interface is defined in the UserService's package, not in a "repository" package. The service defines what it needs. Any type that satisfies that interface works.

Standard Library

Go's standard library covers a vast amount of ground with a consistently high-quality API. Knowing it well means you reach for a third-party dependency less often — and when you do, you understand what it's built on top of.

net/http: Building Servers

// Handler interface — implement this to handle HTTP requests
type Handler interface {
    ServeHTTP(ResponseWriter, *Request)
}

// HandlerFunc — adapter to use a function as a Handler
http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
    // r.Method, r.URL, r.Header, r.Body
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusOK)
    fmt.Fprintln(w, `{"status":"ok"}`)
})

// Use a ServeMux for routing
mux := http.NewServeMux()
mux.HandleFunc("GET /users/{id}", getUser) // Go 1.22+ method+path patterns
mux.Handle("/static/", http.FileServer(http.Dir("static")))

// Configure the server — don't use http.ListenAndServe in production
server := &http.Server{
    Addr:         ":8080",
    Handler:      mux,
    ReadTimeout:  5 * time.Second,
    WriteTimeout: 10 * time.Second,
    IdleTimeout:  120 * time.Second,
}
log.Fatal(server.ListenAndServe())
Always set timeouts on http.Server. The zero value means no timeout — a slow or malicious client can hold open a connection forever, exhausting your goroutine pool.

io & bufio

// io.Reader / io.Writer are the foundation of all I/O

// Copy between reader and writer
n, err := io.Copy(dst, src)         // copies until EOF
n, err = io.CopyN(dst, src, 1024)   // copies exactly N bytes

// Read all
data, err := io.ReadAll(r)          // read entire reader into memory

// Discard
n, err = io.Copy(io.Discard, resp.Body) // drain body

// bufio wraps Reader/Writer with buffering

for scanner.Scan() {
    line := scanner.Text()
}

writer := bufio.NewWriter(w)
fmt.Fprintln(writer, "hello")
writer.Flush() // don't forget to flush!

Testing

Go's built-in testing package is simple but powerful. No external test runner needed — go test discovers and runs all tests. The community has converged on table-driven tests as the dominant style.

Basic Tests

// File: calculator_test.go
package calculator

import "testing"

func TestAdd(t *testing.T) {
    result := Add(2, 3)
    if result != 5 {
        t.Errorf("Add(2, 3) = %d; want 5", result)
    }
}

// t.Fatal stops the test immediately — use when subsequent code would panic
// t.Error marks failure but continues
// t.Logf logs only when test fails or -v flag is used

Table-Driven Tests

The idiomatic Go pattern: define test cases as a slice of structs, range over them. One test function covers all cases, failure messages include the case name:

func TestDivide(t *testing.T) {
    tests := []struct {
        name    string
        a, b    float64
        want    float64
        wantErr bool
    }{
        {"normal", 10, 2, 5, false},
        {"negative divisor", 10, -2, -5, false},
        {"divide by zero", 10, 0, 0, true},
        {"zero numerator", 0, 5, 0, false},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got, err := Divide(tt.a, tt.b)
            if (err != nil) != tt.wantErr {
                t.Errorf("wantErr=%v but got err=%v", tt.wantErr, err)
                return
            }
            if !tt.wantErr && got != tt.want {
                t.Errorf("Divide(%v, %v) = %v; want %v", tt.a, tt.b, got, tt.want)
            }
        })
    }
}

t.Run creates a subtest with its own name. Run a specific subtest: go test -run TestDivide/divide_by_zero. Failures show the exact case that failed.

Test Helpers & cleanup

// t.Helper() marks a function as a test helper
// error messages show the caller's line, not the helper's
func assertEqual(t *testing.T, got, want interface{}) {
    t.Helper()
    if got != want {
        t.Errorf("got %v; want %v", got, want)
    }
}

// t.Cleanup registers a cleanup function — runs after the test
func TestWithTempFile(t *testing.T) {
    f, err := os.CreateTemp("", "test-*")
    if err != nil {
        t.Fatal(err)
    }
    t.Cleanup(func() { os.Remove(f.Name()) })
    // test with f...
}

// t.TempDir() creates a temp dir that's automatically removed
dir := t.TempDir()

Testify

The github.com/stretchr/testify package is used in most Go codebases for more expressive assertions:

import (
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func TestUser(t *testing.T) {
    user, err := GetUser(1)

    require.NoError(t, err)          // like t.Fatal — stops test on failure
    require.NotNil(t, user)

    assert.Equal(t, "Alice", user.Name) // t.Error — continues on failure
    assert.True(t, user.Active)
    assert.ErrorIs(t, err, ErrNotFound) // checks error chain
}

Use require when failure makes subsequent assertions meaningless (nil pointer, wrong type). Use assert when you want to see all failures in one run.

Idiomatic Go

Idiomatic Go is not about stylistic preference — it's about the patterns and conventions the community has converged on because they make Go code easier to read, maintain, and extend. Reading code written this way feels natural; reading code written against these conventions feels labored.

Naming Conventions

Go uses shorter names than most languages. This is not laziness — it's a considered convention:

  • Single-letter names for short-lived variables: i, j, k for loop indices; r for reader, w for writer, n for count, b for byte slice
  • Short, descriptive names at package scope: not userRepository, just repo; not httpRequestHandler, just handler
  • Acronyms stay all-caps: URL, HTTP, ID, API, not Url, Http, Id
  • Interface names: single-method interfaces are named by the method + -er: Reader, Writer, Stringer, Handler
  • Package names: lowercase, single word where possible, noun. Not utils, not helpers, not common — these names say nothing about what the package does
  • Receiver names: short, typically one or two characters, consistent across all methods of the type. For User, use u; for Cache, use c. Not self or this.

Error Handling Idioms

// The standard check immediately after the call
if err != nil {
    return fmt.Errorf("doThing: %w", err)
}

// Guard clauses — handle the error case first, keep happy path unindented
func processUser(id int) error {
    user, err := fetchUser(id)
    if err != nil {
        return fmt.Errorf("processUser: fetch: %w", err)
    }
    if !user.Active {
        return ErrInactiveUser
    }
    // happy path — unindented, easy to follow
    return save(user)
}

// avoid nested if-err chains:
// BAD
if err == nil {
    if result, err := step2(); err == nil {
        if err := step3(result); err == nil { ... }
    }
}

Prefer Composition

Go has no inheritance. The idiomatic alternative is composition — building complex behavior from simple, focused types:

// Composition via embedding + additional behavior
type TimedCache struct {
    Cache                  // embed Cache's methods
    ttl   time.Duration
}

func (t *TimedCache) Set(k string, v interface{}) {
    // override Set to add expiry logic
    t.Cache.Set(k, v)
    t.scheduleExpiry(k, t.ttl)
}

// Middleware pattern — wrap a handler with additional behavior
type LoggingMiddleware struct {
    next   http.Handler
    logger *log.Logger
}

func (l *LoggingMiddleware) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    start := time.Now()
    l.next.ServeHTTP(w, r)
    l.logger.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
}

Functional Options Pattern

When a constructor has many optional parameters, the functional options pattern is cleaner than a config struct and more extensible than positional arguments:

type Server struct {
    port    int
    timeout time.Duration
    logger  *log.Logger
}

type Option func(*Server)

func WithPort(port int) Option {
    return func(s *Server) { s.port = port }
}

func WithTimeout(d time.Duration) Option {
    return func(s *Server) { s.timeout = d }
}

func NewServer(opts ...Option) *Server {
    s := &Server{
        port:    8080,               // sensible defaults
        timeout: 30 * time.Second,
    }
    for _, opt := range opts {
        opt(s)
    }
    return s
}

// Clean call site — only specify what differs from defaults
srv := NewServer(
    WithPort(9000),
    WithTimeout(10 * time.Second),
)

Don't Over-Abstract

Go's philosophy is to keep things simple. Patterns that feel natural in Java or C++ often don't belong in Go:

  • Don't create interfaces for things with only one implementation — premature abstraction makes code harder to follow. Add the interface when you need a second implementation or need to test with a fake.
  • Don't create packages just to organize — a utils or helpers package that wraps a few functions adds a layer of indirection with no benefit. Keep the functions in the package that uses them.
  • Repetition is not always bad — duplicating 5 lines of similar code is often clearer than extracting a function that takes 4 parameters to handle all the variations. Go proverb: "A little copying is better than a little dependency."
  • Don't wrap errors you're going to ignore — if you're always swallowing the error or logging and returning, you're hiding bugs. Either handle it or propagate it.

Key Go Proverbs

Rob Pike's Go proverbs capture the philosophy better than any rule list:

  • "Don't communicate by sharing memory; share memory by communicating."
  • "The bigger the interface, the weaker the abstraction."
  • "Make the zero value useful." — design types so their zero value works correctly without explicit initialization.
  • "A little copying is better than a little dependency."
  • "Clear is better than clever." — code is read far more than it's written. Optimize for the reader.
  • "Errors are values." — treat errors as data you can inspect, wrap, and transform. Don't just check and discard.
  • "Documentation is for users." — document exported identifiers as if you're explaining them to someone who doesn't have access to your source code.
