Authentication & Authorization

  • AuthN vs AuthZ
    Authentication (AuthN): verifying who you are — identity. "Prove you are Alice." Authorization (AuthZ): determining what you can do — permissions. "Alice can read invoices but not delete them." Common mistakes: (1) checking AuthN but skipping AuthZ — assuming all logged-in users can do everything. (2) Trusting client-supplied role/permission claims without verifying server-side. (3) Failing open when an auth service is unreachable. Always fail closed.
  • Sessions vs JWT
    Session-based: server generates a session ID, stores session data server-side (in Redis/DB), client holds cookie. Easy to invalidate (delete from store). Stateful — scales horizontally with shared session store. JWT (JSON Web Token): header.payload.signature — base64url encoded (not encrypted by default — anyone can decode the payload!). Stateless — server just verifies signature, no DB lookup. Cannot be invalidated without a blocklist (negating stateless benefit). Use short expiry (15 min) + refresh token flow. Never store sensitive data in payload. Common errors: alg: none attack, accepting any algorithm, symmetric vs asymmetric key confusion.
  • OAuth 2.0 & OIDC
    OAuth 2.0 is an authorization framework (delegate access, not authentication). OIDC (OpenID Connect) adds identity on top — provides an ID token (JWT) with user info. Authorization Code Flow (web apps): redirect to IdP → user authenticates → code returned → exchange code for tokens on backend. Code is single-use, tokens never in browser history. PKCE (Proof Key for Code Exchange): for SPAs/mobile where there's no client secret. Generate code_verifier → hash → code_challenge. Prevents auth code interception. Client Credentials (M2M): service authenticates directly with client_id + secret. Always validate the state parameter to prevent CSRF on the redirect.
  • CORS
    Browser's Same-Origin Policy blocks cross-origin requests (origin = scheme + host + port). CORS allows servers to explicitly permit cross-origin access. Key response headers: Access-Control-Allow-Origin, -Allow-Methods, -Allow-Headers, -Allow-Credentials (if true, origin cannot be *), -Max-Age (preflight cache duration). Never set Access-Control-Allow-Origin: * with credentials — browsers will block it, and it's a security risk. Server-side validation: validate the Origin header on the server, don't rely solely on browser enforcement. APIs called server-to-server aren't subject to CORS.

Common Vulnerabilities

  • SQL Injection
    User input is interpreted as SQL. Classic: ' OR '1'='1 in a login form constructed via string concatenation. Prevention: always use parameterized queries (prepared statements) — the DB driver separates code from data. Never concatenate user input into SQL strings. ORMs protect by default but watch for raw query escape hatches (.raw(), query()). Also watch for second-order injection: data stored safely but then used unsafely in a later query. Additional layers: least-privilege DB users (app user can't DROP tables), WAF rules.
  • XSS (Cross-Site Scripting)
    Injecting malicious scripts into pages viewed by other users. Types: Stored: script saved in DB, served to all visitors (e.g. forum post). Reflected: script in URL, reflected in response (e.g. search results page). DOM-based: client-side JS writes attacker-controlled data to the DOM. Prevention: output encoding (escape HTML entities before rendering), never use innerHTML with untrusted data, use textContent instead. Content Security Policy (CSP) headers — restrict which scripts can run. HttpOnly cookies — prevent JS access to session cookies.
  • CSRF (Cross-Site Request Forgery)
    A malicious site tricks the victim's browser into making an authenticated request to your site (the browser automatically sends cookies). Example: <img src="https://bank.com/transfer?to=attacker&amount=1000">. Prevention: SameSite cookie attribute: SameSite=Strict (never cross-site) or SameSite=Lax (allows top-level navigations). Best first defense. CSRF token: include a secret token in forms, validate server-side. Token must be per-session or per-request. Check Origin / Referer headers. If using JSON APIs with Content-Type: application/json, simple CSRF is already blocked (preflight required).
  • OWASP Top 10 (2021)
    A01 Broken Access Control — most common. Missing authz checks, IDOR (Insecure Direct Object Reference: /invoice/1234 works for any user). A02 Cryptographic Failures — transmitting sensitive data in plaintext, weak algorithms (MD5/SHA1 for passwords — use bcrypt/argon2). A03 Injection — SQL, NoSQL, OS command, LDAP injection. A04 Insecure Design — security must be designed in, not bolted on. A05 Security Misconfiguration — default creds, debug mode in prod, verbose errors. A06 Vulnerable Components — outdated dependencies with known CVEs. A07 Auth Failures — broken auth, weak passwords, no brute-force protection. A08 Software Integrity Failures — unsigned updates, CI/CD compromise. A09 Security Logging Failures — no audit trail for security events. A10 SSRF — Server-Side Request Forgery: attacker makes server fetch attacker-controlled URL (e.g. EC2 metadata endpoint).

Transport & Secrets

  • TLS Certificates
    Certificate lifecycle: generate private key → create CSR (Certificate Signing Request) → CA signs it → install cert + intermediate chain. Types: DV (Domain Validation — automated, Let's Encrypt), OV (Organization Validation), EV (Extended Validation — company name in cert, expensive). Let's Encrypt: free DV certs, 90-day expiry, ACME protocol for automation (certbot). HSTS: Strict-Transport-Security: max-age=31536000; includeSubDomains; preload — browser refuses HTTP connections. Preload list is baked into browsers. Certificate Transparency (CT): all publicly trusted certs must be logged — detects misissuance.
  • Secrets Management
    Never hardcode secrets in source code — git history is permanent. git log -S "password" will find it. Environment variables: baseline protection. Still end up in process dumps, logs, error reports if not careful. Secrets managers: AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, Azure Key Vault — provide rotation, fine-grained access policies, audit logs, dynamic secrets (DB credentials generated per-request). Pre-commit hooks: git-secrets, detect-secrets, truffleHog — scan for secrets before commit. Rotation: rotate secrets regularly and immediately on suspected compromise. Design apps to support zero-downtime rotation (dual-read period).

Quick Review

Why shouldn't you store sensitive data in a JWT payload?

JWT payload is base64url encoded, not encrypted. Anyone who intercepts the token (or has access to the client) can decode it and read the payload. The signature only proves the payload hasn't been tampered with — it doesn't hide it.

If you need to include sensitive claims, use JWE (JSON Web Encryption) instead of JWS (JSON Web Signature). Or keep sensitive data server-side in the session store and put only a session ID in the cookie.

What is an IDOR vulnerability and how do you prevent it?

Insecure Direct Object Reference (IDOR): an API exposes an internal identifier (DB ID) and doesn't verify that the requesting user owns that resource.

Example: GET /api/invoices/1234 — if any authenticated user can fetch any invoice just by changing the ID, that's IDOR.

Prevention: always check ownership/authorization server-side: SELECT * FROM invoices WHERE id = $id AND user_id = $current_user. Consider using UUIDs instead of sequential IDs (harder to enumerate, but not a security measure on its own). Add authorization middleware that verifies resource ownership for every request.

What's the difference between hashing a password with MD5/SHA1 vs bcrypt?

MD5/SHA1 are cryptographic hash functions designed to be fast — an attacker can compute billions of hashes per second on GPU hardware, making brute force feasible.

bcrypt/argon2/scrypt are key derivation functions designed to be deliberately slow and memory-hard. bcrypt with cost factor 12 takes ~250ms per hash on modern hardware — 4M× slower than MD5. For a typical user login this is imperceptible; for an attacker brute-forcing a stolen hash database, it's the difference between cracking in hours vs centuries.

Also: always add a unique random salt per password (bcrypt does this automatically) to prevent rainbow table attacks and ensure identical passwords hash differently.
