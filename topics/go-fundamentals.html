🔤 Go Fundamentals

The building blocks of every Go program. Master these and everything else clicks into place.

📐

Types & Variables

Go is statically typed: every variable has a type known at compile time. There are no implicit conversions — if you want an int64 where you have an int32, you must convert explicitly. This strictness is one of Go's greatest assets: the compiler catches entire categories of bugs before your code runs.

Declaration Styles

Go offers two declaration styles. The var keyword is explicit and works at package scope or inside functions. The short declaration := is implicit and only works inside functions:

// var — explicit type, works anywhere
var name string = "Alice"
var count int    // zero value: 0

// := — infers type from right side, functions only
name := "Alice"
count := 0

// Multiple assignment
x, y := 10, 20
a, b := b, a  // swap in one line — no temp variable needed

Use var at package scope and when you want to declare without initializing. Use := inside functions for brevity. The compiler will error if you declare a variable with := and never use it — this prevents dead code accumulation.

Zero Values

Every type in Go has a zero value — what you get when you declare a variable without initializing it. There is no undefined or uninitialized memory:

var i int        // 0
var f float64    // 0.0
var b bool       // false
var s string     // ""  (empty string, not nil)
var p *int       // nil
var sl []int     // nil (nil slice — length 0, works fine with append)
var m map[string]int  // nil (nil map — reads return zero, writes panic!)
A nil map returns zero values on read but panics on write. Always initialize maps before writing: m := make(map[string]int) or m := map[string]int{}.

Basic Types

Go's built-in types:

  • Integers: int, int8, int16, int32, int64, and unsigned variants (uint, uint8/byte, uint16, uint32, uint64). Use int unless you have a specific reason not to — it's the size of the platform's native word (64-bit on 64-bit systems).
  • Floats: float32, float64. Prefer float64 — more precision, less surprising.
  • String: immutable sequence of bytes (UTF-8 by convention). Indexing a string gives a byte, not a rune. Use []rune(s) to work with Unicode code points.
  • Rune: alias for int32, represents a Unicode code point.
  • bool: true or false. No truthy/falsy coercion.

Multiple Return Values

Functions can return multiple values — the idiomatic way to return both a result and an error:

func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("division by zero")
    }
    return a / b, nil
}

result, err := divide(10, 3)
if err != nil {
    log.Fatal(err)
}
fmt.Println(result) // 3.3333...

// Discard a return value with _
result, _ = divide(10, 3)

Named return values name the return variables in the function signature. They're initialized to zero values and can be returned with a bare return (a "naked return"). Use sparingly — naked returns in long functions hurt readability. Named returns are mainly useful for documentation or when defer needs to modify the returned value:

func minMax(arr []int) (min, max int) {
    min, max = arr[0], arr[0]
    for _, v := range arr[1:] {
        if v < min { min = v }
        if v > max { max = v }
    }
    return // naked return — returns min and max
}

Constants & iota

const Pi = 3.14159

// iota — increments by 1 for each const in a block
type Direction int
const (
    North Direction = iota // 0
    East                   // 1
    South                  // 2
    West                   // 3
)

// iota with expressions
type ByteSize float64
const (
    _           = iota // ignore first
    KB ByteSize = 1 << (10 * iota) // 1024
    MB                              // 1048576
    GB                              // 1073741824
)

Functions, Closures & Defer

Functions are first-class citizens in Go: they can be assigned to variables, passed as arguments, and returned from other functions. This makes higher-order functions, callbacks, and middleware patterns idiomatic.

Functions as Values

// Assign a function to a variable
add := func(a, b int) int { return a + b }
result := add(3, 4) // 7

// Pass a function as an argument
func apply(nums []int, fn func(int) int) []int {
    out := make([]int, len(nums))
    for i, n := range nums {
        out[i] = fn(n)
    }
    return out
}

doubled := apply([]int{1, 2, 3}, func(n int) int { return n * 2 })

// Return a function from a function
func multiplier(factor int) func(int) int {
    return func(n int) int {
        return n * factor
    }
}
triple := multiplier(3)
fmt.Println(triple(5)) // 15

Closures

A closure is a function that captures variables from its enclosing scope. The captured variables are shared between the closure and the enclosing scope — mutations are visible to both sides:

func counter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}

c := counter()
fmt.Println(c()) // 1
fmt.Println(c()) // 2
fmt.Println(c()) // 3

// Each call to counter() creates an independent counter
d := counter()
fmt.Println(d()) // 1 — fresh count, not shared with c
Classic loop closure bug: capturing a loop variable by reference, not by value. 
// WRONG — all closures share the same i
for i := 0; i < 3; i++ {
    go func() { fmt.Println(i) }() // prints 3 3 3
}

// CORRECT — pass i as an argument
for i := 0; i < 3; i++ {
    go func(i int) { fmt.Println(i) }(i) // prints 0 1 2 (any order)

Variadic Functions

func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

sum(1, 2, 3)        // direct call
nums := []int{1, 2, 3}
sum(nums...)        // spread a slice with ...

defer

defer pushes a function call onto a stack that executes after the surrounding function returns, in LIFO order. It runs even if the function panics. The canonical use is cleanup — closing files, releasing locks, closing database connections — right next to the code that opens them:

func readFile(path string) (string, error) {
    f, err := os.Open(path)
    if err != nil {
        return "", err
    }
    defer f.Close() // guaranteed to run when readFile returns

    data, err := io.ReadAll(f)
    return string(data), err
}

// defer arguments are evaluated immediately, not when deferred fn runs
x := 10
defer fmt.Println(x) // prints 10, not whatever x is later
x = 20

Multiple defers run in reverse order (LIFO). This is intentional: you acquire resources in order A → B → C and release them as C → B → A, like unwinding a call stack.

panic & recover

panic stops normal execution and begins unwinding the stack, running deferred functions along the way. recover stops the unwind and returns the value passed to panic. Use this sparingly — for truly unexpected invariant violations, not for normal error handling:

func safeDiv(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered panic: %v", r)
        }
    }()
    return a / b, nil // panics if b == 0
}

// The right use of panic: programmer error, never valid inputs
func mustPositive(n int) int {
    if n <= 0 {
        panic(fmt.Sprintf("mustPositive: got %d", n))
    }
    return n
}

Rule of thumb: panic for programmer errors (precondition violations, impossible states), errors for expected failures (bad input, network timeout, file not found). Most library code should never panic on bad input from callers — return an error instead.

Structs & Interfaces

Go doesn't have classes or inheritance. Instead it has structs (data) and interfaces (behavior). Composition via embedding replaces inheritance. This leads to simpler, flatter code with less coupling.

Structs

type User struct {
    ID    int
    Name  string
    Email string
    Admin bool
}

// Initialization — field names make it readable and order-independent
u := User{
    ID:    1,
    Name:  "Alice",
    Email: "alice@example.com",
}

// Accessing fields
fmt.Println(u.Name)

// Pointer to struct — Go auto-dereferences
p := &User{Name: "Bob"}
fmt.Println(p.Name)  // same as (*p).Name

Methods

Methods are functions with a receiver — the type they operate on. The receiver is like self in Python but is explicit and can be either a value or a pointer:

// Value receiver — gets a copy; doesn't modify original
func (u User) Greet() string {
    return "Hello, " + u.Name
}

// Pointer receiver — operates on original; can mutate
func (u *User) Promote() {
    u.Admin = true
}

u := User{Name: "Alice"}
u.Promote()       // Go automatically takes &u if needed
fmt.Println(u.Greet()) // "Hello, Alice"

Value vs pointer receiver rules:

  • If the method needs to mutate the receiver, use a pointer receiver.
  • If the struct is large, use a pointer receiver to avoid copying it on every call.
  • If the struct contains a sync.Mutex or similar, it must use a pointer receiver — copying a mutex after first use is a data race.
  • For consistency: if any method on a type uses a pointer receiver, all methods should use pointer receivers.

Embedding (Composition over Inheritance)

Embedding a type inside a struct promotes all the embedded type's methods to the outer struct. This is how Go does code reuse without inheritance:

type Animal struct {
    Name string
}

func (a Animal) Speak() string {
    return a.Name + " makes a sound"
}

type Dog struct {
    Animal       // embedded — no field name
    Breed string
}

d := Dog{
    Animal: Animal{Name: "Rex"},
    Breed:  "Labrador",
}

d.Speak()      // promoted — same as d.Animal.Speak()
d.Animal.Speak() // also works — explicit
d.Name          // promoted field — same as d.Animal.Name

Embedding is not inheritance. Dog does not "is-a" Animal in the type system — you can't assign a Dog to an Animal variable. What embedding gives you is method promotion and field promotion, nothing more. This is deliberate: the "is-a" relation creates tight coupling that embedding deliberately avoids.

Interfaces

An interface is a set of method signatures. Any type that implements all those methods satisfies the interface — implicitly, with no implements declaration. This is structural typing (duck typing with compile-time checking):

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Stringer interface {
    String() string
}

// Any type with a Write method automatically satisfies Writer.
// The type never needs to say so.

// Compose interfaces from smaller ones
type ReadWriter interface {
    io.Reader
    io.Writer
}

The power of implicit satisfaction: you can define an interface in your package that matches a type in a third-party package, and that third-party type satisfies your interface without any modification. This allows you to build abstractions against code you don't control.

// Your package defines just what it needs
type Storage interface {
    Get(key string) ([]byte, error)
    Set(key string, val []byte) error
}

// In tests, use a fake
type fakeStorage struct{ data map[string][]byte }
func (f *fakeStorage) Get(key string) ([]byte, error) { ... }
func (f *fakeStorage) Set(key string, val []byte) error { ... }

// In production, use Redis — which satisfies Storage without knowing about it

Interface Values & nil

An interface value has two components: a concrete type and a concrete value. A nil interface value has both as nil. But a non-nil interface holding a nil pointer is not a nil interface:

var w io.Writer        // nil interface — both type and value are nil
w == nil               // true

var buf *bytes.Buffer  // nil *bytes.Buffer
w = buf                // non-nil interface! type = *bytes.Buffer, value = nil
w == nil               // FALSE — this is the notorious Go nil interface trap
Never return a concrete nil pointer as an interface — return a nil interface directly. The function below has the bug: if logging is false, it returns a non-nil interface holding a nil *LogWriter. 
// BUGGY
func newWriter(logging bool) io.Writer {
    var lw *LogWriter
    if logging {
        lw = &LogWriter{}
    }
    return lw // returns non-nil interface if logging=false!
}

// CORRECT
func newWriter(logging bool) io.Writer {
    if logging {
        return &LogWriter{}
    }
    return nil
}

Type Assertions & Type Switches

// Type assertion — extract the concrete type from an interface
var i interface{} = "hello"
s, ok := i.(string) // ok = true, s = "hello"
n, ok := i.(int)    // ok = false, n = 0 (zero value, no panic)
s = i.(string)      // without ok: panics if wrong type

// Type switch — pattern-match on concrete type
func describe(i interface{}) string {
    switch v := i.(type) {
    case int:
        return fmt.Sprintf("int: %d", v)
    case string:
        return fmt.Sprintf("string: %q", v)
    case bool:
        return fmt.Sprintf("bool: %t", v)
    default:
        return fmt.Sprintf("unknown: %T", v)
    }
}

Error Handling

In Go, errors are values — the built-in error interface with a single method Error() string. There is no exception mechanism for recoverable failures. You check errors explicitly, inline, at every call site. This is not boilerplate — it forces you to think about every failure mode and decide what to do with it.

The error Interface

// The built-in error interface
type error interface {
    Error() string
}

// errors.New creates a simple error value
err := errors.New("something went wrong")

// fmt.Errorf creates a formatted error
err = fmt.Errorf("user %d not found", userID)

// Checking for nil
if err != nil {
    // handle it
}

Custom Error Types

Implement the error interface on any type to create rich, inspectable errors:

type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation error on %s: %s", e.Field, e.Message)
}

// Callers can type-assert to get the structured data
err := validate(input)
var ve *ValidationError
if errors.As(err, &ve) {
    fmt.Printf("invalid field: %s\n", ve.Field)
}

Wrapping Errors: %w

When a lower-level error propagates up the call stack, wrap it with context using fmt.Errorf("...: %w", err). The %w verb records the wrapped error in a chain that callers can inspect:

func getUser(id int) (*User, error) {
    row, err := db.Query("SELECT * FROM users WHERE id = ?", id)
    if err != nil {
        return nil, fmt.Errorf("getUser(%d): %w", id, err)
    }
    // ...
}

// The error message chains nicely:
// "getUser(42): sql: no rows in result set"

// Callers can unwrap to check for specific errors:
if errors.Is(err, sql.ErrNoRows) {
    // handle not-found
}

errors.Is and errors.As

Never compare errors with == when they might be wrapped. Use these instead:

  • errors.Is(err, target) — walks the error chain checking if any error in the chain matches target. Use for sentinel errors.
  • errors.As(err, &target) — walks the chain checking if any error can be assigned to target. Use for typed errors where you need the fields.
// Sentinel errors — exported variables that callers check against
var ErrNotFound = errors.New("not found")
var ErrUnauthorized = errors.New("unauthorized")

// errors.Is unwraps automatically
wrappedErr := fmt.Errorf("service failed: %w", ErrNotFound)
errors.Is(wrappedErr, ErrNotFound) // true

// errors.As to extract typed error from chain
var dbErr *DBError
if errors.As(err, &dbErr) {
    fmt.Println(dbErr.Code) // access structured fields
}

Error Handling Patterns

Add context, don't discard it. Errors bubble up through function calls — each layer should add what it knows:

// Good — each layer adds context
func processOrder(orderID string) error {
    order, err := fetchOrder(orderID)
    if err != nil {
        return fmt.Errorf("processOrder %s: %w", orderID, err)
    }
    if err := chargeCard(order); err != nil {
        return fmt.Errorf("processOrder %s: charge: %w", orderID, err)
    }
    return nil
}
// Error: "processOrder ord_123: charge: card declined"
Don't log and return. If you log an error and also return it, the error will be logged again by the caller, and again by its caller. Pick one: either handle the error (log it, return a response) or propagate it upward. Doing both produces duplicate log noise that makes incidents harder to debug.

Panic is not an error return. Libraries should return errors, not panic on bad inputs. Reserve panic for programmer errors: violated preconditions, impossible states, corrupt internal invariants. The must pattern is common for initialization-time errors where there's no point continuing:

// mustCompile panics if regex is invalid — only for compile-time constants
var emailRe = regexp.MustCompile(`^[a-z]+@[a-z]+\.[a-z]+$`)

Pointers & Value Semantics

A pointer holds the memory address of a value rather than the value itself. Go has pointers but no pointer arithmetic — you can't do ptr++ to walk memory. This keeps pointers safe while still giving you control over when things are copied vs shared.

Pointer Basics

x := 42
p := &x       // p is of type *int, holds address of x
fmt.Println(*p) // dereference: prints 42
*p = 100       // modify through pointer
fmt.Println(x) // 100 — x changed!

// new() allocates zeroed memory and returns a pointer
p2 := new(int) // *int pointing to 0
*p2 = 7

When to Use Pointers

The question is: do you want the callee to see the original, or its own copy?

  • Use a pointer when the function needs to modify the value, when the struct is large (copying is expensive), when you want to distinguish between "zero value" and "not provided" (use *string vs string), or when the type contains a mutex.
  • Use a value for small types (int, float64, small structs), when the function only needs to read, or for immutable data you want to protect from mutation.
// Pointer: function modifies the struct
func (u *User) SetEmail(email string) {
    u.Email = email
}

// Value: function only reads, small struct
func (p Point) Distance() float64 {
    return math.Sqrt(p.X*p.X + p.Y*p.Y)
}

Slices, Maps, and Channels Are Reference Types

Slices, maps, and channels are already reference types — they contain internal pointers to their underlying data. Passing them to functions does not copy the data:

func doubleAll(nums []int) {
    for i := range nums {
        nums[i] *= 2 // modifies the original slice's backing array
    }
}

s := []int{1, 2, 3}
doubleAll(s)
fmt.Println(s) // [2 4 6] — original was modified
Slice header vs backing array. A slice is a three-word struct: pointer to backing array, length, capacity. When you pass a slice to a function, you pass a copy of the header — the pointer inside still points to the same backing array. Mutations to elements affect the original. But append may allocate a new array if capacity is exceeded — the caller's slice header won't see the new elements unless you return the new slice. 
func appendItem(s []int, item int) []int {
    return append(s, item) // must return the new slice
}

nil Pointers

Dereferencing a nil pointer panics. Always check before dereferencing a pointer you didn't initialize yourself:

var p *User
if p != nil {
    fmt.Println(p.Name) // safe
}
fmt.Println(p.Name) // panic: nil pointer dereference

A common pattern is to use a pointer to a struct as an optional value — nil means "not present," non-nil means "present." This is more explicit than a boolean flag alongside a value.

Packages & Modules

Go's package system is the unit of code organization and encapsulation. Modules (introduced in Go 1.11) manage dependencies. Together they define how Go projects are structured, shared, and versioned.

Packages

Every Go file belongs to a package, declared at the top: package foo. All files in the same directory must use the same package name. Import paths are directory paths relative to the module root:

// File: myapp/users/repository.go
package users

// File: myapp/main.go
package main

import "myapp/users"

Exported vs unexported identifiers: Identifiers starting with an uppercase letter are exported (public). All others are unexported (package-private). There's no public/private keyword — the naming convention is enforced by the compiler:

package users

type User struct {        // exported — visible outside package
    ID   int              // exported field
    name string           // unexported field — only accessible in this package
}

func NewUser(name string) *User { ... } // exported constructor
func validate(u *User) error { ... }    // unexported helper

Good package design: export the minimal surface area needed. If you're unsure whether something should be exported, don't export it — it's easier to export later than to unexport something that callers have started depending on.

Modules & go.mod

A module is a collection of packages with a common import path prefix, versioned together. The go.mod file is the module definition:

module github.com/acme/myapp

go 1.22

require (
    github.com/gin-gonic/gin v1.9.1
    golang.org/x/crypto v0.21.0
)

// replace — redirect to a local path (for development)
replace github.com/acme/shared => ../shared

Key commands:

  • go mod init github.com/you/project — initialize a new module
  • go get github.com/pkg@v1.2.3 — add/update a dependency
  • go mod tidy — remove unused deps, add missing ones
  • go mod vendor — copy deps into a vendor/ directory for reproducible builds

init() Functions

Each package can have one or more init() functions — they run automatically when the package is first imported, before main():

func init() {
    // runs once when this package is imported
    // common uses: register drivers, validate config, set up global state
    sql.Register("postgres", &pgDriver{})
}

Use init() sparingly. Hidden initialization makes code harder to test and reason about. Prefer explicit initialization in main() or constructor functions. The main valid use case is registering things in a global registry (database drivers, codec implementations, etc.).

The Blank Import

Importing a package solely for its init() side effect uses the blank identifier:

import _ "github.com/lib/pq" // registers the postgres driver via init()

Essential Go Tooling

  • go build ./... — compile all packages in the module
  • go test ./... — run all tests
  • go vet ./... — catch common mistakes (misused Printf verbs, unreachable code, etc.)
  • gofmt -w . — format code (non-negotiable; all Go code is formatted the same way)
  • golangci-lint run — run a suite of linters (the de facto standard in production codebases)
  • go doc net/http — read package documentation from the terminal
