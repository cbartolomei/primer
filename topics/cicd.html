🔁 CI/CD

From code commit to production — pipelines, deployment strategies, tooling, and the practices that let teams ship dozens of times per day.

🏗️

CI Fundamentals

Continuous Integration is the practice of integrating code into a shared branch frequently — at least daily — and verifying each integration with an automated build and test run. The motivation is economic: bugs found at commit time cost minutes to fix; bugs found in production cost hours of debugging, a potential incident, and eroded user trust. Merge conflicts follow the same curve — a branch that diverges for two weeks accumulates two weeks of conflict debt, paid all at once when the PR opens.

Pipeline Anatomy

A CI pipeline is a sequence of stages, each gating the next. A typical pipeline for a web service:

  1. Trigger: a push, pull request, or scheduled event fires the pipeline.
  2. Checkout: the runner clones the repository at the triggering commit SHA.
  3. Lint: static analysis catches style violations, unused imports, and obvious bugs before any test runs.
  4. Unit tests: fast, isolated tests that don't touch external systems. Measured in milliseconds per test.
  5. Integration tests: tests that spin up real dependencies (a database, a cache). Seconds per test.
  6. Build artifact: compile a binary or build a Docker image from the commit that passed tests.
  7. Push to registry: push the image to a container registry tagged with the commit SHA. This artifact is the deployable unit.

Each stage gates the next. If lint fails, unit tests don't run. If unit tests fail, integration tests don't run. You never build an artifact from broken code.

The Test Pyramid

The test pyramid is a heuristic for how many tests to write at each layer. At the base: many unit tests — hundreds or thousands, each running in milliseconds, testing a single function or class in isolation with mocked dependencies. In the middle: fewer integration tests — dozens to hundreds, spinning up real dependencies, running in seconds. At the top: very few end-to-end tests — the full browser or API flow, taking minutes each.

Invert the pyramid (few unit tests, many E2E tests) and CI becomes unusably slow. A CI run that takes 45 minutes doesn't give fast feedback — developers stop waiting for it and push again, causing a queue. The goal is a CI run that completes in under 10 minutes.

Fail Fast

Run the cheapest checks first. A lint failure that catches an undefined variable takes 30 seconds. There is no reason to start a 5-minute integration test suite until lint passes. Order your pipeline stages by ascending cost: lint → unit tests → integration tests → E2E tests. If any stage fails, later stages are cancelled. The developer gets the fastest possible signal about the cheapest possible failure.

Test Parallelization

Split test suites across multiple runners to reduce wall-clock time. GitHub Actions matrix strategy runs the same job with different parameters simultaneously. pytest-xdist distributes Python tests across workers. Jest's --maxWorkers flag parallelizes across CPU cores. The goal is that adding more tests doesn't proportionally increase CI time — horizontal scaling keeps the pipeline fast as the codebase grows.

Dependency Caching

Restoring dependencies from cache is dramatically faster than downloading them from the internet on every run. A cache hit on npm install for a large project saves 90 seconds. The cache key is a hash of the lockfile — if package-lock.json doesn't change, the cache is valid. If it does change (a dependency was added or updated), the cache is busted and dependencies are re-fetched.

- uses: actions/cache@v3
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
    restore-keys: ${{ runner.os }}-npm-

The restore-keys fallback finds the most recent cache for the OS even when the exact key misses — useful when a dependency is added, because you restore the old cache and only download the delta rather than everything from scratch. The same pattern applies to Go modules (~/go/pkg/mod), pip virtualenvs, and Docker layer caches.

Artifact Management

Build once, promote the same artifact through environments. This is a critical principle: never rebuild from source for staging or production. You want to verify that the exact binary that passed CI is the one running in production. Rebuilding introduces the possibility that a non-deterministic build, a transient network failure, or a changed upstream dependency produces a different artifact — one that was never tested.

After a successful CI run, push the Docker image to a registry (Docker Hub, Amazon ECR, GitHub Container Registry, Artifactory) tagged with the commit SHA. Staging and production deployments pull that exact tag. The image is the unit of promotion — not the git commit, not the source code.

Continuous Delivery vs Continuous Deployment

Continuous Delivery means the pipeline automatically produces a deployable artifact and deploys it to a pre-production environment (staging, QA), but a human decision gates production. The artifact is always ready; the release is a deliberate act. Continuous Deployment means every commit that passes CI is automatically deployed to production — no human gate. This requires high test coverage, feature flags for incomplete work, and robust rollback capability.

Most teams land in between: auto-deploy to staging on every merge to main, then either manual approval for production or automatic promotion after a staging soak period with no error-rate increase. The spectrum matters — the closer you get to continuous deployment, the more you invest in test infrastructure, observability, and progressive delivery tooling.

Deployment Strategies

The goal of a deployment strategy is to ship a new version with zero user-visible downtime and the smallest possible blast radius if something goes wrong. The strategies differ in how they manage the transition between old and new, and how quickly you can recover from a bad deploy.

Rolling Update

Replace instances one by one, or in small batches. This is the default in Kubernetes Deployments. Kubernetes spins up new pods with the new image before terminating old ones, controlled by maxSurge (how many extra pods can exist during rollout) and maxUnavailable (how many pods can be down). For a 10-replica deployment with maxSurge: 2, maxUnavailable: 1, you'll have between 9 and 12 pods during the rollout.

The key constraint: during a rolling update, both the old and new version of your application are serving traffic simultaneously. Your database schema must be backward compatible with both versions. A column renamed in the migration will break the old pods that are still running. This is why schema migrations require the expand/contract pattern (covered below).

Blue/Green

Maintain two identical production environments. Blue is the current live environment. Green is the new version, deployed and tested in isolation. When green is ready, flip the load balancer — 100% of traffic instantly moves to green. Rollback is equally instant: flip back to blue. No gradual transition, no compatibility window.

The cost is 2× production capacity during the switchover. For stateless services, this is straightforward. For stateful systems, blue/green is more complex — your database is shared between environments, so schema changes still need to be compatible with both application versions. Blue/green is best for large releases where you want a clean, instant escape hatch and can afford the temporary capacity cost.

Canary Deployments

Route a small fraction of traffic — typically 1–5% — to the new version, while the majority stays on the old version. Monitor error rates, latency p99, and business metrics on the canary. If the canary looks healthy after a soak period, gradually increase traffic: 1% → 5% → 25% → 100%. If error rates spike, route 0% to the canary and investigate.

The blast radius calculation: if 1% of users hit a new bug that causes 500 errors, your overall error rate rises from 0% to 0.01%. That's easy to catch in a dashboard before it affects the other 99%. Compare this to a rolling deploy where the bug reaches 100% of users before you notice the error rate climb. Tools: Argo Rollouts, Istio virtual services, AWS Application Load Balancer weighted target groups, Flagger for automated promotion based on Prometheus metrics.

Feature Flags

Deploy code to production in a disabled state and enable it per user, per segment, or by percentage rollout. Feature flags decouple deployment from release — the code ships with the rest of the codebase, but the feature is off until you decide to turn it on. This enables dark launches: you can instrument a new code path and measure its performance impact (memory, latency) with real production traffic while zero users see any UI change.

The key insight: with feature flags, every deploy is effectively a no-op from the user's perspective. The risk is in the flag flip, which is an instantaneous, reversible action — not a deployment. If a newly enabled feature causes an incident, you flip the flag off. No rollback, no revert commit, no deployment. The incident window is seconds. Tools: LaunchDarkly (SaaS, robust targeting rules), Unleash (open source, self-hostable), Flagsmith (open source with a cloud tier).

Smoke Tests and Automated Rollback

After every deployment, run a fast smoke test suite covering only the critical path: can a user log in? Can they perform the core action? Does checkout complete? Smoke tests should finish in under 2 minutes. If they fail, trigger automatic rollback before the bad version affects more users.

Argo Rollouts supports metric-based automated rollback: define a success condition (e.g., error rate below 1%, p99 latency below 500ms) and a rollback condition (error rate above 5%). During a canary promotion, Argo continuously queries Prometheus. If the rollback condition is met, Argo automatically aborts the rollout and routes all traffic back to the stable version. Spinnaker supports similar automated rollback via deployment pipelines and Kayenta for canary analysis.

Database Migrations: Expand/Contract

Database migrations are the trickiest part of zero-downtime deploys. A naive approach — rename a column, deploy new code — breaks the old pods still running during a rolling deploy. The expand/contract pattern solves this by separating schema changes into multiple phases, each backward compatible with the running application version:

  1. Expand: add the new column as nullable (so old code that doesn't write to it doesn't fail). Deploy new application code that writes to both the old and new column. At this point, both old and new app code can run against the same schema.
  2. Migrate: run a background job to backfill the new column for all existing rows. This can take hours for large tables — run it offline or in small batches to avoid locking.
  3. Contract: once all rows are backfilled and old application code is fully replaced, add the NOT NULL constraint, remove the old column, and drop any compatibility dual-write logic from the application.

Never rename a column in a single deployment. Never add a NOT NULL column without a default in a single step. Each schema change should be deployable independently from the application change that uses it. Migrations run first; application changes follow.

GitHub Actions

GitHub Actions is GitHub's native CI/CD system. Workflows are YAML files stored in .github/workflows/ in your repository, triggered by GitHub events. Because it lives inside GitHub, it has first-class access to your repositories, pull requests, issues, packages, and secrets — no webhook configuration or token management needed to connect to code review.

Core Concepts

  • Workflow: a YAML file defining the automation. One or more workflows per repository.
  • Event: what triggers the workflow — a push, a pull request, a release, a schedule, or a manual dispatch.
  • Job: a set of steps that run sequentially on a single runner. Jobs within a workflow run in parallel by default; use needs to create dependencies between them.
  • Step: an individual shell command (run:) or a reusable action (uses:). Steps within a job share the runner's filesystem and environment.
  • Action: a reusable unit of automation — can be from the GitHub Marketplace, a public repo, or a local action in your own repository. Actions abstract common tasks: checking out code, setting up a language runtime, building Docker images, deploying to a cloud provider.
  • Runner: the virtual machine that executes a job. GitHub provides hosted runners (Ubuntu, Windows, macOS). You can also run self-hosted runners on your own infrastructure.

Triggers

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * *'   # nightly at 2am
  workflow_dispatch:        # manual trigger with optional inputs

The pull_request trigger runs on PRs targeting the specified branch — the most common pattern for CI validation before merge. workflow_dispatch adds a "Run workflow" button in the GitHub UI, useful for release pipelines or ad-hoc jobs. push with paths: filtering runs the workflow only when specific files change — useful for a monorepo where you don't want to run every service's tests on every commit.

A Realistic CI Workflow

name: CI
on:
  push:
    branches: [main]
  pull_request:
jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: postgres
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.23'
      - uses: actions/cache@v3
        with:
          path: ~/go/pkg/mod
          key: ${{ runner.os }}-go-${{ hashFiles('**/go.sum') }}
      - run: go test ./...
        env:
          DATABASE_URL: postgres://postgres:postgres@localhost/testdb
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}

The services block spins up a Postgres container as a sidecar for the test job — accessible on localhost from steps. The health-cmd option makes the step wait until Postgres is accepting connections before running tests. The build job uses needs: test to ensure it only runs after tests pass. The image is tagged with github.sha — a content-addressable, immutable tag that maps directly to the commit that was tested.

Matrix Builds

strategy:
  matrix:
    go-version: ['1.21', '1.22', '1.23']
    os: [ubuntu-latest, macos-latest]
runs-on: ${{ matrix.os }}

The matrix strategy expands into multiple parallel jobs — one per combination of values. This example runs 6 jobs: 3 Go versions × 2 operating systems. All run simultaneously. Use matrix builds to verify compatibility across language versions before a major dependency upgrade, or to catch OS-specific bugs in libraries that use CGo or platform syscalls.

Reusable Workflows

A workflow triggered by workflow_call can be called from other workflows, even across repositories in the same organization. This is the right abstraction for shared CI/CD logic — define your organization's standard lint/test/deploy workflow once, and each service's workflow calls it rather than duplicating the YAML. Changes to the shared workflow propagate to all callers. Inputs and secrets can be passed from the caller to the called workflow.

OIDC for Cloud Authentication

Instead of storing long-lived cloud credentials as GitHub secrets (which need rotation, can be leaked in logs, and represent persistent blast radius), use OpenID Connect (OIDC). GitHub Actions can prove to AWS, GCP, or Azure that a workflow is running from a specific repository, branch, and environment. The cloud provider issues a short-lived token (typically valid for 15 minutes) that grants only the necessary permissions. There are no static credentials to rotate or accidentally commit. AWS supports this via IAM OIDC identity providers; GCP via Workload Identity Federation.

Self-Hosted Runners

GitHub-hosted runners are isolated, ephemeral VMs with no access to your private network. When your jobs need to access internal resources — a private database, an on-prem service, an internal artifact registry — you need self-hosted runners. Other reasons: specialized hardware (GPU instances for ML, ARM for Apple Silicon compatibility testing), cost at scale (GitHub-hosted runners are billed per minute and become expensive for organizations with hundreds of engineers), or regulatory requirements that prohibit running builds on shared infrastructure.

Self-hosted runners register with GitHub and poll for queued jobs. They run on any infrastructure: EC2, GKE, bare metal. Actions Runner Controller (ARC) manages self-hosted runners as Kubernetes pods, scaling them automatically based on job queue depth.

Environments and Protection Rules

GitHub Environments let you define named deployment targets (staging, production) with associated rules. Protection rules can require approval from specific reviewers before a job targeting that environment runs — this is the human gate for production deployments. Wait timers add a minimum soak period. Secrets scoped to an environment are only visible to jobs that target it, so your production database credentials are never available to a job running on a PR branch.

Jenkins & Alternatives

CI/CD tooling is a crowded space. The right choice depends on where your code lives, your compliance requirements, your team's ops capacity, and your scale. Here's how the major options compare.

Jenkins

Jenkins is the grandfather of CI/CD — open source, self-hosted, and infinitely extensible via roughly 1,800 plugins. Jenkinsfile defines pipelines as code in a Groovy DSL with two styles: declarative (structured, opinionated, recommended for new pipelines) and scripted (full Groovy, maximum flexibility for complex logic). The file lives in the repository alongside application code, giving you version control, peer review, and history for pipeline changes.

The value proposition: Jenkins runs on your infrastructure and can reach anything on your network — private databases, on-prem services, internal APIs, air-gapped environments. The ecosystem is mature and the internet is full of examples for any integration you need. Shared Libraries let you define reusable pipeline steps in a separate repository and import them across all your Jenkinsfiles — the equivalent of GitHub Actions' reusable workflows.

The cost: you own everything. Jenkins itself is infrastructure. You manage the server, apply plugin updates (carefully — plugins break each other), handle security patches, configure agents, and scale capacity. Jenkins becomes a service your platform team runs for the rest of engineering. When to choose Jenkins: enterprises with complex compliance requirements that mandate on-prem build infrastructure, organizations already deeply invested in the ecosystem, or environments where no external network access is permitted.

GitLab CI

GitLab CI is deeply integrated with GitLab's code review, security scanning, container registry, and deployment tracking. The pipeline is defined in .gitlab-ci.yml at the repository root. Pipelines are organized into stages and jobs — all jobs in a stage run in parallel; stages run sequentially. GitLab Runners are the execution agents and can be shared across projects or dedicated to a specific project.

If your organization is on GitLab, the integration is seamless — merge request pipelines show inline CI status, the built-in container registry means no external registry configuration, and deployment tracking links pipeline runs to environments. Auto DevOps can detect your application's tech stack and automatically configure a sensible CI/CD pipeline without any YAML — practical for teams without dedicated platform engineering. GitLab's security scanning (SAST, DAST, dependency scanning) integrates directly into merge request pipelines.

CircleCI

CircleCI is a hosted CI service with strong parallelism and Docker support. Orbs are reusable configuration packages in the CircleCI ecosystem — analogous to GitHub Actions actions. The standout features: automatic test splitting distributes your test suite across parallel runners based on timing history (not manual sharding), and Docker Layer Caching (DLC) caches Docker build layers between runs, dramatically reducing image build time. Resource classes let you specify CPU and RAM per job — useful when some jobs need 8 CPUs for compilation and others need only 1.

CircleCI works well for teams that want more control and performance than GitHub Actions provides but don't want the operational burden of running Jenkins. The hosted model means CircleCI manages the infrastructure; you manage the pipeline configuration.

Buildkite

Buildkite uses a hybrid model: SaaS control plane plus your own agents. Buildkite orchestrates the pipeline and provides the UI and API; you run the agents on your own infrastructure (cloud VMs, Kubernetes pods, on-prem servers, wherever). Jobs run on your agents with full access to your network, but you don't manage the orchestration layer. This gives you the network access of self-hosted Jenkins with the operational simplicity of a SaaS product.

Pipelines can be defined in YAML or generated dynamically at runtime — a job can output YAML that defines subsequent jobs, enabling powerful patterns like building a pipeline that reflects the actual changed services in a monorepo. Popular at organizations with strict data locality requirements, bespoke hardware needs, or those that have outgrown GitHub Actions but don't want to run Jenkins.

Concepts That Transfer Across All Tools

Regardless of which tool you use, the same principles apply:

  • Secrets management: never hardcode credentials in pipeline YAML or application code. Inject secrets at runtime via the tool's secrets store (GitHub encrypted secrets, Jenkins credentials store, CircleCI environment variables). Secrets should never appear in build logs — use masking.
  • Artifact promotion: build once, promote the same artifact through environments. The Docker image tagged :abc1234 that passed CI is the same image deployed to staging, then promoted to production.
  • Environment gates: staging deployments are automatic; production deployments require approval, a soak period, or both. The gate is the only difference between environments — not different build processes.
  • Pipeline as code: Jenkinsfile, .github/workflows/, .gitlab-ci.yml, .buildkite/pipeline.yml — pipeline definitions live in the repository, version-controlled alongside application code. Pipeline changes go through code review. You can git blame a pipeline regression.

GitOps

GitOps is a deployment model where a git repository is the single source of truth for the desired state of a system — typically a Kubernetes cluster. You never apply changes manually. You open a pull request, get it reviewed, merge it, and an agent running inside the cluster reconciles the live state with the git state. The git history is the complete, auditable record of every change to production.

Pull vs Push Model

Traditional CI/CD pipelines use a push model: after a successful build, the CI system runs kubectl apply or helm upgrade to update the cluster. This requires the CI system to hold cluster credentials — typically a kubeconfig or service account token with significant permissions. A compromised CI system (a malicious third-party action, a leaked secret) means cluster access.

GitOps uses a pull model: an agent runs inside the cluster with a Kubernetes ServiceAccount. The agent polls the git repository and applies changes from within the cluster. No external system holds cluster credentials. The CI system's job ends at pushing a new image tag to a registry or updating a manifest in git — it never touches the cluster directly. The attack surface is inverted: to affect the cluster, an attacker needs either cluster access (to compromise the agent) or write access to the git repository (protected by code review and branch protection), not CI system credentials.

Argo CD

Argo CD is the most widely adopted GitOps tool for Kubernetes. Its core resource is the Application: a declaration of what git repository and path contain the desired manifests, and what cluster and namespace they should be applied to.

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-api
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests
    targetRevision: HEAD
    path: apps/my-api/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true      # delete resources removed from git
      selfHeal: true   # revert manual changes

prune: true means that if you delete a manifest from git, Argo CD deletes the corresponding resource from the cluster. Without it, removed manifests leave orphaned resources. selfHeal: true means that if someone manually modifies a resource in the cluster (via kubectl edit or the Kubernetes dashboard), Argo CD will revert the change within minutes. Without it, Argo CD detects the drift but doesn't act — it alerts instead.

Health assessment: Argo CD understands the health semantics of common Kubernetes resources. A Deployment is healthy when all desired replicas are available. A Service is healthy when it exists. An Ingress is healthy when it has an address. You can define custom health checks for CRDs. The Argo CD UI shows the sync status (is the cluster matching git?) and health status (are the resources running correctly?) for every resource in every Application.

App of Apps: a pattern where one root Application manages a set of child Applications. The root Application syncs a directory of Application manifests; Argo CD creates all the child Applications. This is the standard approach for multi-service deployments — a single PR to the root can add or remove entire services from a cluster.

Drift Detection

Argo CD polls the cluster and the git repository every 3 minutes by default (configurable). If it detects a difference between the live state and the git state — an unexpected resource annotation, a manually scaled Deployment, a deleted ConfigMap — it marks the Application as OutOfSync. With selfHeal: true, it immediately reconciles back to git. With selfHeal: false, it raises an alert but takes no action, letting the team investigate before reverting. This drift detection is what makes GitOps auditable: any unauthorized or manual change is surfaced within minutes.

Image Update Automation (Flux)

A gap in naive GitOps: who updates the image tag in the manifest repo when a new image is built? One option is to have the CI pipeline open a PR to the manifest repo. Flux provides a more automated path: the image reflector controller watches a container registry and detects new tags matching a pattern (e.g., all tags on the main branch). The image automation controller then commits an image tag update directly to the manifest repository (or opens a PR for review), triggering Argo CD or Flux to sync the new image to the cluster. Full GitOps for image promotion — the registry push is the input; the cluster update is the automated output.

Multi-Cluster Management

A single Argo CD control plane can manage multiple clusters. Register each remote cluster via the Argo CD CLI (it stores a service account token for each). Define Applications targeting different destination.server values — one for staging, one for production, one for a regional cluster. The Argo CD UI gives a unified view of sync and health status across all clusters. This is the standard pattern for organizations running Kubernetes in multiple regions or with distinct staging and production clusters.

Repository Structure

Monorepo: all Kubernetes manifests for all services and environments in one repository, organized by application and environment (e.g., apps/my-api/staging/, apps/my-api/production/). Simpler to start, easy to see all changes to the cluster in one pull request, easier cross-service coordination (update an ingress config affecting multiple services in one commit). Can become hard to navigate at large scale.

Polyrepo: a separate manifest repository per service, owned by the team that owns the service. Better team autonomy, cleaner access control (a team's CI pipeline only has write access to its own manifest repo). The tradeoff: cross-service changes require coordinating PRs across multiple repositories, and there's no single place to view the full cluster state in git.

Most teams start with a monorepo and split into polyrepos as team boundaries and scale demand it. The Argo CD ApplicationSet controller automates managing many Applications from a single template — useful in a polyrepo model to avoid manually creating an Application for every new service.

Trunk-Based Development

Trunk-based development (TBD) is a source control practice where all developers integrate their changes to a single shared branch — the trunk, or main — at least once per day, ideally multiple times. Feature branches exist, but they live for one to two days at most. Every commit to main is potentially deployable. This is how Google, Meta, and most hyperscalers operate at scale.

The Problem with Long-Lived Branches

Long-lived feature branches accumulate integration debt. A branch that diverges for two weeks carries two weeks of merge conflicts with it — conflicts with other branches that were merged to main while yours was in development. Teams working this way begin to dread merging. They delay it because it's painful. Features slip because "the merge isn't done yet." The merge itself becomes a multi-day event. And crucially, none of the tests on the branch reflect how the code interacts with the rest of the codebase — that integration happens at merge time, when it's most expensive to fix.

The pattern is self-reinforcing: painful merges lead to fewer merges, which leads to longer-lived branches, which leads to more painful merges. TBD breaks the cycle by making integration continuous rather than deferred.

TBD Principles

  • Integrate to main at least daily: ideally multiple times per day. Small, frequent integrations mean small, resolvable conflicts.
  • Branches live max one to two days: if a feature isn't done in two days, it ships behind a feature flag, not on a branch.
  • Every commit to main is potentially deployable: main must always be green. A failing test on main is everyone's emergency, not just the author's.
  • The team owns main's health: a broken main blocks all other engineers. Fix it immediately or revert — don't leave it broken overnight.

Feature Flags as the Enabler

The question teams always ask: "How do I ship a half-built feature to main?" The answer is feature flags. The new checkout page redesign can live in main behind a flag for three weeks — disabled for all users. Engineers merge their work daily. The flag is off. Users see nothing. When the feature is ready, the flag is flipped. If it breaks production, the flag is flipped back. No revert commit, no hotfix branch, no deployment — just a flag change, which takes seconds and is instantly reversible.

// Using a flag service
if flags.IsEnabled(ctx, "new-checkout-v2", userID) {
    return newCheckoutHandler(w, r)
}
return legacyCheckoutHandler(w, r)

Feature flags transform deployment risk into flag-flip risk. Deployments become boring. The risk surface is the flag flip, which is reversible in seconds. This is why continuous deployment — deploying every commit to main automatically — is feasible at companies that do it well: the deployments themselves carry little risk because incomplete work is always flag-guarded.

Branch Policies Under TBD

TBD doesn't mean no branches — it means short-lived branches with a defined maximum lifetime. Common policies: feature branches named feature/username/description, required CI passing before merge, no branch older than two days (auto-labeled stale, auto-closed after three days). Some teams practice branchless development — committing directly to main with pair programming serving as the code review mechanism. This is practical for small, senior teams with very high test coverage but unusual in most organizations.

Required CI before merge is non-negotiable in TBD. If CI is untrustworthy (flaky tests, a suite that takes 45 minutes), engineers bypass the requirement or merge broken code. TBD depends on a fast, reliable CI pipeline. Invest in flaky test elimination and parallelization before adopting TBD.

Release Branches

TBD uses release branches, but differently from Gitflow. When you need to stabilize for a release — typically for mobile apps or packaged software with a defined release window — cut a release branch from main at the point you want to stabilize. Only bug fixes are cherry-picked to the release branch. New feature development continues on main unimpeded. After the release ships, the branch is deleted. There are no permanent long-lived branches in TBD.

For web services that deploy continuously, release branches are usually unnecessary — you deploy directly from main, and "stabilization" means slowing down your deployment cadence during a sensitive period, not creating a branch.

Gitflow Comparison

Gitflow defines a branching model with permanent branches: main (production), develop (integration), feature branches (off develop), release branches (off develop, merged to both main and develop), and hotfix branches (off main). It was designed for software with discrete versioned releases — packaged software, mobile apps, SDKs — where you ship a version and then maintain it.

Gitflow doesn't fit web services that deploy dozens of times per day. The develop branch is a holding area that delays integration. The permanent release branch means you're maintaining parallel histories. The model was designed when deploying software meant shipping a CD — it optimizes for the wrong thing when you're deploying to servers you control. The moment you can fix a bug by deploying to production, Gitflow's multi-branch release management becomes overhead with no corresponding benefit.

Practical Adoption Path

You can't flip a team from Gitflow to TBD overnight. A progressive adoption path:

  1. Enforce branch lifetime limits: auto-label PRs older than three days as stale. This surfaces the problem without forcing an immediate behavior change.
  2. Invest in test coverage and CI speed: TBD requires a trustworthy, fast CI pipeline. Fix flaky tests. Parallelize the slow suites. Get CI under 10 minutes. You need engineers to trust that green CI means good code.
  3. Introduce feature flags: start with one or two new features behind flags. Show the team that incomplete work can safely live in main. This is the mental model shift that makes TBD feel safe.
  4. Shorten branch lifetime progressively: go from "no limit" to "one week max" to "three days max" to "one day max." Each reduction should feel natural because CI is fast and feature flags absorb the incompleteness.
  5. Measure cycle time: track the time from first commit to production deployment. This is your health metric. TBD should reduce it from days to hours. If it doesn't, find the bottleneck — usually CI speed, code review latency, or deployment pipeline complexity.
