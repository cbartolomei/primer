🐧 Linux & the OS Layer

The runtime every backend service lives inside. Processes, file descriptors, signals, memory, and the kernel primitives that explain everything from container crashes to performance bottlenecks.

⚙️

Processes, Threads & the Kernel

The kernel boundary: user space vs kernel space. Your application code runs in user space. System calls (syscalls) are the only way to cross into kernel space — to open a file, allocate memory, send a network packet, create a process. Every read(), write(), malloc() (via brk/mmap), connect() is a syscall. The CPU switches to kernel mode, executes the kernel code, returns the result to user space. This is why syscall overhead matters for high-frequency operations.

Processes

A process is a running instance of a program. Each process has: its own virtual address space (cannot see another process's memory), one or more threads, open file descriptors, a working directory, environment variables, and a PID. The kernel tracks all of this in a task_struct (Linux's process control block). fork() creates a new process by copying the parent (copy-on-write — no actual copying until writes happen). exec replaces the current process image with a new program.

Threads

Threads share the same virtual address space as their parent process. Creating a thread (pthread_create, goroutine, Java thread) is cheaper than fork — no address space copy. The kernel schedules threads, not processes. A Go program with 1000 goroutines multiplexes them onto OS threads (M:N scheduling — the Go runtime manages this). Too many OS threads (not goroutines): each thread has an 8MB default stack, consuming memory even when idle.

Process States

  • Running: actively on a CPU.
  • Runnable: ready, waiting for a CPU slot.
  • Sleeping (S): waiting for I/O — interruptible, wakes on signal.
  • Uninterruptible sleep (D): waiting for I/O that cannot be interrupted, often disk or NFS. Shows as D in ps. A stuck D-state process usually indicates an I/O or kernel bug.
  • Zombie (Z): exited but parent hasn't called wait() — holding a PID slot.
  • Stopped (T): paused by SIGSTOP.

Context Switching & the Scheduler

A context switch is the OS swapping the CPU from one thread to another. It saves register state, flushes the CPU pipeline, and potentially invalidates caches. Cost: roughly 1–10µs. Under heavy load a machine can spend meaningful CPU time on switches rather than real work. vmstat 1 shows cs (context switches per second). Too many OS threads → too many context switches.

Linux uses CFS (Completely Fair Scheduler). Each runnable task accumulates "virtual runtime" — tasks with less virtual runtime get priority. Nice values (−20 to +19) tilt the balance: nice=-20 gets more CPU, nice=19 gets less. renice changes a running process's nice value; chrt sets real-time scheduling policies.

Viewing Processes

Key columns in ps aux: PID, PPID (parent), %CPU, %MEM, VSZ (virtual memory size), RSS (resident set size — actual RAM in use), STAT (process state), COMMAND.

# All processes sorted by CPU, top 20
ps aux --sort=-%cpu | head -20

# Process tree with PIDs
pstree -p

# Show all threads of a specific process
ps -T -p <pid>

# How many voluntary context switches has this process had?
cat /proc/<pid>/status | grep voluntary

# Trace network syscalls for a running process (use carefully — overhead is real)
strace -p <pid> -e trace=network

# All open files and sockets for a process
lsof -p <pid>

The /proc Filesystem

/proc is not real files. It's a virtual filesystem — the kernel generates content on-the-fly when you read from it. Reading /proc/meminfo asks the kernel to serialize current memory stats into text. Writing to some files changes kernel behavior. No disk I/O involved.

Per-Process Directories — /proc/[pid]/

  • cmdline — the full command line that started the process (null-separated).
  • environ — environment variables (null-separated). Use cat /proc/1/environ | tr '\0' '\n' to read.
  • fd/ — symlinks to every open file descriptor. ls -la /proc/<pid>/fd shows all open files, sockets, and pipes.
  • maps — virtual memory map: every mapped region, its permissions, what file backs it.
  • smaps — detailed RSS per mapping. Use smaps_rollup for per-process totals.
  • status — human-readable summary: name, state, memory, thread count, UID.
  • io — I/O statistics: bytes read/written, syscall counts.
  • limits — current ulimits for this process.
  • oom_score — OOM killer score. Higher = more likely to be killed when memory runs out.
  • cgroup — which cgroup this process belongs to. Critical for understanding container resource accounting.

System-Wide Files

  • /proc/meminfo — total/free/available RAM, swap, buffer/cache breakdown.
  • /proc/cpuinfo — CPU model, core count, flags. Look for vmx/svm for virtualization support.
  • /proc/loadavg — load averages (1m, 5m, 15m) + running/total processes + last PID.
  • /proc/net/dev — network interface stats: bytes and packets sent/received.
  • /proc/sys/ — tunable kernel parameters, the same tree exposed by sysctl.
  • /proc/diskstats — disk I/O counters.
  • /proc/interrupts — hardware interrupt counts per CPU.

Debugging Patterns

# What is this process doing right now?
cat /proc/<pid>/status

# What files and sockets does it have open?
ls -la /proc/<pid>/fd

# Is this process in a container? What cgroup limits apply?
cat /proc/<pid>/cgroup
cat /proc/<pid>/limits

# Where is memory growing? Find the biggest heap mapping
cat /proc/<pid>/smaps | grep -A10 "heap"

# List all active kernel parameters matching a pattern
sysctl -a | grep net.core

# Read a specific parameter
sysctl net.ipv4.tcp_max_syn_backlog

# Tune a parameter at runtime (lost on reboot without /etc/sysctl.conf)
sysctl -w net.core.somaxconn=65535

Container Relevance

Inside a container, /proc still reflects the host for some data. /proc/cpuinfo reports host CPU count — not the container's CPU quota. This is why a JVM in a container with a 1-CPU limit might spin up 32 threads: it read /proc/cpuinfo, saw 32 cores, and sized the thread pool accordingly. The fix: JVM flag -XX:+UseContainerSupport (Java 10+) reads cgroup limits instead of raw /proc/cpuinfo.

File Descriptors & I/O

Everything is a file — the defining Unix abstraction. Files, directories, sockets, pipes, devices, terminals — all accessed via file descriptors. Your program doesn't need to know whether fd 5 is a disk file or a network socket: read() and write() work either way. This is why cat file | gzip | ssh server 'cat > file.gz' works — stdin, stdout, and pipes all share the same interface.

The File Descriptor Table

Each process has a table (array) of open file descriptors. Entries 0, 1, 2 are pre-opened: stdin (0), stdout (1), stderr (2). When you call open(), the kernel returns the lowest available integer ≥ 0. The descriptor is just an index into this table. The table entry points to a kernel file description — shared between fds that refer to the same open file, which matters for dup2 and fork.

File Descriptor Limits

Two tiers: the soft limit (ulimit -n) is the current cap, raiseable up to the hard limit. The hard limit is the ceiling — only root can raise it. The system-wide ceiling lives at /proc/sys/fs/file-max. The default soft limit of 1024 is absurdly low for servers. Most production boxes set it to 65536 or higher.

# Check the limit for your current shell
ulimit -n

# Raise for this session
ulimit -n 65536

# Set permanently in /etc/security/limits.conf
echo "* soft nofile 65536" >> /etc/security/limits.conf
echo "* hard nofile 65536" >> /etc/security/limits.conf

# Check limits of an already-running process
cat /proc/<pid>/limits | grep "open files"

# Count open fds for a process
ls /proc/<pid>/fd | wc -l

# Find processes with the most open fds (fd leak investigation)
for pid in /proc/[0-9]*/fd; do echo "$(ls $pid | wc -l) $pid"; done | sort -rn | head -20

What "Too Many Open Files" Actually Means

When a process hits its fd limit, any syscall that allocates a new fd — open, accept, socket, pipe — returns EMFILE. In Go: accept: too many open files. In Java: java.io.IOException: Too many open files. Common causes: an fd leak (not closing connections or files after use), a high-traffic server legitimately needing more fds, or the default limit never being raised in the first place.

Pipes

A pipe is a kernel buffer (64KB by default on Linux) with two fds — a read end and a write end. When the buffer fills, the writer blocks. When it empties, the reader blocks. This is the backpressure mechanism that makes cmd1 | cmd2 work without buffering all of cmd1's output in memory.

Blocking vs Non-Blocking I/O & epoll

Blocking I/O: read() waits until data is available. The thread suspends; the kernel wakes it when data arrives. Simple, but one thread per connection doesn't scale to thousands of connections.

Non-blocking I/O (O_NONBLOCK): read() returns EAGAIN immediately if no data is ready. The thread must try again later — typically via epoll.

epoll is the Linux mechanism for monitoring many fds simultaneously. epoll_wait blocks until any of the registered fds become readable or writable. One thread can handle thousands of connections. This is the foundation of Node.js (single-threaded event loop), Nginx, Go's netpoll, and every high-performance network server.

O_CLOEXEC: close-on-exec flag. Without it, fds are inherited by children created via fork+exec — an fd leak into child processes. Modern languages and runtimes set this automatically for sockets.

Signals & Process Control

Signals are asynchronous notifications sent to a process by the kernel or another process. The process can handle the signal (run a handler function), ignore it, or let the kernel apply the default action (often termination). Signals interrupt whatever the process is currently doing — even sleep().

Key Signals Every Backend Engineer Needs

  • SIGTERM (15) — graceful shutdown request. The default signal sent by kill. Your process should handle this: finish in-flight requests, close connections, flush buffers, then exit cleanly. If you don't handle it, the kernel terminates immediately with no cleanup.
  • SIGKILL (9) — immediate termination. Cannot be caught, ignored, or blocked. The kernel kills the process instantly — no handler runs, no cleanup happens. Use only when SIGTERM fails after a timeout. kill -9 is a last resort.
  • SIGINT (2) — interrupt from keyboard (Ctrl+C). Default: terminate. Commonly handled to trigger graceful shutdown in CLI tools and servers.
  • SIGHUP (1) — originally for terminal disconnect. Conventionally repurposed: send to a daemon to reload its configuration without restarting. Nginx: kill -HUP $(cat /run/nginx.pid) reloads config with zero downtime.
  • SIGCHLD — sent to parent when a child process changes state (exits or is stopped). Parent should call waitpid() to collect the exit status; otherwise the child becomes a zombie.
  • SIGUSR1 / SIGUSR2 — user-defined, no built-in meaning. Used for: rotating log files, dumping goroutine stack traces, triggering stats output.
  • SIGPIPE — sent when writing to a pipe or socket with no reader. Default action is termination. Servers almost always ignore this and handle the EPIPE error from write() instead.
  • SIGQUIT (3) — like SIGTERM but also dumps a core file. Go handles this specially: it prints all goroutine stacks to stderr, which is extremely useful for debugging stuck programs.

Graceful Shutdown in Practice

// Go: handle SIGTERM and SIGINT for graceful HTTP server shutdown
srv := &http.Server{Addr: ":8080", Handler: handler}

go func() {
    if err := srv.ListenAndServe(); err != http.ErrServerClosed {
        log.Fatal(err)
    }
}()

quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGTERM, syscall.SIGINT)
<-quit

// Give in-flight requests 30s to complete
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()
if err := srv.Shutdown(ctx); err != nil {
    log.Fatal("Shutdown forced:", err)
}

Kubernetes and Signals

When Kubernetes terminates a pod, it sends SIGTERM to PID 1 in the container. After terminationGracePeriodSeconds (default 30s), if the container is still running, it sends SIGKILL.

Critical: PID 1 in a container must forward signals to child processes. If your Dockerfile says CMD ["node", "app.js"], Node is PID 1 and receives SIGTERM correctly. If it says CMD ["sh", "-c", "node app.js"], sh is PID 1 and may silently swallow SIGTERM without forwarding it to Node. Fix: use exec in shell scripts to replace the shell process (exec node app.js), or use tini as a proper init process that handles signal forwarding and zombie reaping.

Sending Signals

kill -TERM <pid>        # graceful shutdown (same as kill <pid>)
kill -HUP <pid>         # reload configuration
kill -USR1 <pid>        # app-defined (often log rotation)
pkill -TERM nginx       # send signal by process name
killall node            # kill all processes named node
kill -0 <pid>           # test if process exists — no signal actually sent

Memory Management

Every process sees its own large flat virtual address space (on 64-bit x86: up to 128TB). None of it is real until used. The kernel maps virtual pages to physical RAM pages on demand via the page table. A process can have 10GB of virtual address space mapped but only 100MB of actual RAM resident.

RSS vs VSZ

  • VSZ (Virtual Size): total virtual address space mapped. Mostly meaningless for understanding real memory usage. A process can map 1GB but touch only 10MB — VSZ is 1GB, RSS is 10MB.
  • RSS (Resident Set Size): physical RAM currently in use by this process. The number you actually care about. Shown in KB by ps aux.
  • Shared memory: if 20 processes each load the same shared library, RSS double-counts it. pmap -x <pid> and /proc/<pid>/smaps distinguish private vs shared pages.

Memory Allocation

Small allocations: the C library malloc manages a heap (growing via the brk syscall). Go has its own allocator. Returning memory to the OS is not guaranteed — allocators often keep freed memory for internal reuse, so RSS may not drop immediately after free().

Large allocations: mmap(MAP_ANONYMOUS) maps a new region of virtual memory. The kernel doesn't allocate physical RAM until pages are actually written (demand paging). Freeing with munmap immediately returns pages to the OS.

Memory-mapped files: mmap() a file — its pages are mapped into virtual address space. Reads trigger page faults that load from disk. Writes dirty the page, eventually flushed by the kernel. Databases use this heavily; the kernel manages caching transparently.

Page Faults

When a process accesses a virtual page with no physical RAM backing yet, the CPU raises a page fault. The kernel handles it: allocates a physical page, zeroes it, updates the page table, returns to the process. This is normal and fast (microseconds). A hard page fault requires loading from disk (swap) — much slower, typically milliseconds.

The OOM Killer

When the kernel runs out of memory and swap, it must kill a process to free RAM. It selects based on oom_score (visible in /proc/<pid>/oom_score) — roughly proportional to memory usage. In containers, you'll see OOMKilled: true in kubectl describe pod when the container exceeded its cgroup memory limit. Adjust /proc/<pid>/oom_score_adj to protect critical processes: −1000 means never kill this process, +1000 means kill it first.

Swap

When RAM is full, the kernel can move least-recently-used pages to disk (a swap partition or swap file). Swapping is catastrophically slow for server workloads — a database that starts swapping will appear to hang. Production practice: either no swap (typical in containers) or enough swap to prevent OOM kills, with the expectation that a swapping process should be restarted. vm.swappiness (0–100) controls aggressiveness; many production setups use vm.swappiness=1.

Memory Debugging

# Overall memory picture
free -h
cat /proc/meminfo

# Per-process RSS, sorted descending
ps aux --sort=-%mem | head -20

# Detailed virtual mappings with RSS per region
pmap -x <pid> | sort -k3 -rn | head -20

# Is this machine swapping? (watch si/so columns)
vmstat 1

# How much RAM is in buffer/page cache vs actually free?
cat /proc/meminfo | grep -E "Buffers|Cached|Available"

Go Memory Notes

Go's garbage collector returns memory to the OS on its own schedule. GOGC controls GC frequency (default 100 = run GC when the heap doubles). GOMEMLIMIT (Go 1.19+) sets a soft memory limit — the GC runs more aggressively before hitting it. runtime.ReadMemStats gives detailed heap stats; pprof heap profiles show allocation hot spots.

ulimits, Tuning & Observability

ulimits are per-process resource limits enforced by the kernel. Soft limits are the current cap (a process can raise its own soft limit up to the hard limit). Hard limits are the ceiling — only root can raise them. Set in /etc/security/limits.conf or /etc/security/limits.d/, inherited by child processes, visible in /proc/<pid>/limits.

Key Limits

nofile    — open file descriptors  (default 1024, raise to 65536+)
nproc     — max processes/threads  (relevant for thread-heavy apps)
stack     — thread stack size      (default 8MB — 1000 threads = 8GB VSZ)
memlock   — memory lockable in RAM (relevant for databases like Redis)
core      — core dump size         (unlimited for debugging, 0 in prod)
# View all current limits for this shell
ulimit -a

# Raise for current session
ulimit -n 65536

# Set per-container in Docker
docker run --ulimit nofile=65536:65536 myapp

Key Kernel Parameters

Tunable via sysctl -w at runtime or persistently in /etc/sysctl.conf:

  • net.core.somaxconn — max connections waiting in the accept queue per socket. Default 128 — far too low for a busy server. Set to 65535.
  • net.ipv4.tcp_max_syn_backlog — max half-open connections during the TCP handshake. Raise for high-traffic servers.
  • net.ipv4.ip_local_port_range — ephemeral port range. Default 32768 60999 ≈ 28K ports. Under high outbound connection churn you can exhaust this. Widen to 1024 65535.
  • net.ipv4.tcp_tw_reuse — allow reuse of TIME_WAIT sockets for new connections. Set to 1 to increase ephemeral port availability.
  • vm.overcommit_memory — how the kernel handles allocation requests. 0 = heuristic, 1 = always allow (Redis recommends this to avoid fork() failures under copy-on-write), 2 = strict.
  • fs.inotify.max_user_watches — max file watches per user. Default 8192. Raise to 524288 on Kubernetes nodes or systems watching many files.

Observability Tools

# CPU: who's using it?
top                                 # interactive, per-process
htop                                # nicer interactive view
mpstat -P ALL 1                     # per-CPU utilization, updated every second
perf top                            # kernel-level hot functions (requires perf installed)

# Memory
free -h                             # overall usage
vmstat 1                            # si/so = swap in/out, bi/bo = block I/O
cat /proc/meminfo                   # detailed breakdown

# Disk I/O
iostat -xz 1                        # per-device: r/s, w/s, await (ms latency), %util
iotop                               # per-process I/O (like top for disk)
# %util near 100% = disk saturated; rising await = latency building up

# Network connections
ss -tulpn                           # listening ports with owning PIDs
ss -s                               # connection state summary (TIME_WAIT count, etc.)
netstat -an | awk '{print $6}' | sort | uniq -c | sort -rn   # TCP state histogram

# All of the above in one view
dstat 1                             # CPU + disk + net + memory, every second

# What is a process actually doing at the syscall level?
strace -p <pid> -c                 # summary: syscall counts and time spent

Load Average

The three numbers in uptime and /proc/loadavg represent the average number of runnable + uninterruptible-sleep threads over 1m/5m/15m. On a 4-core machine: load 4.0 means all CPUs fully busy; load 8.0 means 4 busy + 4 waiting. When load spikes, identify the cause: high CPU (mpstat shows %usr high) or high I/O (%iowait high in mpstat, await high in iostat).

dmesg — The Kernel Ring Buffer

The first place to look for: OOM kills (Out of memory: Kill process), disk errors (blk_update_request: I/O error), network issues, and hardware faults. dmesg -T adds human-readable timestamps. dmesg -l err,crit,alert,emerg filters to errors only.
