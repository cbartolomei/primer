⚡ Go Concurrency

Go's concurrency model is one of its defining strengths. Goroutines and channels make concurrent code readable and composable — but the primitives have sharp edges you need to understand.

🧵

Goroutines

A goroutine is a lightweight thread managed by the Go runtime, not the OS. You can start thousands — even millions — of goroutines in a single program. The keyword go is all it takes to start one:

go someFunction()         // start function as goroutine
go func() {               // anonymous function as goroutine
    fmt.Println("hello")
}()

// Goroutine with arguments — capture by value, not reference
name := "Alice"
go func(n string) {
    fmt.Println(n) // safe: n is a copy
}(name)

How Goroutines Work: M:N Scheduling

Go uses an M:N scheduler: M goroutines are multiplexed onto N OS threads. The runtime manages this automatically.

The scheduler has three key concepts:

  • G — a goroutine, the unit of concurrent work
  • M — an OS thread ("machine"), what actually runs on a CPU core
  • P — a processor, a logical CPU context that holds a run queue of goroutines. GOMAXPROCS controls how many Ps exist (default: number of CPU cores)

When a goroutine blocks on I/O or a system call, the scheduler parks that goroutine and gives the M to a different runnable goroutine. This is why you can have 100,000 goroutines blocking on network I/O without needing 100,000 OS threads — the blocked goroutines just don't occupy an M while waiting.

Goroutine Stacks

OS threads start with 1–8 MB stacks that are fixed at creation. Goroutines start with a tiny stack (~2 KB on current Go) that grows and shrinks dynamically. The runtime can handle stack growth transparently — goroutines don't panic on stack overflow the way C programs do. This small initial stack is the main reason millions of goroutines are practical.

Goroutine Leaks: The #1 Bug

A goroutine leaks when it starts but never terminates. Over time, leaked goroutines accumulate memory and hold whatever resources they're waiting on (locks, channels, connections). In long-running servers this is fatal.

The most common leak: a goroutine blocked forever waiting to receive from a channel that no one ever sends to, or blocked sending to a channel no one reads:

// LEAK — goroutine blocks forever if result is never read
func search(query string) {
    ch := make(chan Result)
    go func() {
        ch <- doSearch(query) // blocks if caller abandoned early
    }()
    // if caller returns here without reading ch, goroutine leaks
    return
}

// FIX — use buffered channel or pass a done signal
func search(ctx context.Context, query string) (Result, error) {
    ch := make(chan Result, 1) // buffered: goroutine won't block
    go func() {
        ch <- doSearch(query)
    }()
    select {
    case r := <-ch:
        return r, nil
    case <-ctx.Done():
        return Result{}, ctx.Err()
    }
}

The rule: every goroutine you start must have a way to terminate. Either it runs to completion, or it listens on a done channel / context cancellation and exits when signaled.

WaitGroup for Fan-Out

When you launch N goroutines and need to wait for all of them to finish:

var wg sync.WaitGroup

for _, url := range urls {
    wg.Add(1)
    go func(u string) {
        defer wg.Done() // always defer Done so it runs even on panic
        fetch(u)
    }(url) // pass url by value to avoid loop closure bug
}

wg.Wait() // blocks until all goroutines call Done
📡

Channels

Channels are the primary communication mechanism between goroutines. The Go mantra: "Do not communicate by sharing memory; instead, share memory by communicating." A channel is a typed conduit — only values of the declared type can flow through it.

ch := make(chan int)        // unbuffered channel of ints
bch := make(chan string, 5) // buffered channel, capacity 5

ch <- 42      // send (blocks until receiver is ready if unbuffered)
v := <-ch     // receive (blocks until a value is sent)
v, ok := <-ch // ok=false if channel is closed and drained

Unbuffered vs Buffered

Unbuffered (capacity 0): send and receive must happen simultaneously. The sender blocks until a receiver is ready, and vice versa. This is a synchronous rendezvous — a guarantee that the communication happened.

Buffered (capacity N): send blocks only when the buffer is full; receive blocks only when the buffer is empty. Acts like a bounded queue between goroutines. Use when:

  • You want to decouple the producer and consumer's timing
  • The goroutine producing results doesn't need to wait for them to be consumed (fire-and-collect)
  • You want to prevent a goroutine from leaking when the caller may not read the result
// Buffered channel as a semaphore — limit concurrency to N
sem := make(chan struct{}, N)

for _, job := range jobs {
    sem <- struct{}{}  // acquire slot (blocks when N running)
    go func(j Job) {
        defer func() { <-sem }() // release slot
        process(j)
    }(job)
}
// Wait for all to finish: drain the semaphore
for i := 0; i < N; i++ {
    sem <- struct{}{}
}

Closing Channels

Only the sender should close a channel. Closing signals "no more values will be sent." Receiving from a closed, drained channel returns the zero value immediately with ok=false:

close(ch) // sender closes when done

// Range over channel — stops when channel is closed and drained
for v := range ch {
    process(v) // range exits when ch is closed
}

// Manual check
v, ok := <-ch
if !ok {
    // channel closed and drained
}
Never close a channel from the receiver side. Sending to a closed channel panics. If multiple goroutines might send, use a sync mechanism (sync.Once or a done channel) to coordinate closing. 
// Pattern: one sender closes, multiple receivers range
func produce(ch chan<- int) {
    defer close(ch) // sender defers close
    for i := 0; i < 10; i++ {
        ch <- i
    }
}

Directional Channels

You can restrict a channel to send-only (chan<- T) or receive-only (<-chan T). Use these in function signatures to make intent explicit and let the compiler enforce it:

func producer(ch chan<- int) {  // can only send
    ch <- 1
}

func consumer(ch <-chan int) {  // can only receive
    v := <-ch
    _ = v
}

func pipeline(in <-chan int, out chan<- int) {
    for v := range in {
        out <- v * 2
    }
    close(out)
}

nil Channels

A nil channel blocks forever on both send and receive. This is actually useful in select statements: you can disable a case by setting its channel to nil:

var ch chan int // nil
<-ch           // blocks forever

// Useful pattern: disable a select case
func merge(a, b <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for a != nil || b != nil {
            select {
            case v, ok := <-a:
                if !ok { a = nil; continue } // disable this case
                out <- v
            case v, ok := <-b:
                if !ok { b = nil; continue }
                out <- v
            }
        }
    }()
    return out
}

select & sync Primitives

The select statement waits on multiple channel operations simultaneously. It's to channels what a switch is to values. The sync package provides traditional mutex-based primitives for cases where shared memory and explicit locking are cleaner than channels.

select

select {
case v := <-ch1:
    fmt.Println("got from ch1:", v)
case ch2 <- x:
    fmt.Println("sent to ch2")
case <-time.After(5 * time.Second):
    fmt.Println("timeout")
default:
    // runs immediately if no channel is ready — makes select non-blocking
    fmt.Println("nothing ready")
}

When multiple cases are ready simultaneously, Go picks one at random (uniform random selection). This prevents starvation.

Common patterns:

// Timeout on a slow operation
select {
case result := <-done:
    return result, nil
case <-time.After(3 * time.Second):
    return nil, errors.New("timeout")
}

// Cancellation with a done channel
for {
    select {
    case <-done:
        return // exit loop when signaled
    case work := <-jobs:
        process(work)
    }
}

// Non-blocking send — drop if not ready
select {
case ch <- event:
    // sent
default:
    // channel full or no receiver — drop event
}

sync.Mutex

Use a mutex when multiple goroutines need to read and write shared state and a channel would be awkward (e.g., protecting a map used as a cache):

type SafeMap struct {
    mu   sync.Mutex
    data map[string]int
}

func (m *SafeMap) Set(k string, v int) {
    m.mu.Lock()
    defer m.mu.Unlock()
    m.data[k] = v
}

func (m *SafeMap) Get(k string) (int, bool) {
    m.mu.Lock()
    defer m.mu.Unlock()
    v, ok := m.data[k]
    return v, ok
}
Do not copy a Mutex after first use. A mutex contains internal state; copying it creates a new, unlocked mutex while the original may be locked — a subtle data race. Always embed a mutex in a struct or use a pointer to one. go vet catches this with the copylocks check.

sync.RWMutex

When reads vastly outnumber writes, sync.RWMutex allows multiple concurrent readers but exclusive writer access:

type Cache struct {
    mu    sync.RWMutex
    items map[string]Item
}

func (c *Cache) Get(key string) (Item, bool) {
    c.mu.RLock()           // multiple goroutines can hold RLock simultaneously
    defer c.mu.RUnlock()
    item, ok := c.items[key]
    return item, ok
}

func (c *Cache) Set(key string, item Item) {
    c.mu.Lock()            // exclusive — no readers or other writers
    defer c.mu.Unlock()
    c.items[key] = item
}

sync.Once

Guarantees a function runs exactly once, regardless of how many goroutines call it. Classic use: lazy singleton initialization:

var (
    instance *Database
    once     sync.Once
)

func GetDB() *Database {
    once.Do(func() {
        instance = connectToDB()
    })
    return instance
}

sync.WaitGroup

Already shown with goroutines — but the key rules:

  • Call wg.Add(n) before starting goroutines, not inside them
  • Always defer wg.Done() so it runs even if the goroutine panics
  • Don't copy a WaitGroup — pass a pointer

sync/atomic

For simple counter increments and flag checks, sync/atomic operations are faster than a mutex because they don't involve OS scheduling:

import "sync/atomic"

var counter int64

atomic.AddInt64(&counter, 1)         // thread-safe increment
n := atomic.LoadInt64(&counter)      // thread-safe read
atomic.StoreInt64(&counter, 0)       // thread-safe write
atomic.CompareAndSwapInt64(&counter, old, new) // CAS

Use atomics for isolated counter/flag operations. For anything more complex (multiple fields updated together), use a mutex.

Context

The context package provides a standard way to carry deadlines, cancellation signals, and request-scoped values across API boundaries and between goroutines. Every long-running operation in Go that might need to be cancelled — an HTTP handler, a database query, a background job — should accept a context.Context as its first argument.

Context Types

// Background — the root context, never cancelled, no deadline
ctx := context.Background()

// TODO — placeholder for when you're not sure which context to use
ctx := context.TODO()

// WithCancel — manually cancel
ctx, cancel := context.WithCancel(parent)
defer cancel() // ALWAYS call cancel to free resources

// WithTimeout — cancel after duration
ctx, cancel := context.WithTimeout(parent, 5*time.Second)
defer cancel()

// WithDeadline — cancel at absolute time
deadline := time.Now().Add(5 * time.Second)
ctx, cancel := context.WithDeadline(parent, deadline)
defer cancel()

// WithValue — attach request-scoped values
type key string
ctx = context.WithValue(ctx, key("requestID"), "req_abc123")
Always call cancel() — even when the timeout has already elapsed, or you've already checked ctx.Done(). Failing to call cancel leaks the goroutine that drives cancellation. Use defer cancel() immediately after creating the context.

Checking Cancellation

// ctx.Done() returns a channel closed when context is cancelled
// ctx.Err() returns the error: Canceled or DeadlineExceeded

func doWork(ctx context.Context) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err() // context.Canceled or context.DeadlineExceeded
        case item := <-workQueue:
            if err := process(ctx, item); err != nil {
                return err
            }
        }
    }
}

// Long blocking call — pass context for cancellation
rows, err := db.QueryContext(ctx, "SELECT ...")
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)

Context Propagation

A context forms a tree: child contexts inherit their parent's deadline and cancellation. When a parent is cancelled, all its children are cancelled too. This propagation is the key design — you cancel a root context (say, an HTTP request's context) and all downstream operations automatically stop:

// HTTP handler — context is tied to the request lifetime
func handleSearch(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context() // already has deadline from server's timeout config

    results, err := searchDB(ctx, r.URL.Query().Get("q"))
    if err != nil {
        if errors.Is(err, context.Canceled) {
            // client disconnected — no need to write response
            return
        }
        http.Error(w, err.Error(), 500)
        return
    }
    json.NewEncoder(w).Encode(results)
}

func searchDB(ctx context.Context, query string) ([]Result, error) {
    // ctx carries the deadline — DB driver respects it
    rows, err := db.QueryContext(ctx, "SELECT * FROM items WHERE name LIKE ?", query)
    // ...
}

WithValue: Use Sparingly

Context values are for request-scoped data that crosses API boundaries: trace IDs, auth tokens, request IDs. They are not a general-purpose parameter-passing mechanism:

// Use unexported key types to avoid collisions between packages
type contextKey string
const requestIDKey contextKey = "requestID"

func WithRequestID(ctx context.Context, id string) context.Context {
    return context.WithValue(ctx, requestIDKey, id)
}

func RequestIDFromContext(ctx context.Context) (string, bool) {
    id, ok := ctx.Value(requestIDKey).(string)
    return id, ok
}
Don't use built-in types (string, int) as context keys — any package could use the same key and cause collisions. Always define an unexported type for your package's context keys.

Context Rules

  • context.Background() is for top-level entry points (main, init, tests). Never use it inside a library.
  • Pass context as the first argument named ctx: func Foo(ctx context.Context, ...)
  • Never store a context inside a struct field — pass it through the call chain instead.
  • Context is immutable — each With* call creates a new child; the parent is unaffected.

Concurrency Patterns

Go's primitives — goroutines, channels, select, sync — compose into higher-level patterns. These are the patterns you'll reach for repeatedly in real production code.

Pipeline

A pipeline chains stages: each stage receives values from upstream via a channel, transforms them, and sends results downstream. Stages run concurrently:

func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}

func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            out <- n * n
        }
    }()
    return out
}

// Compose into a pipeline
nums := generate(2, 3, 4)
squares := square(nums)
for v := range squares {
    fmt.Println(v) // 4, 9, 16
}

Fan-Out / Fan-In

Fan-out: distribute work from one channel to multiple goroutines. Fan-in: merge results from multiple goroutines into one channel:

// Fan-out: start N workers all reading from the same input channel
func fanOut(in <-chan Work, n int) []<-chan Result {
    channels := make([]<-chan Result, n)
    for i := 0; i < n; i++ {
        channels[i] = worker(in)
    }
    return channels
}

// Fan-in: merge N result channels into one
func fanIn(channels ...<-chan Result) <-chan Result {
    out := make(chan Result)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan Result) {
            defer wg.Done()
            for r := range c {
                out <- r
            }
        }(ch)
    }
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}

Worker Pool

Bound the number of concurrent goroutines by pre-allocating a fixed set of workers that all read from a shared job queue:

func workerPool(ctx context.Context, jobs <-chan Job, numWorkers int) <-chan Result {
    results := make(chan Result, numWorkers)

    var wg sync.WaitGroup
    for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for {
                select {
                case job, ok := <-jobs:
                    if !ok {
                        return // jobs channel closed
                    }
                    results <- process(ctx, job)
                case <-ctx.Done():
                    return
                }
            }
        }()
    }

    go func() {
        wg.Wait()
        close(results)
    }()

    return results
}

errgroup

golang.org/x/sync/errgroup is the idiomatic way to run multiple goroutines and collect the first error. It handles WaitGroup and error collection for you:

import "golang.org/x/sync/errgroup"

func fetchAll(ctx context.Context, urls []string) ([]Page, error) {
    g, ctx := errgroup.WithContext(ctx) // ctx cancelled if any goroutine errors
    pages := make([]Page, len(urls))

    for i, url := range urls {
        i, url := i, url // capture loop vars
        g.Go(func() error {
            p, err := fetch(ctx, url)
            if err != nil {
                return fmt.Errorf("fetch %s: %w", url, err)
            }
            pages[i] = p
            return nil
        })
    }

    if err := g.Wait(); err != nil {
        return nil, err // returns first non-nil error
    }
    return pages, nil
}

// Limit concurrency with errgroup.SetLimit
g.SetLimit(10) // at most 10 goroutines at once

Done Channel for Cancellation

Before context was idiomatic, the done channel pattern was the standard way to signal cancellation. You'll still see it in code that predates context:

func worker(done <-chan struct{}) {
    for {
        select {
        case <-done:
            return // shutdown signal received
        default:
            doWork()
        }
    }
}

done := make(chan struct{})
go worker(done)

// Signal all goroutines listening on done to stop
close(done) // closing broadcasts to all receivers

Closing a channel broadcasts to all goroutines receiving from it — they all unblock simultaneously. This makes it the right primitive for "stop all workers" scenarios, unlike a regular send which only wakes one receiver.

Choosing: Channels vs Mutexes

The guidance "share memory by communicating" is a principle, not a rule. Mutexes are simpler and faster for protecting shared state. Use channels when:

  • You're passing ownership of data between goroutines (the sender gives up access)
  • You're distributing work (job queues, pipelines)
  • You're signaling an event (done channels, cancellation)

Use mutexes when:

  • Multiple goroutines need to read/write shared data concurrently (caches, counters)
  • You're protecting a simple invariant and channels would add indirection
  • You need RWMutex semantics (many readers, occasional writers)
