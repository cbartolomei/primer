🔭 Observability

If you're running distributed services and you're fuzzy here, you're flying blind. This is where to start.

🏛️

The Three Pillars

The goal of observability is to understand the internal state of a system from its external outputs — without having to deploy new code every time something breaks. Logs, metrics, and traces each answer a different question. You need all three.
  • Structured Logging
    Write logs as machine-readable JSON, not human-readable strings. {"level":"error","msg":"payment failed","trace_id":"abc123","user_id":"u_789","amount":4200} is infinitely more useful than "ERROR: payment failed for user 789" — you can filter, aggregate, and join on structured fields. Log aggregation systems (Datadog, Loki, CloudWatch Insights) treat each field as a queryable column.

    Include a correlation ID in every log line. This is a trace_id (or request_id) generated at the edge and threaded through every service call for a single user request. Without it, finding all the logs for one failed request across 10 services is close to impossible. With it, you grep trace_id=abc123 and see the full story.

    Log level discipline matters: ERROR = something needs human attention now. WARN = degraded but not yet broken (expect it occasionally). INFO = normal operational events worth recording. DEBUG = everything else — never on in production by default (the volume will destroy your log budget and drown signal in noise). Treat log volume as a cost to optimize.
  • Metrics Fundamentals
    Metrics are numerical measurements over time, aggregated across many requests. They tell you how much and how fast. Four types in Prometheus:

    Counter: only goes up (request count, error count, bytes sent). Never query the raw counter — use rate() to get per-second rate. A counter reset to zero (on restart) will briefly look like a massive spike; rate() handles this correctly. Gauge: can go up or down (active connections, queue depth, memory usage, temperature). Query the raw value. Histogram: samples observations into buckets (le="0.1", le="0.5", le="1.0" for latency in seconds). Lets you compute quantiles server-side with histogram_quantile(). This is what you want for latency — p50, p95, p99. Summary: computes quantiles client-side. Cheaper, but quantiles can't be aggregated across instances — two summaries from two pods can't be combined. Prefer histograms.

    Cardinality is the footgun that takes down Prometheus. Every unique combination of label values is a separate time series stored in memory. If you put user_id, request_id, or any high-cardinality value as a Prometheus label, you will OOM the Prometheus server. Labels are for dimensions that have bounded, low-cardinality values: service, method, status_code, region.
  • Distributed Traces
    A trace is a record of a single request's journey through your system — a directed acyclic graph of spans. Each span represents one unit of work: an HTTP handler, a DB query, a cache lookup, an outbound RPC call. Spans have: a name, start time, duration, status (OK/ERROR), attributes (key-value metadata), and events (timestamped log-like annotations).

    Spans link together via trace_id (shared by all spans in a trace) and parent_span_id (points to the span that triggered this one). This forms the tree. When you view a trace in Jaeger or Tempo, you see a waterfall: the root span at top, child spans nested below, the width representing duration. A glance tells you which service is slow and whether work is happening sequentially or in parallel.

    Context propagation is how the trace stays connected across services. The trace_id and span_id travel in HTTP headers (W3C traceparent standard) or message headers. If one service forgets to forward the header, the trace breaks — the downstream service starts a new root trace and you lose the connection. This is the most common tracing failure mode.

    Sampling: tracing every request at high volume is prohibitively expensive. Head-based sampling decides at the root span (fast, simple, but you might drop the interesting 1% of errors). Tail-based sampling buffers spans and decides after the trace completes (expensive infra, but you can keep 100% of error traces). A practical default: sample 1% of normal traffic + 100% of errors.
  • RED & USE Methods
    These are your first questions in every incident — a structured way to stop thrashing and actually diagnose.

    RED (for services): Rate (requests/sec — is traffic normal?), Errors (error rate — are requests failing?), Duration (latency — are they slow?). Check RED first when a service is behaving badly. If rate is normal, errors are low, and latency is fine, the service itself is fine — the problem is elsewhere.

    USE (for resources): Utilization (% time busy — is the CPU/disk/network saturated?), Saturation (queue depth — is work piling up?), Errors (hardware errors, dropped packets). USE tells you whether a resource constraint is causing the service problem you found with RED.

    The Google SRE Book's Four Golden Signals overlap: Latency (=Duration), Traffic (=Rate), Errors, Saturation. Memorize whichever set you prefer — the point is a consistent starting framework so you don't waste 20 minutes looking in the wrong place.
  • Why Three Pillars Aren't Enough Alone
    Metrics tell you that something is wrong. An alert fires: p99 latency spiked to 4s. But metrics have no context — they can't tell you which user, which endpoint, which DB query, which code path. They're aggregates over thousands of requests.

    Logs tell you what happened in detail, but they're disconnected. You have a stream of log lines from 20 pod instances across 6 services. Without a trace ID to connect them, finding the logs for that one slow request is like finding a needle in a haystack.

    Traces show you where the time went and the causal chain — but a trace without logs gives you duration without reason. Why did the DB span take 3.5 seconds? The trace shows you which query was slow; the logs on that span tell you why (lock wait? missing index? N+1?).

    The workflow in practice: alert fires (metric) → find the request in traces (identify the slow span) → read logs for that span (understand the root cause). The trace_id links all three. This is why injecting it into your structured logs is non-negotiable.

Prometheus & Metrics in Practice

  • Prometheus Data Model
    Every time series is identified by a metric name + a set of label key-value pairs. http_requests_total{service="api",method="GET",status="200"} and http_requests_total{service="api",method="POST",status="500"} are two separate time series. Labels let you slice and dice: "show me error rate broken down by service and endpoint."

    Prometheus is pull-based: it scrapes an HTTP /metrics endpoint on your service at a configurable interval (typically 15s). Your service exposes metrics in Prometheus text format via a client library (prometheus-client for Python, prom-client for Node, micrometer for JVM). This is deliberately simple — your service doesn't need to know where Prometheus is.

    Recording rules let you pre-compute expensive PromQL expressions on ingestion: record: job:request_latency_seconds:p99. Dashboards that would take seconds to render can now query the pre-computed result instead. Use recording rules for any query your dashboard runs on every load.
  • PromQL Essentials
    The five expressions you'll use 80% of the time:

    rate(http_requests_total[5m]) — per-second average rate over a 5-minute window. Use rate() for counters, not increase() (which returns total increase and doesn't handle resets as gracefully).
    sum by (service) (rate(http_errors_total[5m])) — aggregate across pods, broken down by service label.
    histogram_quantile(0.99, sum by (le) (rate(request_duration_seconds_bucket[5m]))) — the p99 latency. The sum by (le) aggregates histogram buckets across all instances before computing the quantile — this is correct. If you skip the sum, you get per-instance quantiles which are less useful.
    rate(http_errors_total[5m]) / rate(http_requests_total[5m]) — error ratio. Useful in alerts: fire when this exceeds 0.01 (1% error rate).
    absent(up{job="api"}) — fires when a metric disappears entirely (service is down and not scraping). Critical alert: your other metrics won't even exist to alert on.
  • Alerting on Symptoms, Not Causes
    The most important alerting principle: alert on user-facing symptoms, not internal causes. An alert should mean "users are experiencing something bad right now" — not "an internal metric crossed a threshold that might eventually cause a problem."

    Bad alert: CPU usage > 80%. CPU at 80% often doesn't affect users at all. You'll get paged at 3am for nothing, get burned out on false positives, and eventually start ignoring alerts — which is catastrophic. Good alert: error rate > 1% for 5 minutes or p99 latency > 500ms for 10 minutes. These mean users are actually experiencing degradation.

    Cause-based metrics (CPU, memory, disk) are useful for dashboards and debugging — once an alert fires, you look at causes to understand why. But the alert itself should be symptom-based.

    Configure alert fatigue prevention in Alertmanager: deduplication (don't fire the same alert 100 times), inhibition (suppress downstream alerts when the root cause fires), grouping (one Slack message for 10 related alerts), and for: 5m clauses to require the condition to persist before paging.

OpenTelemetry

Before OpenTelemetry, every observability vendor had their own SDK — Jaeger client, Zipkin client, Datadog tracer, New Relic agent. If you switched vendors, you re-instrumented your codebase. OTel solves this by separating instrumentation (in your code) from the backend (wherever data goes).
  • What OpenTelemetry Is
    OpenTelemetry (OTel) is a vendor-neutral standard for generating, collecting, and exporting telemetry — covering traces, metrics, and logs under one API and SDK. The wire protocol is OTLP (OpenTelemetry Protocol). Instrument once with OTel, export to any backend: Jaeger, Tempo, Datadog, Honeycomb, New Relic, Prometheus.

    Auto-instrumentation hooks into popular frameworks and libraries automatically — no code changes required. A Java OTel agent attaches at JVM startup and patches HTTP clients, JDBC drivers, gRPC stubs, etc. Node.js has @opentelemetry/auto-instrumentations-node that patches Express, pg, Redis, etc. You get spans for every outbound call for free.

    Manual instrumentation is for your business logic — the things no framework can know about. Start a span around a critical computation, set attributes (span.setAttribute("order.id", orderId)), record events (span.addEvent("payment.authorized")), and set status (span.setStatus({code: SpanStatusCode.ERROR}) on failure). Manual spans are what make traces actually useful for debugging domain logic, not just framework calls.
  • Context Propagation
    Traces only work if the trace context travels with the request across every hop. This happens via propagation headers. The W3C standard is the traceparent header: traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 (version-traceId-parentSpanId-flags). When Service B receives this header, it creates a child span under the parent span from Service A.

    Where propagation breaks: any middleware or framework that doesn't forward headers (e.g., a custom HTTP client wrapper that strips unknown headers), async workers that don't pass context to background jobs, message queue consumers that don't read headers from messages. When propagation breaks, the trace appears as multiple disconnected root traces instead of one connected tree.

    For message queues: inject trace context into message headers when producing. Extract it when consuming. Kafka message headers and AMQP message properties both support this. OTel SDKs provide propagator.inject(carrier) and propagator.extract(carrier) to handle this generically across transport types.
  • OTel Collector
    The OTel Collector is a standalone agent/gateway that receives telemetry from your services, processes it, and exports it to one or many backends. It's not required (you can export directly from the SDK), but it's strongly recommended in production.

    The collector pipeline has three stages: Receivers (accept data via OTLP, Jaeger, Zipkin, Prometheus scrape, etc.), Processors (batch, filter, enrich, sample, redact PII), Exporters (send to Jaeger, Tempo, Datadog, S3, etc.).

    Key advantage: your application code only sends to the local collector (or sidecar). The collector handles routing, batching, retry logic, and backend configuration. Switching from Jaeger to Tempo? Update the collector config, not your application code. Running in Kubernetes: deploy as a DaemonSet (one collector per node) for efficient batching, or as a sidecar per pod if you need per-service sampling configuration.

SLOs & Error Budgets

SLOs are how you make the reliability vs velocity tradeoff explicit and data-driven. Without them, every reliability decision is a negotiation between "engineers who want to ship" and "engineers who are scared of outages." With them, you have an objective budget to spend.
  • SLI, SLO, SLA
    SLI (Service Level Indicator): a specific, measurable property of your service. Good SLIs are directly about user experience. Example: "proportion of HTTP requests that returned a 2xx response within 300ms, measured over the past 30 days." Always define the measurement window and what counts as "good."

    SLO (Service Level Objective): the target value for your SLI. "99.5% of requests will meet the SLI." This is an internal goal you hold yourself to. Choose a number based on what users actually need, not "five nines because it sounds good." 99.5% SLO = 3.65 hours of allowed downtime per month. 99.99% SLO = 4.38 minutes. Every additional 9 is exponentially harder.

    SLA (Service Level Agreement): a contract with external parties (customers, business units) that specifies consequences for breaching the SLO — typically credits, refunds, or contract terms. Your internal SLO should be slightly stricter than your external SLA so you have a warning margin before you're contractually in breach.
  • Error Budgets & Burn Rate
    Your error budget = 1 − SLO. At 99.5% SLO, you have a 0.5% error budget per month. This is the allowed amount of "bad" events. Once it's spent, you've violated your SLO.

    The key insight: the error budget makes the reliability/velocity tradeoff explicit. If your budget is healthy (plenty remaining), shipping fast and accepting some risk is rational. If you've burned through 80% of your monthly budget by week two, you freeze risky deployments and focus on reliability work. This stops both sides of the debate from being purely opinion-based.

    Burn rate alerts are more useful than simple threshold alerts. A burn rate of 1× means you'll exhaust exactly 100% of your budget by end of month. A burn rate of 14.4× means you'll exhaust the entire monthly budget in 2 hours — that's a critical incident. Alerting on burn rate (rather than raw error rate) lets you detect both slow burns (missing SLO by end of month) and fast burns (active incident) with appropriate urgency. Google's SRE Workbook has a complete burn rate alerting framework worth reading.

Test Your Understanding

You have a latency spike on a service. Walk through how you'd investigate using all three pillars.

1. Metrics first: confirm the spike is real and characterize it. Is it p50 or just p99? All endpoints or one? Started when? Check traffic and error rate too — a latency spike with rising errors is more urgent than a silent one.

2. Traces: find specific examples of slow requests during the spike window. Filter traces by duration > your p99 threshold. Look at the waterfall: is the time in your service's own code, a DB span, a downstream RPC, or a queue wait? This tells you exactly where to look.

3. Logs: take the trace ID from a slow trace. Search logs for that trace_id across all services. Read the slow span's logs for context — lock waits, slow query logs, connection pool exhaustion, retry loops.

This three-step process — alert → trace → log — takes 5 minutes with good tooling vs 45 minutes of grep-and-prayer without it.

Why is histogram_quantile more reliable than a summary for p99 latency?

A summary computes quantiles client-side and exports them as fixed values (e.g., latency{quantile="0.99"} 0.35). If you have 10 pods, you have 10 separate p99 values that cannot be mathematically aggregated — the p99 across all pods is not the average of 10 pod-level p99s.

A histogram exports raw bucket counts (latency_bucket{le="0.1"}, latency_bucket{le="0.5"}, etc.). Bucket counts can be summed across pods: sum by (le) (rate(latency_bucket[5m])). Then histogram_quantile(0.99, ...) computes the p99 from the aggregate distribution across all pods. This gives you the true fleet-wide p99, not an average of individual p99s (which would underestimate the true value).

What's the difference between monitoring and observability?

Monitoring is checking for known failure modes: "alert when error rate > 5%." It assumes you can enumerate all the ways a system can fail in advance. It works well for simple, predictable systems.

Observability is the ability to understand a system's internal state from its outputs alone — including unknown unknowns you couldn't predict in advance. A highly observable system lets you ask arbitrary questions at debug time without shipping new instrumentation. The distinction matters in distributed systems, where the failure modes are combinatorially complex and constantly surprising.

In practice: you need both. Monitoring (SLO alerts, symptom-based alerts) wakes you up. Observability (high-cardinality events, rich traces, structured logs) lets you actually understand what happened once you're awake.

Your error budget is burning too fast. What do you do?

The SLO framework gives you a clear decision tree:

  1. Identify the cause: what events are consuming budget? Use your SLI measurement to find which requests are failing or slow. Traces and logs to understand why.
  2. Freeze risky changes: no new feature deployments until budget is restored. If a deploy caused the burn, roll it back.
  3. Reliability work takes priority: the error budget is the business case for stopping feature work. Use it — that's what it's there for.
  4. If budget is consistently exhausted: either your SLO is too tight (lower it to reflect actual achievable reliability), or your system has structural reliability issues that need investment (better error handling, more redundancy, circuit breakers).
