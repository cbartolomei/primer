🔭 Observability

If you're running distributed services and you're fuzzy here, you're flying blind. Start here.

🏛️

The Three Pillars

Observability is the ability to understand a system's internal state from its external outputs — without shipping new code every time something breaks. Three types of telemetry give you this: logs, metrics, and traces. Each answers a different question. You need all three because none of them is sufficient alone.

Logs

A log is a timestamped record of a discrete event. The shift that matters most is from plaintext logs to structured logs — JSON objects with consistent, queryable fields:

// Bad — hard to query, easy to misparse
"ERROR: payment failed for user 789, amount=4200"

// Good — every field is queryable
{"level":"error","msg":"payment failed","service":"billing",
 "trace_id":"4bf92f35","user_id":"u_789","amount_cents":4200,
 "error":"insufficient_funds"}

Log aggregation systems (Datadog, Grafana Loki, AWS CloudWatch Insights) treat each JSON field as a column you can filter and aggregate on. Structured logs turn your log stream into a queryable database.

Include a trace_id on every log line. This is generated at the edge (your API gateway or the first service that handles a request) and threaded through every downstream call. Without it, correlating logs for a single failed request across 10 services means grep-and-prayer. With it, one query gives you the full story.

Log level discipline: ERROR = human attention required now. WARN = degraded but not broken, expect occasionally. INFO = normal operational milestones worth recording. DEBUG = never on by default in production — the volume destroys your log budget and buries signal in noise. Treat log volume as a cost to actively manage.

Metrics

Metrics are numerical measurements aggregated over time. They tell you how much and how fast at a system level. Four types in Prometheus:

  • Counter — only ever increases (request count, error count, bytes sent). Never query the raw value; use rate() to get per-second rate over a window. Counters reset to zero on process restart — rate() handles this gracefully, increase() does too.
  • Gauge — can go up or down (memory usage, active connections, queue depth). Query the raw value.
  • Histogram — distributes observations into pre-defined buckets (e.g., request latency: ≤10ms, ≤50ms, ≤100ms, ≤500ms). Lets you compute percentiles server-side with histogram_quantile(). This is what you want for latency.
  • Summary — computes quantiles client-side. Cheaper to query but quantiles can't be aggregated across instances. Two pods' p99 values can't be combined into a fleet-wide p99. Prefer histograms.

Cardinality is the footgun that takes down Prometheus. Every unique combination of label values is a separate time series stored in memory. If you put user_id, request_id, or any high-cardinality value as a Prometheus label, you will OOM your Prometheus server. Labels are for low-cardinality dimensions with bounded values: service, method, status_code, region.

Traces

A trace records a single request's journey through your system as a tree of spans. Each span represents one unit of work — an HTTP handler, a database query, a cache lookup, an outbound RPC. Spans carry: a name, start time, duration, status (OK/ERROR), attributes (key-value metadata), and events (timestamped annotations like "retrying").

Spans link together via a shared trace_id and parent-child span_id references. When you view a trace in Jaeger or Tempo, you see a waterfall: the root span at top, children nested below, width representing duration. In seconds you can see which service is slow and whether work is happening in parallel or sequentially.

Context propagation is how a trace stays connected as the request crosses service boundaries. The trace context travels as HTTP headers — the W3C standard is traceparent: 00-{traceId}-{parentSpanId}-{flags}. If any service forgets to forward this header, the trace breaks: the downstream service starts a new disconnected root trace. This is the most common tracing failure mode. Check your HTTP client middleware.

Sampling: tracing every request at high volume is expensive. Head-based sampling decides at the root span (fast, but you might drop the 1% of requests that are interesting). Tail-based sampling buffers all spans and decides after the trace completes (you can keep 100% of error traces, but requires more infrastructure). A practical starting point: 1% of successful traffic + 100% of errors.

How They Work Together

Metrics tell you that something is wrong. An alert fires: p99 latency spiked to 4 seconds. But metrics are aggregates — they can't tell you which user, which request, which code path.

Traces show you where the time went. You find slow traces during the spike window and look at the waterfall: is it your service's own code, a database span, a downstream RPC, a queue wait? This tells you exactly where to look.

Logs tell you why. Take the trace_id from a slow trace. Search logs for that ID. Read the logs on the slow span: lock contention? N+1 queries? Connection pool exhausted? This is where you find the root cause.

The workflow: alert fires (metric) → find bad requests (trace) → read what happened (logs). The trace_id links all three. Injecting it into your structured logs is non-negotiable.

RED & USE Methods

These give you a consistent starting framework so you don't waste time looking in the wrong place during an incident.

RED (for services): Rate (requests/sec — is traffic normal?), Errors (error rate — are requests failing?), Duration (latency — are they slow?). Check RED first when a service behaves badly. If rate is normal, errors are low, and latency is fine, the service is fine — the problem is elsewhere.

USE (for resources): Utilization (% time the resource is busy), Saturation (is work queuing up?), Errors (hardware errors, dropped packets). USE tells you whether a resource constraint is causing the service problem you found with RED.

Google's SRE Book calls these the Four Golden Signals: Latency, Traffic, Errors, Saturation. Same idea, different framing. Memorize whichever sticks — the point is a consistent, fast starting point in every incident.

Prometheus & Metrics in Practice

Prometheus is a pull-based metrics system: it scrapes an HTTP /metrics endpoint on your services at a configured interval (typically 15s). Your service exposes metrics in Prometheus text format via a client library. This is deliberately simple — your service doesn't need to know where Prometheus is.

Data Model

Every time series is identified by a metric name + a set of label key-value pairs. These two are completely separate time series:

http_requests_total{service="api", method="GET",  status="200"}
http_requests_total{service="api", method="POST", status="500"}

Labels let you slice and dice: "show me error rate broken down by service and endpoint." The flip side is that every unique label combination is a separate time series stored in memory. Adding a label that has 10,000 possible values (like user_id) creates 10,000 time series. This is the cardinality problem — it's brought down real Prometheus clusters. Use labels only for dimensions with bounded, low-cardinality values.

PromQL Essentials

The five expressions you'll use 80% of the time:

# Per-second rate of requests over a 5-minute window
rate(http_requests_total[5m])

# Error rate as a ratio — alert when this exceeds 0.01
rate(http_errors_total[5m]) / rate(http_requests_total[5m])

# Aggregate across all pods, broken down by service
sum by (service) (rate(http_requests_total[5m]))

# Fleet-wide p99 latency from a histogram
# The sum by (le) aggregates buckets across all instances FIRST
# then histogram_quantile computes the percentile — order matters
histogram_quantile(0.99,
  sum by (le) (rate(http_request_duration_seconds_bucket[5m]))
)

# Alert when a service disappears entirely (no metrics = service down)
absent(up{job="api"})

Recording rules let you pre-compute expensive PromQL expressions at ingestion time. Any query your dashboard runs on every page load should be a recording rule — otherwise every dashboard open triggers a heavy query. Define them in your Prometheus config or as PrometheusRule CRDs in Kubernetes.

Alerting on Symptoms, Not Causes

The most important alerting principle: alert on what users experience, not on internal resource metrics.

Bad: CPU usage > 80% — CPU at 80% often doesn't affect users at all. You get paged at 3am for nothing, accumulate alert fatigue, and start ignoring pages. That's when real incidents go unnoticed.
Good: error rate > 1% for 5 minutes or p99 latency > 500ms for 10 minutes. These mean users are experiencing something bad right now.

Cause-based metrics (CPU, memory, disk) belong on dashboards — you look at them after an alert fires to understand why. They shouldn't be the alert themselves.

Use Alertmanager for: deduplication (don't fire the same alert 100 times), grouping (one Slack message for 10 related alerts), inhibition (suppress downstream alerts when the root cause is already firing), and silencing (mute alerts during planned maintenance). Always use for: 5m on alerts to require the condition to persist before paging — transient spikes shouldn't wake people up.

OpenTelemetry

Before OpenTelemetry, every observability vendor had their own SDK — Jaeger client, Zipkin client, Datadog tracer, New Relic agent. Switching vendors meant re-instrumenting your entire codebase. OpenTelemetry (OTel) solves this by standardizing the instrumentation layer: instrument once, export anywhere.

OTel is a vendor-neutral standard for generating, collecting, and exporting telemetry — covering traces, metrics, and logs under one API and SDK. The wire protocol is OTLP (OpenTelemetry Protocol). Instrument with OTel once, then route to Jaeger, Tempo, Datadog, Honeycomb, or any OTLP-compatible backend by changing config, not code.

Auto-Instrumentation vs Manual

Auto-instrumentation hooks into popular frameworks and libraries without any code changes. A Java OTel agent attaches at JVM startup and patches HTTP clients, JDBC drivers, gRPC stubs, Kafka consumers, etc. Node.js has @opentelemetry/auto-instrumentations-node that patches Express, pg, Redis, axios, and more. You get spans for every outbound call for free, often in under an hour of setup.

Manual instrumentation is for your business logic — the things no framework can know about. You create spans around critical operations, add attributes, and record events:

const span = tracer.startSpan('process-payment', {
  attributes: {
    'payment.method': 'card',
    'payment.amount_cents': 4200,
    'order.id': orderId,
  }
});

try {
  const result = await chargeCard(orderId, amount);
  span.addEvent('payment.authorized', { auth_code: result.authCode });
  span.setStatus({ code: SpanStatusCode.OK });
  return result;
} catch (err) {
  span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
  span.recordException(err);
  throw err;
} finally {
  span.end();
}

Manual spans are what make traces actually useful for debugging domain logic — not just showing you "the HTTP handler called the DB."

Context Propagation in Depth

For a trace to stay connected across services, the trace context must travel with every request. Over HTTP, this is the W3C traceparent header:

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             ^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^  ^^^^^^^^^^^^^^^^  ^^
             version    trace ID (128-bit)        parent span ID    flags

When Service B receives this header, its OTel SDK automatically creates a child span under the parent. The trace tree builds itself.

Where propagation commonly breaks: custom HTTP client wrappers that strip unknown headers, async workers that don't carry context to background jobs, message queue consumers that don't read context from message headers. For message queues, inject trace context into message headers on produce and extract on consume — OTel SDKs provide propagator.inject(carrier) and propagator.extract(carrier) to make this generic.

The OTel Collector

The OTel Collector is a standalone agent/gateway: it receives telemetry from your services, processes it, and exports it to one or many backends. Not required, but strongly recommended in production.

The pipeline has three stages: Receivers (accept OTLP, Jaeger, Zipkin, Prometheus scrape, etc.), Processors (batch, filter, transform, sample, redact PII from span attributes), Exporters (send to Jaeger, Grafana Tempo, Datadog, S3 for long-term storage, etc.).

Key benefit: your application code only sends to the local collector. The collector handles routing, retry, backend config. Switch from Jaeger to Grafana Tempo? Update the collector config file. No application changes needed. In Kubernetes, deploy the collector as a DaemonSet (one per node, efficient batching) or as a sidecar if you need per-service sampling config.

SLOs & Error Budgets

SLOs are how you make the reliability vs velocity tradeoff explicit and data-driven. Without them, every reliability decision is a negotiation between teams. With them, you have an objective budget that makes the conversation concrete.

SLI → SLO → SLA

An SLI (Service Level Indicator) is a specific, measurable property of your service directly tied to user experience. Good SLIs: "proportion of HTTP requests returning 2xx within 300ms, measured over 30 days." Bad SLIs: "CPU < 80%" (users don't experience CPU directly).

An SLO (Service Level Objective) is your target value for the SLI: "99.5% of requests will meet the SLI." This is an internal goal. Choose numbers based on what users actually need, not prestige. Every additional 9 is exponentially harder:

SLOMonthly downtime allowedWeekly downtime allowed
99%7h 18m1h 41m
99.5%3h 39m50m
99.9%43m 49s10m
99.95%21m 54s5m
99.99%4m 22s1m

An SLA (Service Level Agreement) is a contract with external parties — customers or business units — that specifies consequences for SLO breaches (credits, refunds, contract terms). Your internal SLO should be slightly stricter than your SLA so you have a warning margin before you're contractually in breach.

Error Budgets

Your error budget = 1 − SLO. At 99.5% SLO, you have a 0.5% monthly error budget. This is the allowed amount of "bad" request-minutes per month. Once it's spent, you've violated your SLO.

The key insight: the error budget makes the reliability/velocity tradeoff explicit and data-driven. If you have budget left, shipping fast and accepting some risk is rational — the data supports it. If you burned through 80% of your budget by week two, you freeze risky deployments and do reliability work. This stops both sides from being purely opinion-based.

Burn Rate Alerting

Simple threshold alerts on SLI violations ("error rate > 0.5%") are slow to catch gradual degradation and noisy during short spikes. Burn rate alerts are better.

Burn rate = how fast you're consuming your error budget relative to the rate that would exhaust it exactly at end of period. A burn rate of 1× means you'll use exactly 100% of budget by month end. A burn rate of 14.4× means you'll exhaust the entire monthly budget in 2 hours — that's a critical incident requiring immediate response.

# PromQL: 1-hour burn rate for a 99.5% SLO
# error_budget_fraction = 1 - 0.995 = 0.005
1 - (
  sum(rate(http_requests_total{status=~"2.."}[1h]))
  /
  sum(rate(http_requests_total[1h]))
) / 0.005

# Alert: burn rate > 14.4 = exhausts monthly budget in 2 hours
# Alert: burn rate > 6   = exhausts monthly budget in 5 hours

Use two alerting windows: a short window (1h) for fast burns (active incidents) and a long window (6h) for slow burns (gradual degradation you might miss). The Google SRE Workbook Chapter 2 has the complete burn rate alerting framework — worth reading in full.
