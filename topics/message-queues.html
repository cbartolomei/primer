📨 Message Queues

Async messaging patterns, delivery semantics, Kafka architecture, and RabbitMQ concepts.

📋

Fundamentals

  • Why Message Queues?
    Decoupling: producer and consumer don't need to be running simultaneously or know about each other. Buffering: absorb traffic spikes — producer can outpace consumer temporarily without dropping work. Async processing: return a response to the user immediately, process the work in the background (email sending, image resizing, report generation). Fan-out: one event triggers multiple independent consumers (order placed → billing service + inventory service + notification service). Reliability: messages persist until acknowledged — work isn't lost if a consumer crashes. Backpressure: queue depth signals that consumers need scaling.
  • Messaging Patterns
    Point-to-Point (Queue): one producer, one consumer. Each message consumed by exactly one consumer. Good for task distribution — work items shared across a pool of workers. Publish/Subscribe (Topic): one producer, many independent consumers (subscribers). Each subscriber gets every message. Good for event broadcasting — "order placed" → all interested services receive it. Request-Reply: producer sends a message with a reply-to queue and correlation-id; consumer processes and sends response to reply-to queue. Async RPC pattern. Competing consumers: multiple consumers on the same queue — messages distributed across them for horizontal scaling.
  • Delivery Semantics
    At-most-once: fire and forget. Message may be lost if consumer crashes before processing. No retries. Lowest overhead. OK for metrics, logs where loss is tolerable. At-least-once: message redelivered until acknowledged. Consumer may process the same message multiple times (on crash before ack). Requires idempotent consumers — processing the same message twice must be safe (use an idempotency key + deduplication store). Exactly-once: hardest guarantee. Requires coordination between producer and broker (Kafka transactions + idempotent producer) and often between broker and consumer's downstream system (transactional outbox). Adds significant overhead. Default to at-least-once + idempotent consumers — it's usually sufficient and much simpler than exactly-once.
🟠

Kafka

  • Kafka Architecture
    Topic: named, append-only, immutable log. Messages are never deleted on consumption — retained for a configurable period (default 7 days) or size. Partition: unit of parallelism and ordering. A topic is split into N partitions. Ordering is guaranteed within a partition, not across the topic. Partition count determines max consumer parallelism. Offset: sequential ID of a message within a partition. Consumers track their position via committed offsets. Broker: a Kafka server. Multiple brokers form a cluster. Replication factor: each partition is replicated across F brokers. Leader partition handles reads/writes; followers replicate. KRaft (new): Kafka without ZooKeeper — metadata managed internally.
  • Consumer Groups & Offsets
    Consumer group: a group of consumers sharing a group ID. Each partition is assigned to exactly one consumer in the group — horizontal scaling. Two groups = two independent consumers of the same data. Offset commit: consumer commits offset after processing. On restart, resumes from last committed offset. Committing before processing → at-most-once; committing after → at-least-once. Consumer lag: difference between latest produced offset and last committed consumer offset. High lag = consumers falling behind. Monitor per-partition lag in production. Rebalancing: triggered when consumers join/leave the group or partitions change. During rebalance, consumption pauses. Cooperative incremental rebalancing (Kafka 2.4+) reduces pause duration.
  • Producer Config & Guarantees
    acks: 0 = fire and forget (fastest, data loss possible), 1 = leader acked (leader can fail before replication), all/-1 = all in-sync replicas acked (no data loss, slowest). Batching: producer buffers messages and sends in batches. linger.ms (wait up to N ms to fill batch) + batch.size. Compress batches with snappy/lz4/zstd. Idempotent producer (enable.idempotence=true): broker deduplicates retried sends within a session. Combined with acks=all gives exactly-once within the producer session. Transactions: produce to multiple partitions atomically + commit consumer offset atomically — true end-to-end exactly-once.
  • Kafka Use Cases
    Event streaming: high-throughput, ordered event log (activity tracking, clickstreams). Change Data Capture (CDC): Debezium reads DB binlog/WAL and publishes row changes to Kafka — sync DB changes to other systems without polling. Log aggregation: centralize logs from many services into one stream for processing. Stream processing: Kafka Streams (JVM library, stateful operations, exactly-once), Apache Flink (more powerful, separate cluster), ksqlDB (SQL over Kafka streams). Message queue replacement: for high-throughput, durable, replayable work queues. Not ideal for: low-latency (<1ms), complex routing logic (RabbitMQ is better), small message counts with simple needs.
🐇

RabbitMQ

  • Exchanges, Queues & Bindings
    Producers publish to exchanges, not queues. Exchanges route messages to queues via bindings (routing rules). Exchange types: Direct — route by exact routing key match. Fanout — broadcast to all bound queues (ignore routing key). Topic — routing key pattern match (logs.*, #.error). Headers — match on message header attributes. Queue properties: durable (survives broker restart), exclusive (deleted when connection closes), auto-delete (deleted when last consumer unsubscribes). Message TTL: expire messages after N ms. Queue max-length: cap queue size — overflow goes to DLX.
  • Dead Letter Queue (DLQ)
    Messages are routed to a Dead Letter Exchange (DLX) when: rejected with requeue=false, TTL expires, queue length limit exceeded. DLQ patterns: Inspect failed messages manually. Set up retry logic: DLQ consumer reprocesses after a delay (via message TTL + another DLX), up to N retries, then move to a parking lot queue for human review. Never silently discard errors — always DLQ failed messages. DLQ depth is a key operational metric to alert on. In Kafka: use a separate dead-letter topic (orders.DLT). Spring Kafka and others handle this natively.

Quick Review

How do you implement an idempotent consumer?

An idempotent consumer produces the same result whether it processes a message once or N times. Common approaches:

  • Idempotency key + deduplication store: include a unique message ID. Before processing, check if ID exists in a store (Redis, DB unique index). If yes, skip. If no, process and store the ID. The check-and-process must be atomic (use a transaction or Redis SET NX).
  • Natural idempotency: design the operation to be inherently idempotent. UPDATE orders SET status = 'shipped' WHERE id = $id is idempotent — running it twice has the same effect as once.
  • Optimistic locking: include expected state in the operation — if state has already moved past this event, the update is a no-op.
Why does Kafka guarantee ordering only within a partition?

Kafka partitions are the unit of replication and sequential writing. Within a partition, messages have monotonically increasing offsets and are written and read in order. Consumers process a partition sequentially.

Across partitions, there is no ordering guarantee — different partitions are consumed by different consumers independently, and their relative timing is not coordinated.

Practical implication: to guarantee ordering for a specific entity (e.g., all events for order_id=123), route all messages for that entity to the same partition using the entity ID as the partition key. This ensures one consumer processes them in order.

What is the transactional outbox pattern?

A classic problem: you want to update your DB and publish a message atomically. If you update the DB then publish, the app can crash between the two steps. If you publish then update, same problem in reverse.

The transactional outbox pattern: write the message to an outbox table in the same DB transaction as your business update. A separate relay process (or Debezium CDC) reads the outbox table and publishes to the message broker, then marks the row as sent.

This guarantees that a message is published if and only if the business transaction commits. The relay delivers at-least-once, so consumers must still be idempotent.
